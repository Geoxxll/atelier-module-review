Plate 02 — HTTP API

HTTP API

浏览器与整套后端之间唯一的一道门——把用户意图翻成工作流，把 Agent 进度实时喂回浏览器。

一句话：这层是 FastAPI 写的一道薄门。它只有两个面——REST 把「提需求 / 批准 / 反馈」翻成 启动 workflow + 发 Signal；SSE 订阅 Core NATS 把 Agent 进度流式推回浏览器。进出都被 httpOnly cookie 里的 RS256 JWT 与 Postgres RLS 两道闸门夹住，业务状态一概不留在这里。

为什么中间要隔一层 HTTP API？

浏览器不能直接对 Temporal、NATS、Postgres 说话——它们不是给公网浏览器用的接口，也没有「谁是谁、能看什么」的概念。所以中间必须有一层翻译 + 守门的门房，这就是 HTTP API。它做的事其实只有两类，恰好对应 Web 的两种通信形态：

  • 请求-响应（REST）——「我要提个需求」「我批准这份 PRD」。API 收到后立刻转成一个动作：要么启动一条 workflow，要么给正在跑的 workflow 发一个 Signal，然后马上返回，绝不在这里等 Agent 干完。
  • 流（SSE）——「这单进展到哪了？」Agent 干活是分钟级的，不能让浏览器轮询。API 开一条长连接，订阅 NATS 上这条 workflow 的事件主题，Agent 每推一步就实时转发一行给浏览器。

关键的设计克制是：API 本身几乎不持有业务状态。「这单跑到哪一步、下一步是什么」全归 Temporal 那台状态机管；API 只在自己的 workflows 表记一本「用户能看到的账」（谁、什么时候、花了多少），其余一律转手。正因为它这么薄，才能把精力全花在两件真正难的事上：多租户隔离和把 NATS 的流桥接成浏览器的流。

贯穿全篇的一条主线 · 门房不留客 REST 端点几乎全是「三行代理」：拿 handle → 发 Signal → 返回。真正的复杂度不在业务逻辑（那在 Agent 和 workflow），而在两道闸门——按 tenant_id 隔离的 RLS，和把 Core NATS 订阅接成 SSE 的桥。「深挖」里逐个拆。

端点全景：一次创建 + 九个 Signal 代理 + 账号地基

一眼看穿：工作流写路径里，创建之外的九个全是 Signal 代理——这印证了「门房不留客」：API 不实现任何 refine / PRD / tech-spec / plan 的业务逻辑，只把动作原样转手给 Temporal 的状态机。REST 与 workflow 的暂停点一一对应，前端要加一个「人来拍板」的点，这里就多一个薄代理。外圈的账号 / 项目 / repo / 用量端点则各是一层薄 CRUD，同样不留业务状态。

两种 DB 连接：严格隔离 vs 上帝视角

整个模块最值得看的静态设计，是并存的两个连接上下文管理器。它俩长得像、用法像，但对 RLS 的态度完全相反——一个把租户闸门焊死，一个刻意拆掉闸门。什么时候用哪个，是安全边界的核心：

两者共同的关键都是 SET LOCAL（事务级、事务结束自动恢复）——连接池会复用连接，若用会话级 SET，A 用户的 tenant_id 会残留给下一个拿到这条连接的 B 用户，直接串租户。事务级设置是隔离能成立的前提，见「深挖 ①」。

为什么要有第二个：可观测台需要「上帝视角」跨租户看所有用户的 run / 记忆，这与生产的严格隔离天然冲突。解法不是在业务连接上开后门，而是另起一个刻意绕 RLS、但用 readonly=True 由数据库兜底「绝不改数据」的只读连接，且只在 DEV_MODE 下才把用它的 admin 路由挂上去。安全边界靠「物理上另一条路 + 只读 + 不挂载」三重保证，而非注释约定。

图版 II · 动态流程

写靠「翻译 + 启动」，读靠「订阅 + 转发」

两个面各有一条清晰的时序：创建请求是「先写账、再启动、马上返回」；SSE 是一条会长活几分钟的长连接，从订阅到收尾各有讲究。

图版 II-a · 写 · POST /features 创建链路DYNAMIC · request path
1
鉴权 · Depends(get_current_user)

从 httpOnly cookie（atelier_access）读 access token，RS256 公钥验签，拿到 user_id / tenant_id / role。缺失/无效/过期一律 401——没有 DEV 兜底分叉，本地联调先 POST /api/dev-login 换真 cookie。

依赖注入：鉴权作为函数参数声明，路由正文拿到的已是 UserContext，逻辑单独可测。

2
写账 · acquire_with_tenant → INSERT workflows

先在自己的 workflows 表写一条 status='running'。连接经 acquire_with_tenant，事务内 SET LOCAL app.tenant_id，RLS 保证只能写进本租户。

为什么先写 DB？——见步骤 3 的顺序取舍。

3
启动 · temporal_client.start_workflow

再把 FeatureWorkflowInput 交给 Temporal 启动 FeatureWorkflow，id=workflow_id（与 DB 主键同值，天然幂等 / 去重）。

顺序刻意「先写 DB 再启 Temporal」：若反过来，DB 写失败会留下「Temporal 在跑但 DB 查不到」的幽灵——比孤儿记录难排查得多。

4
立即返回 · 202 Accepted

返回 {workflow_id} 就结束——绝不在这里等 Agent 干完。前端拿着这个 id 立刻去开 SSE（图版 II-b）看进度。

核心是「不等」：Agent 干活是分钟级的，HTTP 请求扛不住也不该扛。创建端点做的全是秒级的事（写一行、发一个启动请求），然后用 202 Accepted 明确告诉前端「收到了，正在异步处理」——真正的进度靠另一条 SSE 通道推。

图版 II-b · 读 · SSE 长连接生命周期DYNAMIC · stream path
1
换票 + 订阅 · consume(ticket) → nc.subscribe

前端先 POST /api/sse-ticket（走 cookie 鉴权）换一枚一次性 ticket，再用 EventSource 打开 ?ticket=。端点先 核销 ticket（取出即删、校验绑定的 workflow_id）——无票/失效/不匹配即 401/403，才订阅 Core NATS 主题并先吐一个 connected 握手事件。

原生 EventSource 带不了自定义 header，又不想把 15min access token 暴露进 URL query，故用短命一次性 ticket 顶上（tickets.py，Redis SETEX 30s + GETDEL）。

2
转发 · async for msg in sub.messages

Agent 每 publish 一步进度，这里就收到、json.loads 解出 type 作为 SSE event 名、原样 yield 给浏览器。每轮先查 request.is_disconnected()，客户端走了就跳出。

Core NATS 而非 JetStream：进度丢了没关系（前端可刷新重连），Core 更轻、延迟更低。

3
收尾回写 · 终止事件 → UPDATE workflows.status（即时双写）

收到 workflow_completed / failed / rejected（或闭环链路的 finalized）时，经 resolve_terminal_status 映射出目标态，UPDATE status/ended_at（带 NON_OVERRIDABLE 幂等护栏）然后 break。回写失败只 warning。

#133：这里只是「有客户端在线时的即时双写」。客户端关标签页时没有这条路径——终态真正的收口交给常驻 event_log_worker（见深挖③）。

4
清理 · finally → sub.unsubscribe()

无论是正常 break、客户端断开（CancelledError），还是异常，finally 都保证退订 NATS 订阅，绝不泄漏。

SSE 的三个隐形要点：①开头一个 connected 事件是握手回执；②每轮循环主动探测 is_disconnected，避免给已走的客户端空转；③finally 退订是长连接不泄漏订阅的命门。admin 侧的 SSE 还要同时多路复用 .event + .llm 两条主题——见「深挖 ⑥」。

端到端 · 走一遍

从「提一个需求」到「进度实时刷出来」

把 REST 创建与 SSE 流两条链路串成一个完整故事，你就看清了 API 层到底在两个面之间干了什么。

场景 · 用户提交一个功能需求并盯着进度

George 在前端提「加一个报销审批功能」，然后看着 PM/Architect 逐步跑出来

T0 · 提交

前端 POST /api/projects/{pid}/features（cookie 自动带上）。get_current_user 从 cookie 里的 RS256 token 解出 tenant，先配额预检、再写账、再启 workflow。

# features.py create_feature
workflow_id = uuid4()
async with acquire_with_tenant(db, user.tenant_id) as conn:
    await check_monthly_quota(conn, user.tenant_id)  # 超限 → 402
    # ① 先写自己的账本（RLS 生效，只能写本租户）
    await conn.execute("INSERT INTO workflows (... status='running' ...)")
# ② 再交给 Temporal 编排（id 与 DB 主键同值）
await temporal_client.start_workflow(FeatureWorkflow.run, ..., id=workflow_id)
return CreateFeatureResponse(workflow_id=workflow_id)  # 202，立即返回
T0+ · 换票订阅

前端拿到 workflow_id，先 POST /api/sse-ticket 换一次性 ticket，再用 EventSource 打开 GET /events?ticket=。端点核销 ticket（校验租户 + workflow 归属）后才订阅 NATS、先回一个 connected。

# issue_sse_ticket：签票前查 workflows 验租户归属
ticket = await ticket_store.issue(user_id=..., tenant_id=..., workflow_id=...)
# stream_workflow_events：无票/失效即 401/403
payload = await ticket_store.consume(ticket)          # 取出即删，防重放
sub = await nc.subscribe(f"workflow.{workflow_id}.event")
yield {"event": "connected", "data": ...}  # 握手回执
T1..Tn · 流

Temporal 里 PM → Architect → Coder 逐步跑，每一步 Agent 都往 workflow.{id}.event publish。API 逐条转发，前端时间线实时长出来。

# 每条 Agent 进度：NATS → API → 浏览器
async for msg in sub.messages:
    if await request.is_disconnected(): break
    ev = json.loads(msg.data.decode())
    yield {"event": ev["type"], "data": json.dumps(ev)}
# 前端收到 agent_started / prd_generated / ... 逐条渲染
中途 · 拍板

跑到「批准 PRD」暂停点，前端弹出 PRD。George 点批准 → POST /approve-prd。API 把它翻成一个 Signal 送回正在等的 workflow，workflow 继续往下走。

# approve_prd —— 典型的「三行 Signal 代理」
handle = temporal_client.get_workflow_handle(workflow_id)
await handle.signal("approve_prd")
return SignalResponse(workflow_id=workflow_id)

同一条 SSE 长连接全程不断：批准是另开的一个短 POST，workflow 收到 Signal 后继续 publish，事件继续从老的 SSE 流里冒出来。

Tn · 收尾

Coder 跑完，workflow publish workflow_completed（闭环链路则是 finalized）。SSE 若还连着就即时回写 DB、break、退订；而不论前端在不在线，常驻 event_log_worker 都会把这条事件落 event_log 并兜底收口 status（#133）。

# 两处共用同一份终态映射 terminal_status.resolve_terminal_status
new_status = resolve_terminal_status(ev["type"], ev)   # completed/failed 或 None
if new_status is not None:                              # SSE 侧：即时双写
    UPDATE workflows SET status=$1, ended_at=now()
      WHERE id=$2 AND status NOT IN (completed,failed,cancelled)  # 幂等护栏
    break
# event_log_worker 常驻消费者：不依赖 SSE 连接，独立跑同一段收口
# finally: await sub.unsubscribe()  ← 无论如何都退订

整条链路 API 只碰了三样东西：workflows / event_log 表（写账 + 收尾回写）、Temporal（启动 + 转 Signal）、NATS（订阅转发）。PRD/代码/审查这些大产物它一概没经手——闭环路径落 git 集成仓库、legacy 路径走 Blackboard，API 全程不知情。

核心机制 · 钻到实现级

六个撑起这层门房的机制

展开看代码级细节。这几处也是面试里最容易被追问「你这层具体怎么保证安全 / 实时」的地方。

① 多租户 RLS：SET LOCAL 为什么是隔离的命门

多租户隔离不是在每条 SQL 手写 WHERE tenant_id=?（一旦漏写就泄漏），而是把闸门下沉到数据库：表上挂 RLS 策略 tenant_id = current_setting('app.tenant_id')，连接进事务前先设好这个变量。关键在用事务级的 SET LOCAL 而非会话级 SET：

async with pool.acquire() as conn:          # 从池里取一条连接
    async with conn.transaction():          # 开事务
        await conn.execute(
            "SELECT set_config('app.tenant_id', $1, true)",  # 第三参 true = SET LOCAL
            str(tenant_id),
        )
        yield conn                          # 事务结束自动清除，连接干净归还
方式作用域连接池复用时
SET（会话级）整条连接存活期A 的 tenant_id 残留 → B 拿到连接后串租户
SET LOCAL（事务级）仅当前事务事务一结束自动恢复 → 连接干净，天然隔离

连接池的本质是「连接被反复复用」，所以任何附着在连接上的状态都必须随事务生命周期自动清理，否则就是跨请求的数据泄漏。这是「连接池 + RLS」组合里最容易踩、也最该讲清楚的一个点。

② 上帝视角连接：怎么安全地「刻意绕过 RLS」

开发者可观测台要跨租户看所有用户，与生产的严格隔离直接冲突。解法不是给业务连接开后门，而是并存一个刻意关掉 RLS 但被三重约束焊死的只读连接：

async with pool.acquire() as conn:
    await conn.set_type_codec("jsonb", encoder=json.dumps, decoder=json.loads, ...)  # ③ jsonb 自动反序列化
    async with conn.transaction(readonly=True):     # ① 事务只读，DML 被 DB 拒
        await conn.execute("SET LOCAL row_security = off")  # ② 事务级关闭 RLS
        yield conn
  • 不设 app.tenant_id + row_security=off：即便表设了 FORCE ROW LEVEL SECURITY 也能跨租户全读。
  • readonly=True 由 DB 兜底：任何意外的写会被 Postgres 直接拒，不靠「只读」的注释约定。
  • DEV_MODE 挂载 + require_admin 双闸（AND）：register_routes(dev_mode) 里生产环境根本不 include admin 路由（物理不存在）；即便 DEV_MODE 把路由挂上，router 级的 Depends(require_admin) 仍要求 role=='admin'，非 admin 一样 403——不是「DEV_MODE=true 就人人可看」。
jsonb 陷阱asyncpg 不像 psycopg2 会自动把 jsonb 解析成 dict，默认当字符串返回。可观测台读 workflows.input / agent_memory.metadata 时若不注册 codec，到前端就是一坨 JSON 字符串、读不到 .description（docs/e2e-common-errors.md #1）。所以 service-role 连接显式 set_type_codec("jsonb", ...) 自动 loads。
③ SSE-over-NATS 桥：把消息总线接成浏览器的流

SSE 是「服务器 → 浏览器」单向长连接的最轻方案（就是一个 text/event-stream 的 HTTP 响应，不像 WebSocket 要升级握手）。这层桥要做的是：先核销一次性 ticket 守门，再把 NATS 的 推（async 迭代订阅）转成 SSE 的 yield（异步生成器），并处理好长连接的三种收场。

# 建连关卡：原生 EventSource 带不了 header，改校验 ?ticket=
payload = await ticket_store.consume(ticket)          # 取出即删（Redis GETDEL）
if payload is None: raise HTTPException(401)           # 无票/过期/已用
if payload.workflow_id != workflow_id: raise HTTPException(403)  # 越权

async def event_generator():
    sub = await nc.subscribe(f"workflow.{id}.event")
    yield {"event": "connected", ...}              # 握手回执
    try:
        async for msg in sub.messages:              # NATS 推 → SSE yield
            if await request.is_disconnected(): break  # 主动探测客户端走没走
            ev = json.loads(msg.data.decode())
            yield {"event": ev.get("type","update"), "data": json.dumps(ev)}
            if ev.get("type") in TERMINAL: ... update status; break
    except asyncio.CancelledError: pass             # 客户端断开 → 协程被取消
    finally: await sub.unsubscribe()                # 命门：无论如何都退订
return EventSourceResponse(event_generator())

三种收场必须都覆盖：正常（收到终止事件 break）、客户端主动断（is_disconnected 探测到 / CancelledError）、异常。三条路都汇进 finally 退订——否则每个断开的浏览器都留一个悬挂的 NATS 订阅，慢慢泄漏。

#133 · 终态不能只靠这条流这里收到终止事件回写 workflows.status 的那段，只在有 EventSource 客户端连着消费时才跑。用户提交后立刻关标签页，这段代码没有任何执行路径，status 会永卡 running。真正兜底的是独立常驻的 event_log_worker.py：它订阅同一 subject 落 event_log 表，并复用同一份 terminal_status.resolve_terminal_status 幂等收口 status——不依赖任何前端是否在线。SSE 侧因此降级为「有客户端时的即时双写」，不再是唯一写手。

为什么 SSE 不用 JetStream？进度事件是「丢了可重来」的即时信号（前端刷新即重新订阅拿当前态），Core NATS 的即发即弃更轻、延迟更低；JetStream 的持久化/重放在这里是负担。任务派发那种「绝不能丢」的才走 JetStream。断线补档另有一条路：GET /events?last_event_id= 从 event_log 按 seq 重放缺口段（#123）。

④ Signal 代理：为什么这层几乎不写业务逻辑

九个 Signal 端点几乎是同一个模板：拿 handle → 发 signal → 返回。业务状态（现在等哪个 Signal、下一步去哪）全在 Temporal 的 workflow 定义里，API 只是把一次 HTTP 动作转手成一个 Signal：

# 无参 Signal
handle = temporal_client.get_workflow_handle(workflow_id)
await handle.signal("approve_prd");  return SignalResponse(...)

# 带结构化载荷的 Signal（Pydantic 契约）
await handle.signal("resolve_tech_spec", TechSpecResolutionInput(approved=True))
await handle.signal("provide_clarification", ClarificationSignalInput(answers=body.answers))

这种「薄」是刻意的架构选择：把状态机的复杂度收拢在 Temporal 一处，API 层就无状态、好扩、好测。代价是发 Signal 是「射后不理」——API 立即返回 accepted，但 workflow 是否真消化了这个 Signal、消化后走到哪，得靠 SSE 那条流反馈，HTTP 响应本身不承诺。这正是「命令与查询分离」（发指令走 REST、看结果走 SSE）在这层的体现。

⑤ 鉴权全景：RS256 + httpOnly cookie + 双 token + admin RBAC

M0「账号地基」把鉴权收敛成唯一一条严格路径：get_current_user 从 httpOnly cookie（atelier_access）读 access token、RS256 公钥验签，取出 sub / tenant_id / role，缺失/无效/过期一律 401。此前并存的 get_current_user_dev_friendly（DEV_MODE 无 token 兜底固定 dev 用户）已删除——不再有绕过 JWT 的分叉。

async def get_current_user(
    access_token = Cookie(alias="atelier_access"),   # httpOnly，同源自动带上
):
    payload = jwt_service.verify_token(access_token)     # RS256 公钥验签，失败→401
    return UserContext(payload["sub"], payload["tenant_id"], payload["role"])
  • 为什么 RS256（非对称）不是 HS256：私钥只在签发方手里，验证方只需公钥——公钥泄露也伪造不出合法 token，为「多处验证、一处签发」留口子。load_signing_key() 是唯一读密钥入口（今天读 PEM 文件，将来换 KMS 只改这一处）。
  • 双 token：access（15min）+ refresh（7天）。POST /auth/refresh 用 refresh cookie 换新 access；refresh 只存 sha256 哈希落 refresh_tokens 表（不存明文），/auth/logout 删行即吊销——纯 JWT 做不到服务端提前吊销，这就是要落库的理由。
  • 身份来源两条：GET /auth/github/login → /callback（GitHub OAuth，state cookie 防 CSRF，首登 _get_or_create_user 建 user + 租户 tenant_id=user.id）；本地 POST /api/dev-login（仅 DEV_MODE 挂载，签 dev 用户的真 token，走与真实登录完全一致的签发/落库/种 cookie 链路）。
  • admin RBAC 第二道闸：require_admin 叠在 get_current_user 之上，role!='admin' 即 403。它与 DEV_MODE 挂载闸正交（AND）：DEV_MODE 只决定 /api/admin/* 存不存在，存在后每个请求仍单独过 role 校验。

SSE 建连是这套 cookie 鉴权的唯一例外：原生 EventSource 带不了自定义 header，又不想让 15min access token 暴露进 URL query——于是先 POST /sse-ticket（走正常 cookie 鉴权 + 查 workflows 验租户）换一枚短命一次性 ticket，再 ?ticket= 建连（见深挖③）。

⑥ admin SSE 多路复用：两条主题合成一条流

整套 /api/admin/* 路由在 router 级挂了 Depends(require_admin)（DEV_MODE 挂载 + role 校验双闸），本端点也不例外。它比生产版 SSE 多一手：要在同一条流里同时看 .event（人话时间线）和 .llm（LLM 调用实时流）。这两条主题刻意隔离（LLM payload 大，不该污染生产订的 .event），但前端想一个流看全。做法是两个订阅各起一个 pump 协程、按 channel 打标塞进同一个队列，下游单流取：

async def pump(sub, channel):
    async for msg in sub.messages: await queue.put((channel, msg.data))
    finally: await queue.put(done)                    # 各投一个哨兵

tasks = [pump(event_sub,"telemetry"), pump(llm_sub,"llm")]
while finished < len(tasks):                          # 两个哨兵都到齐才收尾
    item = await queue.get()
    if item is done: finished += 1; continue
    channel, data = item
    yield {"event": channel, "data": ...}              # 按 channel 命名 SSE event
cleanup race收尾顺序有讲究：先 cancel 并 await 两个 pump 真正停下，再退订两个订阅。反过来的话，pump 可能在 unsubscribe 之后仍尝试从已关闭的订阅取消息，触发清理竞态。此外 admin SSE 纯只读——收到终止事件也不回写 workflows.status（那是生产路径的职责），避免旁路影响生产状态机。
面试可讲 · 取舍点

这层「薄门房」能讲出什么设计品味

不是「我写了些 FastAPI 路由」，而是每个看似简单的决定背后都有一句「为什么必须这样」。

为什么先写 DB 再启 Temporal？

孤儿优于幽灵

先写 DB：若 Temporal 启动失败，只留一条可被定时任务扫出的 孤儿 running 记录。反过来若先启 Temporal，DB 写失败会留下「在跑却查不到」的幽灵，排查代价高得多。

SSE 为什么用 Core NATS 不用 JetStream？

进度可丢，要的是轻

进度事件丢了前端刷新即重订当前态，无需持久化/重放。Core NATS 即发即弃、延迟更低；JetStream 的持久化留给「绝不能丢」的任务派发。

多租户隔离到底靠什么？

RLS + SET LOCAL 下沉

隔离下沉到数据库策略，不靠每条 SQL 手写 WHERE（漏一条就泄漏）。事务级 SET LOCAL 是命门——否则连接池复用会把上个租户的 id 残留给下一个。

为什么 REST 端点这么薄？

状态机收拢在一处

九个 Signal 代理不写任何业务逻辑，状态全归 Temporal。API 无状态 → 好扩好测。代价是发 Signal「射后不理」，结果靠 SSE 反馈——命令查询分离。

上帝视角绕 RLS 不危险吗？

只读 + RBAC + 不挂载

另起只读连接（readonly 由 DB 兜底拒写）、事务级关 RLS、router 挂 require_admin（role 校验）、且仅 DEV_MODE 挂载——生产物理不存在，存在也要 admin。靠结构而非注释。

鉴权为什么是 cookie + RS256？

httpOnly 防 XSS，公钥独立验

access token 落 httpOnly cookie（JS 读不到，防 XSS 窃取），RS256 私钥单点签发、验证方只需公钥。双 token（15min access + 7天 refresh），refresh 落库存哈希可服务端吊销。

SSE 建连凭证怎么带？

一次性 ticket 而非塞 token

原生 EventSource 带不了 header，又不想让 access token 暴露进 URL query。先 POST 换一枚绑 workflow 的一次性 ticket（Redis 30s + GETDEL 取即删），建连时核销——防重放、限范围。

三端为什么统一 data_converter？

契约序列化对齐

API client / workflow worker / agent worker 三端统一 pydantic_data_converter，否则 start_workflow 传的 FeatureWorkflowInput 到 worker 反序列化不出来。跨进程契约必须同一套编解码。
