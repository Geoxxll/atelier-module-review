闭环 Coder

让 Agent 像真人程序员一样：在真实工作区里改一点、跑一次测试、看报错、再改——而不是「闭眼默写整个 PR」。

一句话：把已批准的 PRD + TechSpec 丢进一个长生命周期 Docker 容器，Agent 用文件工具（read/write/edit/glob/grep/bash）增量实现到自测绿。范式从 legacy 的开环一次性生成换成闭环读→改→测→纠偏；沙箱角色从「判分器」转为「隔离边界」。盲评 A/B：闭环 0.90 vs 开环 0.73。

「闭眼默写」和「边写边跑」的差别

给你一道编程题，两种交卷方式：

  • 开环（legacy 做法）——把整个文件从头到尾一次性写完，中途不运行、不看任何反馈，写完直接交。写错了？整份推倒重写。这就是 legacy CoderAgent 干的事：LLM 凭 PRD 一次 structured output 吐出整套 FileChange[]，扔给一次性容器被动判分。
  • 闭环（本模块做法）——先写个草稿，跑一下测试，看到 AssertionError，读报错，改那一行，再跑……逐步逼近正确。这正是现代 coding agent（Claude Code 等）的姿势，也是本模块的核心。

差别不是「谁更勤快」，而是能不能利用环境反馈中途纠偏。开环的质量上限被「第一次就写对」卡死；闭环能爬出错误——但也带来新问题：如果 Agent 把文件改到语法都不对、测试连跑都跑不起来，它就从「纠偏」退化成「蒙眼空转」。这个陷阱（以及堵住它的四道护栏）是本模块最值得讲的一段，见「深挖 ③」。

贯穿全篇的一句话 · 自测绿 ≠ 成功 闭环 Coder 的终止门要求「done 前必须自测绿」，但这只是工程信号、不是成功判定——防的是 Agent 自写自测糊弄绿（self-gaming）。真正的成功由外部裁决：level-0 靠盲评、level-1 靠对抗式 Reviewer 在干净容器独立重跑。这条「Coder 不信自己的自测」是它与 Reviewer 的接缝。

Agent 手里的六把工具

全部工具统一走 LiveSandbox.exec（docker exec）在容器内执行，与 bash 同一隔离边界——保证「工具看到的工作区 == Agent 跑测试看到的工作区」，且产物统一由容器进程写入 bind-mount、宿主机侧属主一致（不出现「一半宿主机 uid、一半容器 root」的混杂）。

专用文件工具不是「bash 一把梭」的替代，而是给高频文件操作干净可控的入口：read 翻页防灌爆上下文、edit 唯一匹配（零/多匹配都 raise）防静默改错——都是 bash 难保证的语义。write/edit 写回 .py 前先 ast.parse，改坏语法当场拒绝、不落盘。

为什么镜像/依赖分两级：开机时 LiveSandbox._provision 按镜像装「通用工具」（python 装 ruff/pytest，零 agent turn）；项目级依赖（requirements/package.json）不在这里——那是 Agent 自己写 manifest 后 bash bulk install。镜像按 TechSpec.tech_stack 选（python:3.11-slim / node:20-slim），greenfield 退默认。

图版 II · 动态流程

一圈 loop 长什么样，护栏在哪一步介入

左边是「健康的一圈」：LLM 决策 → 工具执行 → 观察反馈 → 纠偏。右边是「#57 护栏」：在 Agent 快把可观测性弄坏时把它拉回来。

图版 II-a · 一次完整闭环迭代DYNAMIC · one loop
1
seed · SEED_SYSTEM_PROMPT

第一步先写 PLAN.md（活计划，`- [ ]` 勾选项，边走边改）+ 维护 append-only DECISIONS.md（记 why，跨 context 存活）。开场 = seed system prompt + build_task_prompt 渲染的 (PRD, TechSpec)。

2
CustomReActAgent · LLM 决策

把历史发给 LLM（经 Gateway），拿回 AIMessage(tool_calls=[…])——「我要 edit calc.py 这一行」。不再调工具即触发终止门。

发给模型前先套一层 context editing：老 ToolMessage 裁成占位防爆窗，规范历史仍全量。

3
工具执行 · docker exec

在容器里真执行 edit。写回前先过语法闸门（.py 做 ast.parse）——改坏就当场拒绝、回错误串、不落盘。通过则 base64 落盘。

4
观察 · bash pytest

Agent 跑 pytest，ToolMessage 回灌 {stdout, stderr, exit_code}。看到 1 failed / AssertionError——拿到「算法对不对」的语义反馈。

5
on_step 观察钩子（#57）

每轮工具跑完，build_step_observer 读最近测试结果喂 BestSoFarTracker：见「可 import 且 pass 数更高」就快照工作区；刚烂进无法 import 态就注入归因引导；连续无进展就请求止损。

6
纠偏 → 回到 2，或 → 终止门

红了就带着报错回到第 2 步继续改。Agent 自认为 done（不再调工具）时，finish_guard 检查历史里有没有一次绿测：没有 → 塞回逼单消息继续 loop；有 → 放行。

7
结束 · rollback_if_better

loop 结束（含撞 max_iterations=15）时，若当前磁盘比历史最佳快照差 → 回退到最佳可 import 版本，不 ship 最后被改坏的那份。然后 collect_git_artifacts 捞产物。

三个钩子撑起整圈：finish_guard（终止门，done 前必绿）+ on_step（每轮观察，护栏动作）+ 结束时 rollback_if_better（回退）。前两个是内核 CustomReActAgent.ainvoke 暴露的通用钩子，护栏逻辑注入进去——内核不认识 #57，护栏也不改内核。

四道护栏 · 按杠杆排序

①
语法闸门（预防 · 杠杆最高 · 模型无关）

write/edit 写回 .py 前 ast.parse，改坏当次拒绝、不写回——文件根本不进入无法 import 态，从源头掐断「语义反馈清零」那一环。

②
best-so-far 快照 / 回退（补救）

每见「可 import 且 pass 更高」就快照；结束回退到历史最佳，不 ship 最后被改坏版本。

③
无进展止损

连续 no_progress_limit=4 轮 pass 数不增（且已有 best 兜底）→ 经 on_step 提前停，max_iter 仍兜底。

④
collection-error 归因

文件刚烂进无法 import 态时注入 IMPORT_BROKEN_MESSAGE：「先恢复可导入，别继续盲改算法」。

预防 > 补救：堵住文件烂进 un-importable 态（①），比烂了再回退（②③④）杠杆高得多——且①模型无关，不靠模型聪明。四道按此排序，第一道就把死亡链的拐点掐死。

实证 · 死亡链消除

核心机制 · 钻到实现级

五个撑起闭环 Coder 的机制

展开看代码级细节。这几处也是面试里最容易被追问「你具体怎么实现的」的地方。

① 自研 ReAct 内核 + finish_guard：怎么强制「done 前必自测绿」

CustomReActAgent 是从 legacy fork 出来的通用 ReAct loop——LLM 决策 → 工具分发 → 执行 → _should_continue（还调不调工具）→ max_iterations 兜底。关键在 ainvoke 里给「想结束」这个时刻插了一个钩子：

# CustomReActAgent.ainvoke —— LLM 不再调工具（想结束）时
if not self._should_continue(ai_message):
    if finish_guard is not None:
        injected = finish_guard(messages)
        if injected is not None:
            messages.append(injected)   # 门未放行：塞回逼单消息
            continue              # 不结束，继续 loop
    break                          # 门放行（或没门）→ 结束

闭环 Coder 传的 finish_guard = self._finish_guard，判据不信 Agent 自述「做完了」，而是扫历史里有没有一次「bash 跑测试命令且 exit_code==0」：

def has_green_test_run(messages):
    # 关联 AIMessage.tool_calls（拿命令） ⇄ ToolMessage（拿 exit_code）
    test_call_ids = {tc.id for … if bash 且 _looks_like_test_command(cmd)}
    for m in messages:
        if isinstance(m, ToolMessage) and m.tool_call_id in test_call_ids:
            if _tool_exit_code(m) == 0: return True   # 一次绿即满足
    return False
  • 为什么用历史扫描而非显式 attempt_completion 工具：① 不新增工具面、对 Agent 透明；② Agent 本来就会用 bash 跑 pytest，复用既有信号最省。
  • 极端兜底：Agent 死活不跑测试，max_iterations 始终兜底，guard 永远拦截也不会无限循环。
  • fork 不是抄：内核 loop 逻辑本就是通用 ReAct、与 GitHub/PR 无关；fork 时剥掉 legacy 原文档里的 PR 插图，加了 finish_guard / on_step 两个 keyword-only 钩子——不传时行为与旧版完全一致。
② Docker LiveSandbox：长存活容器、资源上限、栈自动嗅探

与开环 DockerSandbox（一次性容器、跑完即焚、被动判分）的根本区别：LiveSandbox 是 task 级长生命周期容器，整个 task 期间在线供多次 exec。

self._container = client.containers.run(
    image, command=["sleep", "infinity"],   # 空转保持在线
    volumes={workspace_dir: {"bind": "/workspace", "mode": "rw"}},  # bind-mount
    working_dir="/workspace", detach=True, network_mode="bridge",
    mem_limit="512m", cpu_period=100000, cpu_quota=100000,  # 512MB · 1 CPU
)
# 每次动作走 exec，demux=True 分离 stdout/stderr
exec_run(["sh","-c",command], workdir="/workspace", demux=True)
→ ExecResult(stdout, stderr, exit_code)
  • 为什么 demux=True：默认会把 stdout/stderr 合流，拿不到干净的 stderr；分离后测试信号解析才准。
  • 产物捞取与容器解耦：workspace 是宿主机 bind-mount 目录，容器即便已销毁，宿主机侧仍能 git add -A 看 diff、读文件树。
  • 栈自动嗅探：resolve_image 分三层——① TechSpec.tech_stack（结构化权威）优先 → ② detect_stack(files) 按特征文件（pyproject.toml / package.json）嗅探兜底 → ③ 退默认 python 镜像。
  • 即用即焚：run() 的 finally 里 sandbox.stop()（force remove，幂等），无僵尸容器。
③ #57 收敛护栏：四道防线怎么协作

核心洞察：闭环的能力上限 = 自测质量 + 工具会不会把可观测性弄坏。一旦文件烂进无法 import 态，自测从「红(有信息)」退化成「collection error(无信息)」、语义反馈清零，闭环从「纠偏」变「蒙眼空转」。四道护栏各司其职：

信号解析——不信自述，从 pytest 输出抽客观读数：

@dataclass(frozen=True)
class TestOutcome:
    passed: int; failed: int; errors: int
    importable: bool   # False = 收集期崩，语义反馈清零的危险信号

# collection error 判据（#57 拐点）：
collection_error = ("during collection" in text
  or ("collected 0 items" in text and errors > 0)
  or (("ImportError" in text or "ModuleNotFoundError" in text) and passed+failed==0))
importable = not (collection_error or (errors > 0 and ran == 0))

best-so-far 追踪——每轮喂一次结果，见更好就快照，结束回退：

def record_outcome(call_id, outcome):     # on_step 每轮喂，按 call_id 去重
    improved = outcome.importable and (best_passed is None or outcome.passed > best_passed)
    if improved: best_files = snapshot_workspace(...); no_progress_count = 0
    else:        no_progress_count += 1

should_stop = best_files is not None and no_progress_count >= no_progress_limit
def rollback_if_better():                 # 结束时：当前比最佳差 → 还原快照
    if cur.importable and cur.passed >= best_passed: return False   # 已最佳，不动
    restore_workspace(workspace, best_files); return True

① 语法闸门write/edit 写回前 · tools.pyast.parse 挡坏字节，文件不进无法 import 态（预防）
② best-so-far 回退结束 · AgenticCoder.runship 历史最佳而非最后状态（补救）

纯 Python + 真文件系统：信号解析是纯逻辑、快照/回退作用在宿主机 bind-mount 目录，全部可脱离 Docker 与 LLM 单测（test_convergence.py 20 例 + test_file_tools.py 语法闸门 5 例 + test_best_so_far_rollback.py host-FS 假沙箱端到端回退）。

④ 上下文管理：闭环 token 怎么不随轮数无界膨胀

闭环 token 开销不如开环可预测——长任务多轮 read/测试日志会撑爆 context window。核心反转：有了沙箱当真相源，代码/旧工具输出都「在磁盘不在 context」，可随时 re-derive（重新 read / 重跑测试），所以旧 ToolMessage 丢了不可惜。

⑤ 为什么 fork 而非 import legacy：解耦不变量

闭环模块拥有自己的 loop 内核——从 legacy coder-agent/react_agent.py 的 CustomReActAgent fork 出来，而不是 import。这是一条硬不变量：禁止本模块 import 任何 coder_agent.*（legacy 内部）。

fork（采用）import legacy（否决）
依赖方向实验包不依赖生产包实验反向依赖生产
legacy 演进互不牵制，各自改改 legacy 可能崩闭环
A/B 公平性两套物理隔离、独立变量共享代码污染归因
成本一次 fork + 剥 PR 叙述省一次拷贝

闭环管线在赢下盲评 A/B（ADR-0013 闸门）、替换生产链路前，legacy coder-agent / reviewer-agent 原样保留做生产链路 + A/B baseline。两套范式并存、物理隔离——这也是为什么 react_kernel / sandbox / provisioning / pagination 提到模块根：它们是 Coder 与 Reviewer 共用、单向被依赖的叶子，谁都不反向依赖谁。

承重不变量依赖方向 = 实验包（agentic-closed-loop）→ 只依赖 stdlib + langchain + structlog + pydantic + docker + atelier_shared.* + 自身，绝不反向依赖生产包。这条被 ADR-0012 钉死，是「先证后推、不替换生产链路」纪律的物理保证。
面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我做了个 coding agent」，而是每一处取舍背后都有一句「为什么必须这样」。

开环换闭环，到底赢在哪？

能利用环境反馈纠偏

开环质量上限被「第一次就写对」卡死；闭环 读→改→测→纠偏 能爬出错误。盲评 A/B 实证：闭环 0.90 vs 开环 0.73，且优势任务相关——赢在可迭代的状态机题，难推理题上弱模型闭环反而高方差。

为什么自研 loop 不用 Claude Agent SDK？

守不变量 + 最大学习价值

SDK 是 Anthropic-native（认 /v1/messages），会捅破 LLM Gateway 不变量（所有调用经 OpenAI-compat gateway、不持厂商凭证）；且把最该学的肌肉外包。已有能跑的 ReAct，fork 加深不是从零。

Agent 自测绿了就算成功吗？

自测绿 ≠ 成功，防 self-gaming

自测绿只是终止信号（工程门），不是成功判定——不然 Agent 会自写自测糊弄绿。成功由外部裁：level-0 盲评、level-1 对抗 Reviewer 干净容器独立重跑。

闭环最容易在哪翻车？

把可观测性自己弄坏

文件烂进无法 import 态 → 自测收集期就崩 → 语义反馈清零 → 蒙眼空转撞 max_iter。#57 四道护栏按杠杆排：预防（语法闸门）> 补救（回退/止损/归因）。

语法闸门为什么杠杆最高？

预防且模型无关

回退/止损是「烂了再救」，语法闸门让文件根本不进入坏态、从源头掐断死亡链；且是纯 ast.parse、不靠模型聪明——弱模型也照样被接住。

为什么 fork legacy 而不 import？

依赖方向 = 实验不依赖生产

实验包反向依赖生产包会让「改 legacy 崩闭环」「共享代码污染 A/B 归因」。fork 一次 + 剥 PR 叙述，换来两套物理隔离、独立变量——ADR-0012 钉死。
