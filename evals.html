评测子系统

项目的「良心」——每加一层能力，都要能被单独 A/B 证明它真的有用。

一句话：一套不 import 任何 agent 的六接缝 harness 内核，把「跑哪些输入 × 哪些变体 → 评判 → 聚合 → 落盘」抽成正交接缝；上面挂两条互不污染的信号通道——客观 held-out 测试定胜负、LLM 质量裁判只上色。存在的唯一理由：多变量同接会污染归因，所以每层能力都得能被单独证明。

为什么一个多 Agent 系统需要「一套」评测，而不是几段脚本？

项目的核心论点是一句话：「每加一层能力，须能被单独 A/B 证明有用」。给 PM 装 Skill、给它记 SOP、给 Coder 上闭环——每一样都是一个「我觉得会更好」的赌注。不验证，就只是自我感觉良好。

但早期是 5 个 scripts/ab_*.py 各自手搓同一套机器：judge 的 prompt、for i in range(N) 的循环、A/B 交换位置、tally 计票、seed 夹具、落盘。写一个新 A/B ≈ 复制 300 行。更糟的是——每个脚本的评判口径都略有不同，结果之间没法比。

eval 子系统就是把这套重复收敛成正交接缝：一个新 eval ≈ 一份 Case 数据 + 一个 Variant 适配器 + 几条 Dimension 配置。核心洞见有二：

  • 内核不认 agent，只认协议——harness 六接缝里没有一行 import pm_agent，脏的 agent 耦合全隔在外层。所以内核能脱 LLM 单测，也能给任何 agent 复用。
  • 客观信号和主观信号分两条通道——「测试真过没过」是硬事实，用来定胜负；「LLM 觉得哪份质量好」是 vibes，只进台账上色、绝不进闸门。混在一起，LLM 的幻觉印象就会盖章覆盖客观事实。
贯穿全篇的设计信条 · 可归因性优先 能被单独 A/B 证明，是这个模块存在的全部意义。单变量纪律（一次只动一个自由变量）+ 双通道分离（客观 vs 主观）+ eval 库护栏（评测绝不碰生产）——三条铁律都是为「让结论可信、可复现、可归因」服务的。

图版 II · 动态流程

一次 A/B 怎么跑：从 Case 到 LEDGER 的一行

静态图看的是「零件怎么摆」，动态图看的是「一次 run_eval 里数据怎么流」——以及职责为什么这么切：只有 Harness 知道位置，Judge 和 Metric 都能脱位置单测。

图版 II-a · run_eval 一次完整 A/B（闭环 α 双通道）DYNAMIC · one run
1
run_eval(spec) · 入口护栏

先钉死 eval 库：settings.database_url = resolve_eval_database_url()——库名不以 eval 结尾直接 raise。任何 DB client 构造前执行。

Fixture 会 DELETE/seed，护栏挡住「配错 env 把 DELETE 打到生产」。

2
load_cases · Fixture.setup(case)

从 task.json 装配 Case——PRD/TechSpec 进 inputs，held-out 测试进 metadata（绝不进 inputs，防泄漏给被测臂）。Fixture seed 好世界状态。

3
asyncio.gather(N trial 并发)

一个 Case 的 N 个 Trial 用 gather 并跑。前提（铁律 5）：Variant 对 Fixture 只读，Trial 间无共享可变状态。

位置分配用全局序号 case_index*n + trial_index 的纯函数 i%2，与完成顺序无关 → 并行下结果可复现解释。

4
两臂 produce → HeldOutGradedArm

开环臂 run()→ProducedCode、闭环臂 run_closed_loop_task()→ProducedWorkspace。wrapper 归一两种形态 → 走同一个 oracle 评分。没有任何臂自评。

5
客观通道 · held-out oracle

把某臂产物拷进干净新容器 + Case 专属 held-out 测试，复用 Reviewer 的 IndependentTestRunner 重跑，解析 pytest 计数 → ObjectiveOutcome(passed, total)。

D2：自测绿不算数，held-out 干净重跑才算——堵死「写个 assert True 糊弄绿」的 self-gaming。

6
质量通道 · blind judge

同一产物剥成 BlindCode（去 PLAN/DECISIONS.md + 各臂自写测试 + commit 元数据）→ counterbalance 定位 → PairwiseJudge 盲评。Judge 不知位置真相。

7
Metric 聚合（纯函数）

客观：PassRateMeans 折成 per-arm 通过率（headline）；质量：PairwiseTally 数已归属 win/lose/tie。两者都不碰位置、不碰 LLM。

8
Recorder → docs/eval/ab_results/

一次 run 落三样：完整 JSON（配置+逐对裁决）、人读 MD、往 LEDGER.md 追加一行。带 git commit + 模型，供跨运行对比。

职责切分的精髓：整条链里只有 Harness 知道「谁被放在了位置 1」——它做 counterbalance（i%2 确定性交替）+ 反推归属（位置→变体名）。Judge 拿到的是没有身份的 PRD-1/PRD-2，Metric 拿到的是已归属的干净胜负流。所以这两个接缝都能脱位置单测（喂假判定流验聚合），这是「纯函数 Metric」这条铁律的实际兑现。

图版 II-b · 两个防偏机制 · counterbalance / 盲评DYNAMIC · debias

位置偏差 · counterbalance

A
病：LLM 有位置偏好

裁判倾向选「排在前面」的那份。若 A 永远落位置 1，胜负就被位置偏差污染。

B
药：全局序号 i%2 交替

ordinal = case_index*n + trial_index，偶数保持 (A,B)、奇数对调 (B,A)。确定性、非随机。

为何全局：语义 eval 用 4 个 n=1 的 Case，若只按 trial_index 则 A 永远位置 1、偏差不抵消（issue #36）。

C
反推归属留给 Harness

Judge 盲选位置 → Harness 用 (pos1,pos2) 把位置翻回变体名 → Metric 只见干净胜负。

身份泄漏 · 盲评剥离

A
病：过程产物泄漏身份

闭环臂会落 PLAN.md/DECISIONS.md，一眼认出「这是闭环」，原样喂盲评全废。

B
药：strip_to_production_code

剥掉过程文档 + 各臂自写测试 + commit 元数据（从 files 渲染、绝不带 git diff），只留生产代码 → BlindCode。

C
纯函数 · 可单测

剥离是脱 LLM 脱 Docker 的纯函数，BlindCode.__str__ 即裁判看到的全部文本。

两个偏差、两把刀：counterbalance 治「A/B 谁在前」的位置偏差（Harness 用 i%2 确定治）；盲评治「产物泄漏臂身份」的身份偏差（纯函数剥离）。注意它们和 halo（裁判形成整体印象后盖章所有维度）是不同的病——halo 要实测校准，归 L4，L0 不碰。别把三者混为一谈。

核心机制 · 钻到实现级

六个撑起整个模块的机制

展开看代码级细节。这几个点也是面试里最容易被追问「你评测到底怎么保证结论可信」的地方。

① 内核不 import 任何 agent：为什么这条依赖方向是命根子？

harness 六接缝里没有一行 import pm_agent。依赖严格单向：外层具体 eval（脏，import 真 agent）→ harness 内核（只认接缝协议）→ atelier_shared。下层绝不知道上层存在。

# 依赖单向（铁律 1）—— 违反即拒 merge
evals/pm/        ─┐
evals/closed_loop ┼─> evals/harness/  ─> atelier_shared
   (脏:import agent)     (纯净:只认协议)
# harness/*.py 里 grep 'import.*agent' 必须为空

这不是洁癖，是三个具体收益：

  • 可脱 LLM 单测：内核 80% 是纯逻辑（counterbalance、反推归属、tally、verdict 解析），喂假判定流就能全覆盖，不用真跑模型/Docker。
  • 可跨 agent 复用：同一个 harness 既跑 PM 的 SOP eval，也跑 Coder 的闭环 A/B——因为它不认识任何具体 agent，只认 Variant.produce(case) 协议。
  • 脏耦合被隔离：换 agent 配置一律走 Agent.for_eval(...) 工厂注入依赖，禁止 Agent.__new__ 黑魔法——脏的 wiring 全隔在外层 evals/pm、evals/closed_loop。
② 双信号通道：客观定胜负、质量只上色，为什么必须分开？

harness 原本只有一条 LLM judge 通道（Judge 产 verdict → Metric 聚合质量）。但「LLM 觉得哪份质量好」是 vibes，吃不了「测试真过没过」这种硬事实。L2-α 补上第二条非-LLM 客观通道：

Variant.produce(case) → EnrichedArtifact
        ┌──────────────┴──────────────────┐
     .output                        .objective_outcome
        ▼                                  ▼
   LLM Judge                       PassRateMeans
   (质量 · 上色)                     (pass-rate · 定胜负)
   进 LEDGER 不进 gate               headline 对比表 + 台账头

两通道同 spec 共存、从同一产物各取所需——质量 judge 只看 output、客观聚合只看 objective_outcome，互不污染。关键决策（D3）：客观 pass-rate 定胜负、LLM 质量只上色进台账、绝不进闸门。

为什么这条红线不能破α 实测里质量 pairwise 甚至反偏 legacy（LLM 觉得开环代码更简洁）。若让质量定胜负，就会用「看起来简洁」的印象盖章覆盖「闭环实际多过 5 条 held-out 测试」的硬事实。客观通道刻意不走 Judge/Metric 接缝，是 Harness 内一条平行的纯函数路径。
③ held-out oracle：干净新容器重跑，堵死 self-gaming

客观信号的来源 run_held_out_oracle 有两条不可退让的原则，都是为了让「测试过了」这件事跨臂可比、无法迎合：

  • held-out 测试臂看不到：每个 Case 自带、两臂都看不到的真验收测试，住 case.metadata、绝不进 case.inputs。臂无从迎合，只能真做对。
  • 干净新容器重跑：把某臂产物拷进全新 staging 目录、从镜像起重新 provision，复现「换台机器 clone 下来能不能跑」。默认命令显式点名 held-out 文件，臂自带的（可能假绿的）测试不进客观计数。
# 为什么自测绿不算数（self-gaming 漏洞本身）
臂写个  def test_x(): assert True  → 自己容器里绿 ✅ → 糊弄
held-out 干净重跑真验收测试  → 揭穿：0/2（真绿 → 2/2）
# @docker_required 集成测实证：真绿 2/2、糊弄绿 0/2

复用而非另起炉灶：clean-container 重跑直接复用对抗式 Reviewer 的 IndependentTestRunner（同一套镜像/provision/沙箱隔离原语），本模块只在它之上加一层计数解析——Reviewer 只要布尔 passed，α 要 partial-credit 的 passed/total，所以解析 pytest 摘要行拿计数。oracle 本身 arm-agnostic：只认「一份产物 + 一组 held-out」，S4 把它同时套到两臂——这是跨臂可比的前提。

④ eval 库护栏：一次配错，DELETE 就打到生产

eval 的 Fixture 会对 agent_memory / agent_sop 做 DELETE/seed（清场 + 灌夹具）。这些写操作必须落在专属库——否则轻则清掉本地联调数据，重则若 EVAL_DATABASE_URL 配错指向生产库，DELETE 直接打到生产。护栏是一个防呆函数：

def resolve_eval_database_url() -> str:
    url = os.environ.get("EVAL_DATABASE_URL", DEFAULT_EVAL_DATABASE_URL)
    dbname = _dbname_of(url)
    # endswith 而非子串：排除 evaluation_prod / medieval 被误放行
    if not dbname.lower().endswith("eval"):
        raise RuntimeError("拒绝在非 eval 库上跑评测")  # 早爆
    return url
  • 物理隔离：专属库 atelier_eval（make migrate-eval 重建，schema 与主库一致），database 之间互不可见。
  • 入口即钉死：eval 入口在构造任何 DB client 前先 settings.database_url = resolve_eval_database_url()，一次把 SopClient/MemoryClient/蒸馏全钉到 eval 库。
  • 跨 config 坑：distill_pm_sop 等 workflow_engine activity 用的是它自己的 settings.database_url，pre_step 里须一并钉到 eval 库——漏一处就漏一条 DELETE 到主库。

用 endswith("eval") 而非子串匹配是刻意的：防止「中间恰好含 eval」的非测试库（如 evaluation_prod、medieval）被误放行。约定 eval 库名必须以 eval 结尾。

⑤ 单变量纪律 + 模型轴去混淆：一个「饱和假象」的教训

A/B 的灵魂是一次只动一个自由变量。α 里唯一该变的是「闭不闭环」——所以 run_alpha 把 gen_model 同时钉到两臂（test_alpha_model_pinning.py 守着「两臂同模型」）。否则「闭环更好」可能只是「闭环臂偷偷用了更强的模型」。

但单个模型格点的结论≠ 架构真值。#58「模型轴去混淆」发现一个经典陷阱——饱和假象：

模型 / 任务集闭环开环
gpt-4o-mini · 6 题0.900.73（gap +0.17）
gemini · 同 6 题1.000.91（gap +0.09 ⚠）
gemini · 去饱和硬题1.000.26（gap +0.74）

乍看「模型越强、闭环优势越窄（+0.17→+0.09）」，像是闭环价值随模型衰减。一查是假象：那 6 题里 5 题对 gemini 两臂双双 1.0（饱和），是天花板抹平了差距、不是闭环失效。换去饱和硬题（升级求值、半开区间差集），gap 反放大到 +0.74。

这题最值钱的一句饱和任务会掩盖架构价值——评测题太简单，强弱都满分，你会误以为「干预没用」。任务设计的副发现：「规则多」≠ 难（cron/semver 规则多但机械，仍饱和），「单个易错的全局不变量」才难。判官全程固定 gpt-4o 不动、只换生成模型，才能把「架构 × 模型强度」这两个轴拆开。
⑥ Recorder / LEDGER：一次 run 三样产物，为可复现对比而生

每跑一次 eval，Recorder 落三样，各有明确分工：

产物给谁看 · 装什么
JSON给机器 · 完整配置 + 逐对裁决 + 汇总，供跨运行 diff
MD给人 · 带对比 table 的小结
LEDGER.md 一行索引 · 时间/评测名/commit/关键结果，看趋势

两个关键设计：

  • 每行带 git commit + 模型：结果含 LLM 随机性，脱离 commit 和模型看数字没有意义。改了 prompt/skill/SOP 后回看趋势，必须连同「当时是哪个版本、哪个模型」一起读。
  • 路径锚定仓库根、不依赖 cwd：_REPO_ROOT = Path(__file__).parents[2]，产物永远落 docs/eval/ab_results/——台账是项目级评测资产，不埋进 evals/ 代码里（铁律 2：代码进模块、产物留 docs）。

还有一条容错契约：单 trial 失败丢弃、不记 0。若某 trial 因 provider 不可控失败（如 Gemini RECITATION 把输出过滤空、重采样仍不过），_run_trial 丢弃它、不进聚合——记 0 会把「provider 过滤」冤枉成「臂能力差」，污染对比。故 run_eval 实际聚合的 trial 数可能 < n×cases，这是跨 provider 评测鲁棒性的既定代价。

面试可讲 · 取舍点

这个模块能讲出什么「评测品味」

不是「我写了个 A/B 脚本」，而是每一处防线背后都有一句「为什么结论会不可信、我怎么堵的」。

为什么 harness 内核不 import 任何 agent？

依赖单向 = 可测 + 可复用

内核只认接缝协议，脏的 agent 耦合隔在外层。收益：80% 纯逻辑可脱 LLM 单测，同一 harness 跨 PM / Coder 复用，换配置走 for_eval 工厂而非 __new__ 黑魔法。

为什么客观定胜负、质量只上色？

防 LLM 印象盖章事实

α 实测质量 pairwise 反偏「看起来简洁」的开环。若让它定胜负，会覆盖「闭环实际多过 5 条测试」的硬事实。硬信号定输赢、vibes 只进台账（D3）。

held-out 测试为什么放 metadata 不放 inputs？

臂看不到才无从迎合

真验收测试进 case.metadata、绝不进 inputs；干净新容器重跑、显式点名 held-out。堵死「写个 assert True 糊弄绿」的 self-gaming（D2）。

counterbalance 为什么用 i%2 而非随机？

并行下可复现解释

Trial 用 gather 并跑，位置若随机则归属依赖完成顺序、不可复现。全局序号 case*n+i 的 i%2 是纯函数，与完成顺序无关。

eval 为什么要专属库 + 护栏？

一次配错就打到生产

Fixture 会 DELETE/seed 记忆表。护栏 endswith("eval") 拒绝非 eval 库、入口即钉死 URL。安全比方便重要——评测绝不碰 dev/生产。

闭环 0.90 vs 开环 0.73 说明闭环碾压吗？

任务相关，且当心饱和假象

不是均值碾压——赢状态机题、输未收敛的 calc。且要防饱和假象：题太简单强弱都满分会误判「干预没用」，换硬题 gap 从 +0.09 放大到 +0.74。

已知限制 · 刻意推后

哪些没做，为什么可以先不做

限制影响当前状态 / 缓解
统计显著性未做小 n（n=2/3）非 powered，方向可信非结论刻意推后到 L4
Judge 校准 / halo 未测跨维度印象污染未量化L4；L0 只治位置+身份偏差

更远的未来：L3 端到端轨迹评测（评整条 workflow 而非单 agent）· L4 评判器元评测（多 judge / 显著性 / halo 校准）· L5 回归门禁（把 A/B 接进 CI，退化即拦）· L6 在线评测。都建在同一个 L0 六接缝内核上——地基已立，往上加层不用重造机器。
