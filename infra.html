基础设施

整个 Atelier 跑起来所依赖的「地基」——本地一条 make up 拉起的容器栈，加一串按文件名有序的数据库迁移。

一句话：docker-compose 用四个容器（Postgres+pgvector / NATS / Temporal / Redis）拼出运行时地基，迁移链 0001–0006 按文件名顺序把 schema 建齐，Postgres RLS 在数据库层强制租户隔离，atelier_eval 独立库 把评测数据物理隔开——所有操作都收敛到一组 make 目标。

先建直觉

为什么「基础设施」值得单开一版图？

上层每个 Agent 都很聪明，但它们都站在同一块地基上：需要一个库存业务状态与向量记忆、一条消息总线派发任务、一个工作流引擎撑崩溃恢复、一个缓存做限流与短期上下文。基础设施这一层的工作，就是把这块地基一条命令拉起、可复现、可隔离。

它做对了三件容易被忽视、但一旦做错代价极高的事：

  • 一键复现——make up 用 docker-compose 起容器、make migrate 把 6 个迁移按文件名顺序灌进去，任何人任何机器得到字节一致的 schema。
  • 租户隔离下沉到数据库——不靠应用层「记得加 WHERE tenant_id」，而是用 Postgres RLS 把隔离变成数据库层的强制策略：漏写 WHERE 也泄不了别人的数据。
  • 评测数据物理隔离——评测会 DELETE/seed 记忆表，绝不能碰 dev/生产，于是 make migrate-eval 专门重建一个 atelier_eval 库。
贯穿全篇的一条线 · 隔离 这一版所有设计其实都在回答同一个问题：「谁的数据 / 谁的环境，怎么保证不串台」——租户之间靠 RLS，dev 与 eval 之间靠独立库，业务库与 Temporal 之间靠独立 DBNAME，幂等重投之间靠幂等键。隔离是这块地基的第一性设计。

图版 II · 动态流程

从零到能跑：先起容器、再灌 schema、后拉 worker

静态图讲「有什么」，动态图讲「怎么跑起来」。左边是本地 bring-up 的四步；右边是每个请求进 DB 时，RLS 如何在事务内被激活。

图版 II-a · 本地 bring-up · make up → migrate → dev-*DYNAMIC · bootstrap
1
make up · 起容器

docker compose up -d --wait：Postgres / Redis / NATS / Temporal 四类容器起来。--wait 阻塞到 healthcheck 全绿才返回。

Postgres 首次启动会跑 init-scripts 建 vector 扩展；Temporal depends_on: postgres service_healthy 等库就绪才起。

2
make migrate · 灌 schema

for f in infra/migrations/*.sql; do docker exec -i atelier-postgres psql -U atelier -d atelier < $f; done —— 按文件名 0001→0006 顺序执行。

psql 直接跑容器内二进制，不依赖宿主机装 postgres 客户端。0002 靠 ON CONFLICT 幂等，重复 migrate 不报错。

3
make migrate-eval · 建评测隔离库（可选）

跑 evals/ 前先建：DROP DATABASE IF EXISTS atelier_eval → CREATE → 建扩展 → 套同批 migrations。得到与主库 schema 一致的干净空库。

评测会 DELETE/seed 记忆表，必须与 dev/生产物理隔离——迁移非幂等，靠「重建」绕开。

4
make dev-* / dev-all · 拉起进程

各 dev-pm / dev-architect / dev-coder / dev-reviewer / dev-engine / dev-api / dev 分别启动 worker/API/Gateway；dev-all 一键全起。

worker 连的是刚起好的 NATS / Temporal / Postgres；make health 逐个探活。

顺序不能乱：容器（1）→ schema（2）→ 进程（4）是硬依赖——worker 连 NATS/Temporal 前它们必须已 healthy，写库前表必须已建。--wait 和 depends_on service_healthy 把「起序」这件事交给 docker-compose 自己保证，而不是靠人肉 sleep。

图版 II-b · RLS 租户隔离 · 每个事务内激活DYNAMIC · tenant scoping

正常业务路径 · acquire_with_tenant

A
pool.acquire() + 开事务

从连接池取一条连接、开一个 transaction()——RLS 的作用域就是这个事务。

B
SET LOCAL app.tenant_id

SELECT set_config('app.tenant_id', $1, true)——第三参 true = SET LOCAL，只在本事务内生效。

事务结束自动恢复，防止连接归还池后 tenant_id 残留给下一个请求（连接复用泄漏）。

C
查询自动被 RLS 过滤

任何 SELECT/INSERT 走 POLICY … USING(tenant_id = current_tenant_id())，DB 层强制只见本租户行。漏写 WHERE 也泄不了。

可观测台路径 · acquire_service_role

A
刻意的「上帝视角」

开发者可观测台（仅 DEV_MODE）要跨租户列出所有 run——这是刻意的例外，与 acquire_with_tenant 并存。

B
不设 tenant_id + 关 RLS

不 SET app.tenant_id（current_tenant_id() 为 NULL）+ SET LOCAL row_security = off，跨租户读全量。

同样事务级、事务结束恢复，不污染连接池。

C
DB 兜底只读

事务用 readonly=True 开，任何意外 DML 被 Postgres 直接拒绝——「上帝视角但绝不改数据」不止靠约定。

两条路径的对称：业务路径设 tenant_id、开 RLS（严格隔离）；可观测台路径不设 tenant_id、关 RLS、只读（跨租户但不可写）。都用 SET LOCAL 把作用域钉死在事务内——这是「连接池复用 + 会话级变量」这对组合最容易翻车的地方，SET LOCAL 是防泄漏的关键。

端到端 · 走一遍

一台干净的机器，如何从零跑到「一个请求命中 RLS 表」

跟着敲一遍：起容器 → 灌迁移 → 一个写 workflows 的请求进事务、被 RLS 自动 scope。你会看到「隔离」是怎么在最底层落地的。

t2 · 请求进事务

前端 POST /api/projects/{id}/features。API 用 acquire_with_tenant 开事务、SET LOCAL 租户 id，再 INSERT 一行 workflows。

# apps/api · acquire_with_tenant(pool, user.tenant_id)
BEGIN;
SELECT set_config('app.tenant_id', '00000000-…-0001', true);  # SET LOCAL
INSERT INTO workflows (tenant_id, project_id, type, status)
  VALUES ('…-0001', '…-0002', 'feature', 'running');
COMMIT;  # 事务结束 → app.tenant_id 自动失效，连接归还池

INSERT 的 tenant_id 必须 == current_tenant_id()，否则 RLS 的 WITH CHECK 直接拒绝——想「替别的租户写」在 DB 层就被挡。

t3 · RLS 验证隔离

换一个租户身份查同一张表——看不到刚才那行。隔离不是应用层「记得加 WHERE」，是数据库层的策略在兜底。

-- 会话 A：设成租户 …0001
SET app.tenant_id = '00000000-…-0001';
SELECT count(*) FROM workflows;   → 1

-- 会话 B：设成租户 …0009（另一个人）
SET app.tenant_id = '00000000-…-0009';
SELECT count(*) FROM workflows;   → 0   # 同一张表，物理隔离

这就是把租户隔离下沉到 Postgres 的价值：任何忘加过滤的查询也泄不了别人的数据——安全默认打开，而非默认关闭。

核心机制 · 钻到实现级

六个撑起这块地基的机制

展开看代码级细节。这几个点也是面试里「你的多租户/评测隔离/幂等具体怎么落地」最容易被追问的地方。

① RLS 多租户：acquire_with_tenant vs service-role

租户隔离没写在应用层，而是下沉到 Postgres：每张业务表 ENABLE ROW LEVEL SECURITY + 一条 POLICY tenant_isolation USING (tenant_id = current_tenant_id())。current_tenant_id() 是个 STABLE 函数，从会话变量读当前租户：

-- 0001_core_tables.sql
CREATE FUNCTION current_tenant_id() RETURNS uuid LANGUAGE sql STABLE AS $$
  SELECT NULLIF(current_setting('app.tenant_id', true), '')::uuid;
$$;
ALTER TABLE workflows ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON workflows USING (tenant_id = current_tenant_id());

应用层只负责在每个事务里把 app.tenant_id 设对，用 SET LOCAL（set_config 第三参 true）钉死在事务作用域内：

维度acquire_with_tenantacquire_service_role
用途正常业务，严格隔离DEV_MODE 可观测台，跨租户读
app.tenant_idSET LOCAL 成当前租户不设（current_tenant_id() = NULL）
RLS开，策略自动过滤SET LOCAL row_security = off
可写性可读可写readonly=True，DML 被 DB 拒
最易翻车的点连接池会复用连接。若用 SET（会话级）而非 SET LOCAL（事务级）设 tenant_id，事务结束后它还残留在连接上——下一个请求拿到这条连接就看到上一个租户的数据。SET LOCAL + 事务结束自动恢复，是防这条泄漏路径的关键。
② pgvector 混合索引：HNSW + GIN，为什么两个都要

记忆检索不是纯向量，而是语义（向量）+ 全文（BM25）双路召回再 RRF 融合。数据库层为此各建一个索引，融合本身在查询时做、不在 DB：

-- 语义路：HNSW 近似最近邻，余弦距离
CREATE INDEX ON agent_memory USING hnsw (embedding vector_cosine_ops)
    WITH (m = 16, ef_construction = 64);   # 召回率 vs 构建速度的平衡点
-- 全文路：tsv 是 generated column，写入自动维护，无需手动更新
ALTER TABLE agent_memory ADD COLUMN tsv tsvector
    GENERATED ALWAYS AS (to_tsvector('english', title || ' ' || content)) STORED;
CREATE INDEX ON agent_memory USING gin (tsv);
  • HNSW（分层可导航小世界图）给近似最近邻，m=16 / ef_construction=64 是 pgvector 推荐的召回/构建平衡点。
  • generated tsvector + GIN：全文列由数据库自动派生，写入无需手动同步；GIN 是倒排索引，适合 ts_rank 全文打分。
  • RRF 融合放在应用层：DB 只负责各自路的排序，两路合并（倒数排名相加）在 memory.py 的 SQL 里做——DB schema 只提供「两个可检索维度」。
③ 3072 维超限：HNSW 撞上 pgvector 2000 维天花板

embedding 列声明为 vector(3072)（对齐所用嵌入模型的维度），但 pgvector 的 HNSW / IVFFlat 索引对可索引维度有 2000 的上限。3072 > 2000，索引构建在这个维度上并不理想——这是登记在案的技术债 #7。

现状与缓解当前规模下记忆行数不大，即便索引退化为暴力扫描也可接受，故先记债、不阻断。真正的解法有两条路：降维（存 ≤2000 维的投影向量供索引，全维仅用于精排）或换嵌入模型 / 换索引方案。核心教训：schema 里写死的维度会和索引能力耦合，选嵌入模型时要连带确认向量库的可索引上限。
④ 幂等表：一条 uuid5 键挡住「Temporal 重试 × JetStream 重投」

投递链路上有两处会重复：Temporal Activity 会 retry，JetStream 会 redeliver，两者叠加导致同一任务被投多次。若每次都跑一遍 LLM，既费钱又可能产出不一致。agent_task_results 用一条幂等键短路：

-- 0005_idempotency.sql
CREATE TABLE agent_task_results (
    task_id     TEXT PRIMARY KEY,   # uuid5(workflow_id, salt) 确定性生成
    result      JSONB NOT NULL,
    created_at  TIMESTAMPTZ NOT NULL DEFAULT now()
);

worker 处理前先按 task_id 查此表——命中就直接用缓存 result 完成 activity，不重跑 LLM。关键前提是 task_id 用 uuid5(workflow_id, salt) 确定性生成（不是 uuid4），同一逻辑任务无论重投几次都算出同一个键，才能被主键唯一性拦住。

刻意不开 JetStream publish-dedup：那会误删重试消息导致死锁。幂等做在消费层（查表短路）而非投递层，是这个决策的核心。

⑤ 评测隔离库：为什么用「DROP→CREATE 重建」而非清表

评测的 Fixture 会 DELETE / seed agent_memory、agent_sop 等表——这些操作绝不能碰 dev/生产数据。护栏是一个专属库 atelier_eval，每次评测前 make migrate-eval 把它整个重建：

# Makefile · migrate-eval（要点）
pg_terminate_backend(...)                # 先断开 eval 库上的活动连接
DROP DATABASE IF EXISTS atelier_eval;    # 整库删
CREATE DATABASE atelier_eval;
CREATE EXTENSION vector; pgcrypto; "uuid-ossp";  # 扩展是 per-database，须显式重建
for f in migrations/*.sql: psql -d atelier_eval < f  # 套同一批迁移
  • 为什么重建而非清表：迁移非幂等（CREATE TABLE 会在已存在时报错），「DROP→CREATE→套迁移」每次得到与主库 schema 一致的干净空库，绕开幂等问题。
  • 扩展 per-database：CREATE EXTENSION vector 只在建它的库里生效，新库必须显式再建一遍，否则 vector(3072) 类型不存在。
  • 物理隔离 > 逻辑约定：不同库天然读不到对方数据，比「小心别删错表」这种约定强得多——评测护栏被结构性保证。
⑥ 容器编排：healthcheck / depends_on / --wait 的起序纪律

四个容器有真实的起序依赖：Temporal 要连 Postgres、worker 要连全部三者。docker-compose 用三样东西把「起序」这件事自动化，而非靠人肉 sleep：

机制作用示例
healthcheck容器自报「我真的能服务了」Postgres pg_isready、NATS GET :8222/healthz、Redis redis-cli ping
depends_on: service_healthy被依赖者 healthy 前不起自己Temporal 等 postgres: condition: service_healthy
up --wait阻塞命令到全部 healthy 才返回docker compose up -d --wait（make up）
restart: unless-stopped崩溃自动拉回，除非手动停四个核心服务都挂

资源与持久化上，Postgres/Redis/NATS 各挂具名卷（down 保留数据、down-v 才清）；GitHub MCP 是可选服务，distroless 镜像无 shell 故不做容器内健康检查、依赖 running 即可。Temporal 复用 Postgres 但走独立 DBNAME=temporal，业务库与引擎库同容器、不同库。

面试可讲 · 取舍点

这块地基能讲出什么「工程品味」

不是「我用 docker-compose 起了几个容器」，而是每个隔离/起序/幂等的决策背后都有一句「为什么必须这样」。

多租户为什么用 RLS 而非应用层过滤？

安全默认打开

RLS 把隔离下沉到 DB：漏写 WHERE 也泄不了。应用层过滤是「默认关闭、记得加才安全」，RLS 是「默认打开、要绕才需显式关」——安全的默认方向相反。

SET vs SET LOCAL，差在哪？

连接池复用泄漏

连接会复用。会话级 SET 的 tenant_id 会残留给下一个请求 → 跨租户泄漏。SET LOCAL 事务级、结束自动恢复，把作用域钉死在单个事务内。

迁移顺序靠什么保证？

文件名即执行序

*.sql glob 字典序执行，前缀 0001–0006 就是顺序。纪律靠命名而非工具——简单但要求「新迁移必须递增编号」这条铁律不破。

评测隔离为什么另起一个库？

物理隔离胜过约定

评测会 DELETE/seed 记忆表。用独立 atelier_eval 库，DROP→CREATE 重建——不同库天然读不到对方数据，比「小心别删错」强得多。

幂等为什么做在消费层？

确定性键 + 查表短路

Temporal retry × JetStream redeliver 会重复投递。uuid5 确定性 task_id 做主键、消费前查表短路，不重跑 LLM。刻意不开 publish-dedup（会误删重试消息死锁）。

起序靠 sleep 还是别的？

healthcheck + depends_on

不用人肉 sleep。容器自报 healthcheck、depends_on: service_healthy 卡起序、up --wait 阻塞到全绿——起序变成 compose 的声明式保证。
