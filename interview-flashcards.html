概念 01 · 主旨 · 贯穿全项目

Temporal async completion + 确定性铁律

整个多 Agent 流水线的持久化执行底座——简历领衔条、贯穿全项目的主旨。

面试官会问
"Activity 派任务给 Agent，为什么不同步等它干完？还有 Temporal 那条『workflow 里不准 uuid4/now』是为什么？"
30 秒脆讲法
Activity 不同步等：NATS at-least-once 派发后 worker 立刻释放去处理别的（吞吐）；Agent 异步完成经 async completion 回写 event history + 放新 task 通知 worker 读结果。确定性铁律：有副作用 I/O 必须 Activity 包裹，workflow 只放确定性编排——回放时从 event history 拿已记录结果、不重入（否则 uuid4/now 回放值不同→历史不匹配→非确定性错），所以 task_id 用 uuid5。
动手：切两种模式，看 worker 那条命
深挖锚点
  • async completion 三件套：task_token 塞 payload · raise_complete_async() 挂起 · handle.complete() 回填。
  • 幂等：消费层按 task_id(uuid5) 查结果表，命中即短路、不重跑 LLM。
  • 与「自己写状态机 + 重试」的区别：Temporal 免费给你持久化 + 断点重放 + 可溯源的 event history。
概念 02 · 核心

闭环 Coder：一次性生成 vs 读改测自愈

你项目的皇冠——不是「用了个框架」，而是「debug 过范式本身的失效模式」。

面试官会问
"你把 open-loop 重构成 closed-loop，到底为什么更好？好在哪个具体机制上？"
30 秒脆讲法
开环一次性思考 + 全量丢出，出错没有第二次机会。闭环基于 ReAct 内核（思考→行动→观察→再思考），出错能读代码解析 error 自愈。但闭环不是免费午餐——我在 eval 里发现它有一个范式独有的死亡链：写代码出语法错误 → 语法错误挡住语义反馈 → agent 失去「逻辑对不对」的口径 → 盲改 → 撞 max_iter → 零分。我上了收敛护栏（提交前语法闸门 + best-so-far 回退），held-out 通过率 0.73 → 0.90。

深挖锚点

  • 为什么是闭环独有的病：开环只思考一次就丢出，根本不进「迭代编辑」这条能被语法错误毒化反馈的循环。
概念 03 · ★ 表达天花板

饱和假象：模型变强，闭环就白做了？

最危险的追问。内容你懂，练的是「30 秒讲脆」。

面试官会问
"模型越来越强，一次生成成功率越来越高，你辛辛苦苦搭的闭环 + 护栏，边际价值不就趋近于零？"
30 秒脆讲法（必背）
换强模型 gap 收窄，乍看像闭环要过时——但那是测量假象（saturation illusion）。正确的控制变量是架构，得同时钉住 model 和任务难度。我只升了 model 没升难度，两臂双双饱和到 1.0，gap 自然塌——这是天花板效应，不是架构失效。把任务重新加难去饱和，gap 立刻回来、放大到 +0.74。结论：闭环价值不随模型变强衰减，是任务饱和把它藏起来了。
动手：拖两根轴，看 gap 什么时候消失、什么时候回来

深挖锚点
  • 强模型 × 简单任务 → 两臂撞 1.0，gap≈0 = 假象；强模型 × 难任务 → gap 反而放大。
  • 戳破手法：加 SWE 式难题去饱和，重跑子集 n=3，gap +0.09 → +0.74。
  • 一句镇场：「先验：架构改进要证明它不随模型能力衰减；饱和任务只是掩盖，不是消解。」
概念 04 · 范式

对抗式 Reviewer：防 self-gaming

对抗 = 干净屋独立重跑 + 上下文隔离。Agent 给自己作业打分，天然不可信。

面试官会问
"你这 Reviewer 为什么叫『对抗式』？跟『让另一个 LLM 帮我 review』有什么本质区别？你在防谁作弊？"
30 秒脆讲法
成功判定不信 Coder 自测（self-gaming：agent 给自己打分不可信）。两个隔离：① 干净容器独立重跑——防 Coder 把容器跑脏让测试假绿（如把 API key 塞全局环境变量不写 .env，脏容器过、干净容器一重跑露馅）；② 上下文隔离——不喂 Coder 推理链，防 Reviewer 被带偏、丧失独立视角。
动手：同一个作弊，两种 Reviewer 的结果天差地别
场景：Coder 把 API_KEY 设成全局环境变量（没写进 .env）就交了。

  • eval 曲折（最亮）：首跑对抗式净有害 −0.23 → 查出是我 Reviewer 的 bug 不是范式（反射性拒绝 / 不提交结构化裁决）。

概念 05 · 并行

并行编排 + 确定性归并升级梯

切 / 并 / 合三段都为「可行性」服务。确定性高危环节，能用确定性解的绝不引入 LLM。

面试官会问
"并行开发最大的敌人是两片各自都对、合起来却冲突。你怎么保证切片能并行、又能合得起来？冲突了不就卡死？"
30 秒脆讲法
切：Planner 逻辑切分给 blocked-by 依赖，「不为并行而并行、该串行绝不破坏」，软自纠 + 硬自纠（查环依赖/需求覆盖/脚手架）。并：wave 内 fan-out，每片独立 worktree。合：依次并入集成头，只接受测试+回归全绿。冲突不是 give up，而是确定性起步的升级梯：git 3-way 干净合 → LLM 调和文本冲突 → fix-Coder 修门红（≤1 次）→ 人工闸。Merge 只上报、从不做决定。

  • 精度点：reconciler 治文本冲突，fix-Coder 治回归门红（语义/测试失败）——两根轴，不是一根。
  • 确定性买到什么：强一致 + 可重放；把不确定性（LLM）关在「确定性真解不了」之后才放出来。
概念 06 · ★ 表达天花板

三类记忆：同源，凭什么拆三套

情景 / 语义 / 程序性，按信号源 · 形态 · 注入语义刻意切分。最深一层：fact vs preference。

面试官会问
"语义记忆和 SOP 都来自 PM 对话，不就是一回事吗？为什么非要两套系统？这是不是过度设计？"
动手一：切三个标签，看三维对照
动手二：fact vs preference —— 为什么注入通道必须相反
fact：用户在做 B2B 报销 SaaS preference：PRD 别出 User Story 段
深挖锚点
  • fact = 描述性：只提供背景不发号施令 → 弱注入（可能过期，冲突以本次为准），绝不覆盖当前设计。
  • preference = 规定性：主动塑造产出形状 → 进 实时指令 > SOP > Skill 优先级链，职责是「用户沉默时的默认」。
  • 更新机制差异（次要但要提）：语义紧跟事实、冲突以本次为主；SOP 有置信计数器缓冲抗抖动。
概念 07 · 方法论脊柱

Eval：怎么破「AI 给 AI 打分」的循环

eval 的最强火力穿在①②身上——是它帮你发现死亡链、量化 Reviewer regime。

面试官会问
"你用 LLM 当裁判，那不就是让 AI 给 AI 打分、循环论证吗？你怎么破这个循环？"
30 秒脆讲法
硬判定不靠 LLM——来自 held-out oracle（agent 没见过的确定性测试）；LLM 只做软质量上色、不下硬裁决，再加盲评 + 顺序 counterbalance。而且每加一层能力必须单变量 A/B——我自己就踩过：α 评测同时变了架构和模型强度，差点得出「闭环没用」的错误结论，拆开才发现是饱和假象。

  • 台账（LEDGER）：改 prompt/skill/SOP 后连同 commit + 模型一起回看趋势。
