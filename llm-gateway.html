Plate 04 — LLM Gateway

LLM Gateway

所有 Agent 与所有 LLM 之间的唯一收费站——对内是「一个 OpenAI 端点」，对外按 model 名分流到多个 provider。

一句话：Gateway 是一个 OpenAI 兼容代理。Agent 永远只当它是「一个 OpenAI 端点」，凭证只在这里、绝不落到 Agent 手上；换/加模型（OpenAI ↔ Gemini）只是网关内部按 model 名前缀分流的事，Agent 代码一行不动。provider 的协议怪癖（Gemini 两处坑）也在这一层透明消解。

为什么所有 LLM 调用都要挤过一个网关？

想象一座桥。桥这头是四个 Agent（PM / Architect / Coder / Reviewer），桥那头是几家不同的 LLM 供应商（OpenAI、Google）。你有两种修法：

  • 放任每个 Agent 自己过河——各自揣着 API key、各自知道该打哪个供应商的地址、各自处理 Gemini 那些奇怪的协议怪癖。四个 Agent × 两家供应商 = 八份重复又易错的胶水代码，而且八个地方都握着密钥。
  • 只修一座收费站 + 翻译台——所有人先到网关，网关统一收票（鉴权）、看目的地分流（路由）、翻译供应商方言（补偿）、记账（计量），再放行。

Atelier 选第二种。这个唯一的收窄口（choke point）之所以值钱，是因为它把四件本该散落各处的横切关注点收敛成了一处：① 凭证只在网关（ADR-0009 不变量，Agent 永不持 key）；② 路由——换/加模型是网关内部的事，Agent 无感；③ 计量——token/成本/延迟在一处集中采集；④ 补偿——供应商协议怪癖收敛在这层，不污染上面的 LangChain / Agent 代码。

贯穿全篇的设计不变量 · Agent 无感 对 Agent 而言，Gateway 永远表现为「一个普通的 OpenAI 端点」（baseURL 指向 :3001/v1）。它内部是一家还是三家供应商、Gemini 有没有坑，Agent 完全看不见也不该看见——这份「provider 无感」是本模块所有设计的出发点。

路由表：只看 model 名的第一个词

图版 II · 动态流程

一次调用怎么走完；Gemini 多轮怎么被「悄悄修好」

左边是一条普通 chat 请求的完整生命周期（含路由决策与计量收尾）；右边是 Gemini thinking 模型多轮工具调用时，网关如何在续轮偷偷把 thought_signature 补回去。

4
adapter.completeRaw(body) · openai.ts

去掉 stream 字段，整个 body 透传给上游 client.chat.completions.create。仅 Gemini 实例在此前后做 inject / RECITATION 重采样 / harvest（见 II-b）。

5
extractUsage → cost-meter.record

从响应 usage 读 token，命中 prompt_tokens_details.cached_tokens 走 cachedInput 价，算出 costUsd，连同 provider/model/tenant/延迟落 llm_call_completed 日志。

6
c.json(result.raw) → Agent

把上游 OpenAI 原始响应结构原样返回。OpenAI SDK / LangChain 直接解析，全程感知不到中间发生过路由、补偿、计量。

关键节奏：校验 →（路由）→ 透传 → 计量 → 原样回传。透传是主轴，路由/补偿/计量都是「贴在透传两侧」的横切逻辑——正因为它们不改 body 的语义，Agent 才对整条链路无感。流式模式同理，usage 也已修——streamRaw 经 onUsage 回调收末个 chunk 的真实 usage、落真实成本（上游没吐 usage 时兜底 0）。

端到端 · 走一遍

一次 gemini-3.5-flash 带工具的评测调用，全程会遇到什么

把路由 → Gemini 端点 → 续轮回填签名 → RECITATION 升温重采样 → 返回，串成一个真实故事。这一条把本模块最有料的两处坑都踩了个遍。

场景 · α 评测生成臂走 Gemini、判官仍 gpt-4o

eval 让 gemini-3.5-flash 用 ReAct 工具写一段 LRU cache，判官用 gpt-4o 打分

路由

eval 用 OpenAI SDK 打 :3001/v1，body 里 model="gemini-3.5-flash"、带 tools。网关 providerForModel 一看前缀 → gemini adapter。同一轮里判官的 gpt-4o 请求前缀不匹配 → 走 openai adapter，两臂在一个网关内共存。

# router.ts providerForModel
"gemini-3.5-flash".startsWith("gemini-")  # → true → provider=gemini
"gpt-4o".startsWith("gemini-")            # → false → provider=openai（判官固定不动）
第 1 轮

gemini adapter 透传 body 到 Google OpenAI-compat 端点。Gemini 返回一个 tool_call，带 thought_signature。网关 harvest 按 id 缓存签名后原样回传——但 LangChain 解析成 AIMessage 时把 extra_content 丢了。

# completeRaw 收尾
this.signatureCache?.harvest(resp)
# cache: { "call_abc123": { google: { thought_signature: "…" } } }
第 2 轮 · 修签名

ReAct 把 tool 结果拼回历史续轮。出站 assistant 的 tool_call 没有签名（被 LangChain 吃了）。网关 inject 一看 id 命中缓存 → 就地回填，Gemini 才不会 400。

# completeRaw 开头
const injected = this.signatureCache?.inject(params) # → 1
logger.info("gemini_signature_injected", { count: 1 })
# 缺这一步：400 "Function call is missing a thought_signature"（根 ERROR #14）
RECITATION

Gemini 生成到「教科书式 LRU cache」时撞了 RECITATION content-filter：返回 200 但 content/tool_calls 空、finish_reason=content_filter，令 structured-output 解析崩。网关检测到 → 有限次升温重采样绕过。

# RECITATION 在 temperature>0 下非确定 → 升温重试（≤2 次）
for (attempt = 1; attempt <= 2; attempt++) {
  if (!finish_reason.includes("content_filter")) break
  logger.warn("gemini_content_filter_retry", { attempt })
  resp = create({ ...params, temperature: 0.6 + attempt*0.2 })  # 0.8 → 1.0
}
# 非 gemini 实例 maxFilterRetries=0，行为不变（根 ERROR #15）
计量 + 返回

拿到有效响应，extractUsage 读 token 算成本，落 llm_call_completed 日志（带 provider=gemini / model / tenant / 延迟）。原始响应原样返回给 eval。判官那条 gpt-4o 走的是干净的 OpenAI 路径，一处补偿都不碰。

整条链路里，eval 代码只知道「我调了个 OpenAI 端点、换了个 model 名」——签名回填、RECITATION 重采样、成本计量全在网关内悄悄发生。这就是「Agent 对 provider 无感」的完整兑现。

核心机制 · 钻到实现级

五个撑起整个网关的机制

展开看代码级细节。这几处也是面试里最容易被追问「你具体怎么做的」的地方。

① 按 model 名路由：一个纯函数 + 一张 adapter 表

「选哪个上游」被刻意收敛成一个无副作用的纯函数——只看 model 名前缀，不碰任何 I/O，因此可单测、可扩 provider：

// 纯函数：只看 model 名决定 provider
export function providerForModel(model: string): Provider {
  return model.startsWith('gemini-') || model.startsWith('models/gemini-')
    ? 'gemini' : 'openai'
}

装配则由 buildAdapterRouter 按「手头有哪些 key」建表——openai 必有，gemini 仅在配了 key 时存在；两个都是同一个 OpenAIAdapter，差异只是 baseURL + key（+ Gemini 挂补偿）：

const openai = new OpenAIAdapter(openaiApiKey)
const gemini = geminiApiKey
  ? new OpenAIAdapter(geminiApiKey, {
      baseURL: GEMINI_OPENAI_BASE_URL,
      geminiSignatureFix: true,   // 只有 Gemini 实例挂补偿
    })
  : undefined

return (model) => {
  const provider = providerForModel(model)
  if (provider === 'gemini' && !gemini)
    throw new Error(`需要 Gemini 但 GEMINI_API_KEY 未配置`)  // 配错早爆
  return provider === 'gemini' ? { adapter: gemini, provider } : { adapter: openai, provider }
}

加一个 provider ≈ 加一条前缀判定 + 一行 new OpenAIAdapter(key, { baseURL })——前提是它有 OpenAI-compat 端点。这就是「换 provider = 换 baseURL + key」这句口号的物理落点。

② thought_signature 回填：跨 LangChain 序列化把签名找回来

Gemini thinking 模型每个 tool_call 带不透明 extra_content.google.thought_signature，多轮续轮必须原样回传，否则上游 400。麻烦在于：langchain_openai 把响应解析成 AIMessage 时只留标准字段（id/name/args），extra_content 被丢弃。网关是唯一同时看得见「原始响应」与「出站请求」的层，于是做一层最小记忆——按 id 缓存、按 id 回填：

// harvest：从 Gemini 响应按 tool_call.id 缓存 extra_content
harvest(resp) {
  for (const tc of resp.choices[].message.tool_calls)
    if (tc.id && tc.extra_content != null) this.remember(tc.id, tc.extra_content)
}
// inject：出站 assistant 的 tool_call 缺签名但 id 命中 → 就地回填
inject(body) {
  for (const m of body.messages) if (m.role === 'assistant')
    for (const tc of m.tool_calls)
      if (tc.id && tc.extra_content == null && this.store.has(tc.id))
        tc.extra_content = this.store.get(tc.id)   // ← 补回来
}
  • 按 id 键：Gemini 生成的随机短 id，跨对话碰撞概率极低；真碰撞最坏是该次 400 被上层重试，不会静默错答。
  • FIFO 上限 5000：防内存泄漏；超限删最旧。缓存是「无状态服务外的最小记忆」，不落库。
  • 调试教训：上游 OpenAI-compat 的 400 常被 SDK 吞成「no body」，真错误要直连端点带 Authorization: Bearer 才看得到。
③ RECITATION 升温重采样：绕开「教科书码」过滤器

Gemini 的 RECITATION content-filter 会拦「逐字复述训练数据」的输出——写规范算法（LRU cache、罗马数字、经典 LeetCode）高度撞库即触发。表现是返回 200 但 content/tool_calls 空、finish_reason=content_filter、completion_tokens=0，让 LangChain structured-output 当场崩。它独立于可配置的 HarmCategory，基本关不掉。

const maxFilterRetries = this.signatureCache ? 2 : 0   // 仅 Gemini 实例 > 0
let resp = await create(params)
for (let attempt = 1; attempt <= maxFilterRetries; attempt++) {
  const fr = resp.choices?.[0]?.finish_reason
  if (!fr || !fr.includes('content_filter')) break          // 没被过滤 → 收工
  logger.warn('gemini_content_filter_retry', { attempt, model })
  resp = await create({ ...params, temperature: 0.6 + attempt * 0.2 })  // 升温重采样
}
  • 为什么升温有效：RECITATION 在 temperature>0 下非确定——换个采样温度就可能不撞库。故只对被过滤的少数次升温（0.8 → 1.0，≤2 次），落日志便于量化这个混淆变量。
  • 非 Gemini 零影响：openai 实例 maxFilterRetries=0，循环一次不进，行为与从前完全一致。
  • 评测启示：provider 的安全/合规过滤会成为跨模型评测的混淆变量（Gemini 拒写教科书码 ≠ 它不会写）——harness 侧还得对不可恢复失败「丢弃而非记 0」。
④ cost-meter：每次调用后把 token 折成钱，落日志 + 落库

计量在透传收尾处发生：extractUsage 从响应 usage 读 token，命中缓存 token 走 cachedInput 价，按定价表折算成 costUsd，再由 CostMeter.record 落一条结构化日志、并 await sink.insert 写 Postgres llm_calls：

function calcCost(model, tokensIn, tokensOut, cachedTokens) {
  const price = PRICING[model] ?? { input: 5.0, output: 15.0, cachedInput: 2.5 }  // 未知 model 兜底价
  return ((tokensIn - cachedTokens) * price.input
        + cachedTokens * price.cachedInput
        + tokensOut * price.output) / 1_000_000                // USD / 1M token
}
// CostMeter.record → 结构化日志（Loki 聚合）+ 落库 llm_calls
logger.info('llm_call_completed', { provider, model, tenantId, workflowId,
  taskKind, tokensIn, tokensOut, cachedTokens, costUsd, latencyMs, cacheLayer })
await this.sink?.insert(rec)   // PgLlmCallsSink → INSERT llm_calls（缺 sink 时退化纯日志）
  • 元数据走 Header 不进 body：tenant/workflow/taskKind/trace 从 X-Atelier-* 自定义 Header 读，绝不污染 OpenAI 格式的请求体。
  • 已落库（现在时）：record() 里 await sink.insert(rec) 真写 Postgres llm_calls（PgLlmCallsSink，供配额预检 / 成本看板聚合）；需装 pg 依赖 + DATABASE_URL，缺 sink 时退化为纯日志、不阻塞网关。CostMeter 只依赖窄接口 LlmCallsSink，换实现（再叠 ClickHouse 等）不动调用方。
  • 已知债：定价表手工维护，需定期对齐官方价；tenantId 非 uuid（Header 缺省兜底 unknown）时跳过写库、只留一条 warn。
⑤ 为什么是「网关不变量」＋ OpenAI-compat 表面

这不是「顺手加个代理」，而是一条被 ADR-0009 钉死的架构不变量：所有 LLM 调用必须经网关，Agent 代码不直接持有 provider 凭证。它换来四件事一次到位：

不做网关（各自直连）做网关（唯一收窄口）
凭证散落在每个 Agent 进程只在网关一处
换/加 provider改所有 Agent改网关内部，Agent 无感
成本可观测各测各的、口径不一一处集中采集，统一口径
provider 怪癖污染每个 Agent / LangChain收敛在 adapter 层透明消解

而「表面为什么选 OpenAI-compat」也是精算过的：OpenAI 的 chat/completions 协议已是事实标准，各 provider（含 Google）都提供 OpenAI-compat 端点，LangChain / OpenAI SDK 开箱即用。选它做统一表面 = 白拿整个生态的 tools / response_format / 结构化输出——网关只做透明代理就能让上层所有特性照常工作。这也解释了为什么后端唯独这一个服务保留 TypeScript（ADR-0009：已稳定、HTTP 边界隔离，不值得重写）。

面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我写了个代理」，而是每一处收敛/隔离背后都有一句「为什么必须这样」。

为什么所有调用都挤过一个网关？

横切关注点收敛

凭证、路由、计量、provider 补偿四件横切的事，散在每个 Agent 就是 N×M 份易错胶水且 N 处握密钥。一个收窄口把它们一次收敛，是 ADR-0009 的核心。

路由为什么只看 model 名？

唯一自由变量

α 评测要「生成臂换 Gemini、判官仍 gpt-4o」在一个网关共存——按 model 名分流让换生成模型只改一个入参，判官天然不动（#58 模型轴去混淆）。

配错 provider 该报错还是兜底？

宁可早爆不静默

点 gemini-* 没配 key → 直接 400，绝不静默打到 OpenAI。静默降级会让「本想跑 Gemini 却偷跑 OpenAI」污染一整轮评测读数，比崩掉更坏。

Gemini 的坑为什么修在网关？

唯一能看见的层

签名被 LangChain 吃掉后 Agent 侧无从补——网关是唯一同时看见「原始响应」和「出站请求」的层。怪癖收敛在 adapter，Agent 保持 provider 无感。

透传代理为什么只做最小校验？

不拦截即不误伤

多校验一个字段就可能拦掉 tools / response_format，让 LangChain 结构化输出失灵。passthrough 一切非必需字段——白拿整个 OpenAI 生态。

计量落库为什么走窄接口注入？

依赖倒置 · 可测

CostMeter 只认 LlmCallsSink 窄接口：生产注入 PgLlmCallsSink 真写 llm_calls，单测传 fake 断言写入。缺 sink（没配 DATABASE_URL）时退化纯日志、不阻塞网关启动——换实现不动调用方。

已知限制 · 刻意推后

哪些没做，为什么可以先不做

限制影响当前缓解 / 计划
成本聚合视图缺cost 已落 llm_calls，但无 per-Agent dashboard原始行齐了，聚合 / 看板待建；配额预检已先用上
流式无签名 harveststreamRaw 只 inject 不 harvestharvest 仅非流式做；闭环 Agent 非流式，不受影响
定价表手工维护官方调价后成本读数会偏PRICING / EMBEDDING_PRICING 需定期对齐
仅 OpenAI-compat 形态真正异构 provider（Anthropic 原生）不可解需新 adapter，非换 baseURL 可覆盖
GEMINI_API_KEY 未进 .env.example新人不知这个可选 env待补文档
两个联调必踩的坑① .env 改了要重启：tsx watch 只热更代码、不重读 --env-file——加/改 key 后老进程仍用旧值，必须 ctrl-C 重起。② localhost 走 IPv6：Python httpx 把 localhost 解析成 ::1，连不上只听 IPv4 的网关——联调时 baseURL 用 127.0.0.1 而非 localhost。

更远的未来：per-Agent 成本拆分看板（cost 已落 llm_calls，但聚合视图 / dashboard 尚缺）· 真正异构 provider adapter（如 Anthropic 原生 API，非换 baseURL 可覆盖）· 定价表自动对齐官方价。当前的接口设计（透传 + adapter 表 + CallRecord + LlmCallsSink 窄接口）已为这些留好扩展点，换实现不动调用方。
