记忆系统

给 Agent 装上跨会话的长期记忆——三类记忆，各记一种「知识」，互不串台。

一句话：情景记忆记「过去做过什么」、程序性 SOP 记「用户希望怎么做」、语义记忆记「关于用户/产品是什么」。每一类独立写入、独立检索、独立注入——刻意不合并，才能各自被 A/B 单独证明有用。

为什么一个 Agent 需要三种记忆？

想象一个刚入职的资深工程师同事。他有用没用，取决于他能不能记住三件性质完全不同的事：

  • 过去做过的活——「上次那个报销审批我是怎么切模块的？」这是情景记忆，一段段带时间的经历。
  • 你这个人的做事口味——「这位老板的 PRD 一定要用 checklist 写验收标准，别用大段文字。」这是程序性 SOP，是怎么做的偏好。
  • 关于你和你产品的事实——「他们在做 2 人团队的 B2B 报销 SaaS，提过想要审计日志。」这是语义记忆，是是什么的离散事实。

这三种记忆的形态、强度、注入方式天然不同——塞进同一套机制就会互相污染，也无法分别证明哪一层真有用。Atelier 的核心设计就是：把这三种差异，在实现的每一层都如实映射出来。这也是「记忆系统」比「一个向量库」值钱的地方。

贯穿全篇的设计美学 · 同源异构 SOP 与语义记忆同源（都从 PM 的同一段对话蒸馏）却异构（形态/强度/注入通道八个维度刻意相反）。这是本模块最值得讲的一个决策，「深挖」里有完整对照表。

物理落点：三层架构（触发 → 能力 → 存储）

刚才那张是「逻辑对照」，这张是「代码怎么摆」。整个模块严格分成三层：上面是触发层（谁在什么时候读/写），中间是能力层（共享库里的存取客户端 + 纯逻辑），下面是存储层（两张物理表）。写路径（绿）和读路径（蓝）从两侧进，都汇到中间的共享库——这就是为什么「抽象一次、复用三类」成本这么低。

George 让 PM 写「报销审批」PRD。PM 默认用 Given-When-Then。George 在澄清循环里说「验收标准别用 GWT，给我能勾选的 checklist」。PRD 定稿、approve。

# compose_prompt 包护栏后进 system prompt
→ PM 主动用 checklist 写验收标准，George 一个字没提

这就是 SOP 的价值：偏好被「学」会了，从第三个项目起沉默即默认照做。

反转 · 抗抖动

某天 George 临时说「这个项目特殊，用 GWT」。会不会立刻把学到的 checklist 冲掉？——不会。

# next_confidence_state(existing="checklist", count=2, new="given_when_then")
# 冲突 → count - 1 = 1 > 0 → 守住既有值，只侵蚀
agent_sop: ac_format=checklist  support_count=1  # 偏好未翻盘，但已跌破门槛暂不注入
# 且本次「用 GWT」是实时指令，优先级最高 → 本次照 GWT 走
# 要真正翻盘成 GWT，需再冲突一次让 count 归零 → 让位新值

一次例外不推翻长期偏好（抗抖动），持续改变会逐步侵蚀直至翻盘（能适应）。这就是「置信计数器」而非简单布尔的意义。

核心机制 · 钻到实现级

六个撑起整个模块的机制

展开看代码级细节。这几个点也是面试里最容易被追问「你具体怎么实现的」的地方。

① 置信计数器：为什么不是简单布尔？

SOP 的 support_count 不是「见过就 +1」的累加器，而是一致 +1 / 冲突 −1 / 归零才翻盘的三态转移。一个纯函数就把「抗抖动 + 能适应 + 过渡期回 baseline」三个诉求全兼顾：

def next_confidence_state(existing_value, existing_count, new_value):
    if existing_value is None:        return (new_value, 1)          # 起步
    if existing_value == new_value:   return (existing_value, existing_count + 1)  # 一致→增强
    eroded = existing_count - 1                                      # 冲突→侵蚀
    if eroded > 0:                    return (existing_value, eroded) # 守住
    return (new_value, 1)                                            # 归零→翻盘
场景转移效果
连续同意1 → 2 → 3 …越过门槛 2 后稳定生效
偶尔一次不同2 → 1只侵蚀不翻盘（抗抖动）
持续改口2→1→0→新值 1归零让位新值（能适应）
处于过渡（count<2）—不注入，回 baseline 默认

写入用 SELECT … FOR UPDATE 在事务内 read-modify-write，避免两个并发蒸馏同时改一行导致计数丢失。纯函数 next_confidence_state 与 I/O 完全分离，13 条纯逻辑单测覆盖全部转移分支。

② RRF 混合检索：向量与关键词，两路排名怎么融合？

语义 / 情景检索不是纯向量，而是 RRF（Reciprocal Rank Fusion，倒数排名融合）——向量召回和 BM25 全文各取 top-20，按排名倒数加权相加。妙处：不需要归一化两路异质分数（余弦距离 vs BM25 分完全不同量纲），只看排名，鲁棒。

WITH vector_results AS (   -- 向量：pgvector 余弦距离 <=>
  SELECT id, 1.0 / (60 + ROW_NUMBER() OVER (ORDER BY embedding <=> $1::vector)) AS rrf
  ... ORDER BY embedding <=> $1::vector LIMIT 20),
bm25_results AS (          -- 全文：ts_rank BM25
  SELECT id, 1.0 / (60 + ROW_NUMBER() OVER (ORDER BY ts_rank(tsv, query) DESC)) AS rrf
  ... LIMIT 20)
SELECT COALESCE(v.id,b.id),
       COALESCE(v.rrf,0) + COALESCE(b.rrf,0) AS combined_score
FROM vector_results v FULL OUTER JOIN bm25_results b USING(id)
ORDER BY combined_score DESC
  • 1.0 / (60 + rank) 就是经典 RRF，常数 60 是学术界调优的经验值：它压低头部差距、让两路都有话语权，避免单一路霸榜。
  • COALESCE(…, 0)：只在一路出现的文档，另一路当 0 分——不惩罚，也不白送。
  • 两步取数：第一步只在窗口函数里排序融合、拿 id + score；第二步才按主键 hydrate content/metadata 大字段，省带宽。
③ 同源异构：SOP 与语义记忆的八维对称

全模块最漂亮的一个决策。两类记忆同信号源（PM 的 conversation_history）、同作用域模型（反事实三分类 global/project/discard），却在八个维度上刻意对称相反——因为「偏好规则」和「事实背景」本质不同，实现就该逐层把差异映射出来：

为什么置信门槛相反？偏好说错一次会持续误导，必须多次确认（≥2）才生效；事实常只说一次（「我们是 2 人团队」），且弱注入无需防误伤，所以无门槛。差异不是随意的，是从「偏好 vs 事实」的本质推出来的——这句话是这个模块面试时最值钱的一句。

④ 检索隔离：一个差点让三类记忆串台的 P0 坑

episodic 和 semantic 共用 agent_memory 表，靠 memory_type + agent_role 过滤区分。坑在于：RRF 有两路子查询（向量 + BM25），只在其中一路加过滤，另一路会把别类记忆漏进 RRF 候选，污染结果。

# 过滤必须同时写进两路子查询，且用 NULL-guard 兼容「不过滤」
WHERE ($2::text IS NULL OR memory_type = $2::text)   # 向量路 + BM25 路各一份
  AND ($3::text IS NULL OR agent_role  = $3::text)
教训混合检索的过滤条件必须「所有召回路一致」。这也解释了为什么 Coder（首个消费者、库里只有 episodic）当时没加过滤能跑通——直到 PM 引入 semantic，共表才逼出这个必须修的隔离问题。$N::text IS NULL OR … 的 NULL-guard 让同一条 SQL 兼容「过滤」与「全召回」两种调用。
⑤ SOP 唯一约束：NULL 的 project_id 怎么参与去重？

SOP 的收敛靠 UNIQUE(tenant, role, scope, project_id, slot_key)——同一槽位只留一行、upsert 而非堆叠。但 global 偏好的 project_id 是 NULL，而 SQL 标准里 NULL ≠ NULL，普通 UNIQUE 会放行两条 global 重复行。

-- Postgres 15+ 的 NULLS NOT DISTINCT：让 NULL 之间也算「相等」
UNIQUE NULLS NOT DISTINCT (tenant_id, agent_role, scope, project_id, slot_key)
-- 查询侧对应用 IS NOT DISTINCT FROM 匹配 NULL
WHERE project_id IS NOT DISTINCT FROM $4::uuid

加上 render_sop_block 里「同槽位 project 覆盖 global」的合并逻辑——一个用户对同一旋钮，最终只会有一条生效偏好注入，物理上不膨胀。这是「有界枚举 + 数据库约束」把「记忆不无限增长」变成结构性保证，而非靠定期清理。

⑥ 语义去重：手写 cosine、无 numpy，为什么？

语义记忆写入前先 embed 新事实，与该 tenant 已有 semantic 记忆逐一算 cosine 相似度，超 0.92 判重复跳过。dedup_against 是纯函数、手写点积/模长、不引 numpy：

def cosine(a, b):
    dot = sum(x*y for x, y in zip(a, b))
    na = sum(x*x for x in a) ** 0.5
    nb = sum(y*y for y in b) ** 0.5
    return dot / (na * nb) if na and nb else 0.0
# dedup：任一已有向量相似度 > 0.92 → 判重复、跳过写入
  • 无 numpy：共享库保持零重依赖，纯函数好单测、好移植（21 条 semantic 纯逻辑单测）。
  • 复用 embedding：去重时算出的向量直接传给 write_episode(embedding=…)，绝不「去重 embed 一次、写库又 embed 一次」——reviewer 当初正是抓到双重 embed 才 fix 的。
  • 去重是「合并同类」不是「防冲突」：矛盾/过期事实 v1 刻意不消解、两条共存，靠注入时的弱措辞护栏兜底。
面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我做了三类记忆」，而是每一个差异背后都有一句「为什么必须这样」。

为什么不合并成一张表 / 一套机制？

可归因性优先

每加一层记忆必须能被单独 A/B 证明有用。多层同接就无法归因（吃过 Skills 冗余的亏）。三刀各有正交评测维度：质量 / 遵从度 / 贴合度。

SOP 为什么用独立结构化表而非向量？

有界即精确

偏好是有限枚举（6 槽位 × 固定取值），精确 SQL 比向量检索更准更快；UNIQUE 约束让它物理不膨胀，还顺带绕开共表的 memory_type 串台坑。

写记忆失败会不会拖垮交付？

fire-and-forget + 优雅降级

三个蒸馏都是异步 Activity，失败只丢一条记忆；所有读取都 try/except 降级回「底座 + Skill」。记忆是增强，不是主链路的单点。

怎么防止「学错偏好」持续误导？

置信计数器 + 优先级链

≥2 次才生效（抗抖动）、冲突侵蚀可翻盘（能适应）、实时指令永远压过 SOP（先验非笼子）。三重护栏。

纯逻辑为什么单独抽出来？

I/O 与决策分离

cosine 去重、置信转移、渲染全是无 I/O 纯函数，可脱离数据库单测（80+ 单测多数是纯逻辑）。DB 客户端只管存取。

向量 + 关键词为什么用 RRF 而非加权和？

只看排名，免归一化

余弦距离和 BM25 分量纲完全不同，加权和要先归一化很脆。RRF 只用排名倒数，天然免归一化、鲁棒、可解释。
