Plate 08 — Merge Agent

Merge Agent 归并

把 N 个并行切片分支，按拓扑波次归并成一条「每个 commit 都回归绿」的集成分支。

一句话：Merge 不是 ReAct agent，是一条零 LLM 的确定性管线——排序、git plumbing、回归门调度、升级梯状态机全是普通函数；LLM/容器只从恰好三个接缝（回归门 · 文本调和 · L1 fix-Coder）进来。合并的决策空间已被显式定案成规则，塞给 agent 自由发挥是拿确定性换方差、还丢 Temporal 友好性。

为什么归并要一条「确定性管线」，而不是又一个 agent？

上游 Planner（图版 07）把一个 feature 切成一张切片依赖图，Orchestrator ②（图版 09）按拓扑波次并行跑 N 个 Coder loop，各自在集成仓库里提交出一条 slice/<slice_id> 分支。现在轮到 Merge：把这些并行分支缝回一条主线。

直觉上你会想「让个 agent 看着冲突随机应变」。但归并的决策空间其实已经被想清楚、能穷举成规则：什么序合、绿了才落头、冲突怎么分类、修不好升到谁——这些都不需要临场发挥。于是 Merge 走了相反的路：

  • 骨架零 LLM、零随机——波内排序、git 三方合并、回归门调度、升级梯状态机全是普通函数，每一步都能切成一个 Temporal activity、可重放、可恢复。
  • LLM/容器只从三个接缝进——回归门（跑测试）、文本冲突调和（single-shot LLM）、L1 fix-Coder（AgenticCoder）。接缝之外的一切都是确定的。
  • 失败只结构化上报、从不做决定——Merge 自己最多做 L1 一次修复；修不好就把结构化失败交回 ② 编排（L2）或人工闸（L3）。
贯穿全篇的核心不变量 · 集成分支每个 commit 恒绿 集成分支的 ref 只在波回归门绿之后才前移。任意时刻的集成头都是「已合切片自测的并集还绿」的状态——也就是下一波、乃至 Temporal 崩溃恢复的安全基线。这个不变量是「绿前不落头」由代码结构（而非纪律）保证的，深挖 ① 讲它怎么落地。

回归门红了往哪升：三级升级梯

归并里唯一「可能出错」的判断是波回归门红了怎么办。Merge 的答案是一条责任边界明确的三级梯子——每级归属不同，Merge 自己只做 L1，往上一律只结构化上报、不做决定。关键纪律：升级是「换姿势」不是「加次数」（每级 ≤1 次；修不好还在原地重修，就是 #57 式空转）。

图版 II · 动态流程

一波怎么合，DAG 耗尽怎么收口

II-a 是一波内逐支归并的主循环（确定性序 → 调和 → 门 → 升级梯 → 前移）；II-b 是所有波跑完后那个更薄的终局操作 finalize_run。

图版 II-a · merge_wave · 一波内逐切片归并DYNAMIC · wave merge
0
起点 · 集成分支 = base_ref

集成分支 ref 起于 base_ref；工作树 detach 到头上做候选合并（候选不经分支 ref——ref 只在绿后 update-ref 前移，「绿前不落头」由结构保证）。

1
确定性序取下一支 · blocked_by 检查

按输入序（= Planner 拓扑序，不重排）取切片。若 blocked_by 链上（传递地）有本波已失败/blocked 的切片 → 标 blocked、从不尝试（无 merge、无门、无候选）。

拓扑序前提：前驱必先于后代被判定，unavailable 集才在判到后代时已含失败前驱。

2
git merge --no-ff branch

干净 3-way 合并候选，零 LLM。--no-ff 让每切片留一个 merge commit → git log --first-parent 直读归并波次史。

3
非零退出 → 先验真冲突态，再分流

非零 ≠ 一定冲突。先验 MERGE_HEAD 存在 + diff --diff-filter=U 非空：
· 无真冲突态（分支不存在等）→ abort → git_error，绝不触调和器/门；
· 真冲突 + reconciler=None → textual_conflict 弃候选；
· 真冲突 + 已注入 → 进 per-file 调和。

4
接缝② · per-file single-shot 调和

每个冲突文件恰一次调用，喂 path + base/ours/theirs（index stage 1/2/3）+ 双方 SliceContext。ours 侧归属 = touched_by 里本波最后动过该文件的已合切片（波前基线改动 → None）。每文件留一条 ReconciliationRecord，收束 merge commit 后与干净合并同路。

5
候选挂临时 ref · 接缝① 波回归门

候选挂 refs/merge-candidates/<slice_id>，注入的门跑候选：干净容器独立重跑集成分支上项目自测套件。逐切片跑（非逐波）——红时零成本归因。

6
门红 → 接缝③ 升级梯 L1（≤1）

门红且注入了 fix_coder → seed=MergeFixSeed（失败输出 + merge diff + 两侧 intent），fix-Coder 在候选工作树追加提交 → 候选 ref 前移 → 门重跑。≤1 由无循环结构保证；调和后的候选门红同样先走这一次 L1。

7
绿 → 集成头前移；红 → 头原地不动

绿：update-ref 集成分支到候选（每 commit 恒绿）；记 touched_by 供后续冲突归属；status=merged。
红（L1 用尽/未注入）：头纹丝不动、候选留临时 ref 尸检、regression_red——独立兄弟照常从最后绿头过门（失败不毒死全波）。

→
本波收尾 → 下一波

工作树落回集成分支。本波绿头即 wave-N+1 的 base_ref（run 级移动基线）。返回 WaveMergeResult（三态聚合）。

为什么逐切片过门而非逐波：逐切片换来红时零成本归因——是哪支砸了构建当场就知道，不用事后 bisect。「所有已合切片自测的并集还绿」就是语义冲突检测（git 干净合上但代码放一起跑不通，如一侧改了符号名、另一侧还用旧名）。解与验分离：调和器只产「合理候选」，正确性一律由回归门这把尺量——调和产物砸构建 → 门红头不动、垃圾只留在候选 ref。

端到端 · 走一遍

3 支切片、2 个波次、1 次冲突升级的全程

跟着一个真实归并跑一遍：clean 落头 → append 冲突调和 → 门红升级 L1 fix-Coder → 门重跑绿 → 收口开 PR。你会看清「绿前不落头」和三接缝到底怎么协作。

# 接缝③ 升级梯 L1（≤1，无循环）
seed = MergeFixSeed(
  slice_ctx=s3, merged_contexts=[s1, s2],

  merge_diff=<s3 带进来的全部改动>, candidate_sha=...)

# 强制提交契约：跑完脏树 → git add -A && commit
候选 ref 前移到修复版本 → 门重跑
重跑绿 · 收口

门重跑 全绿 → 集成头前移到修复候选，fix_attempted=True 留痕。DAG 耗尽，进 finalize_run。

① feat/reimbursement-approval ← 集成头  # update-ref, 零新提交

全程只有三处 LLM/容器触点（门 · 调和 · fix），骨架其余每一步都是可重放的普通函数——这正是「确定性管线」的价值：出了问题能精确定位到哪一支、哪个接缝。

反转 · 若 L1 也修不好

假设 fix-Coder 那一次没修好，门重跑仍红——Merge 不做第二次尝试（≤1 硬顶）。

# 结构化失败上报，Merge 只上报不决定
SliceMergeResult(slice_id="s3", status=failed,
  failure_kind="regression_red", fix_attempted=True,
  candidate_ref="refs/merge-candidates/s3")  # 尸检 ref 零删除
# 集成头停在 s2 绿头；s3 的独立兄弟照常过门（失败不毒死全波）
WaveMergeResult.failed_slice_ids = ["s3"]  # → ② 编排 L2 重派

升级是「换姿势」不是「加次数」：L1 修不好，就该 L2 在真实集成头上重派全新 loop（换个更强的解法），而不是在 merge 层原地无限重修——那是 #57 式空转。

核心机制 · 钻到实现级

六个撑起整条管线的机制

展开看代码级细节。这几个点也是面试里最容易被追问「你具体怎么保证正确性」的地方。

① 绿前不落头：为什么集成分支每个 commit 都恒绿？

核心不变量「集成分支每个 commit 恒绿」不是靠纪律，是靠代码结构保证的。诀窍：候选合并全在 detached HEAD 上做，集成分支的 refs/heads/<branch> 只在门绿之后才 update-ref 前移。

# 起点：工作树 detach 到 base_ref，分支 ref 独立于工作树
head = rev_parse(repo, base_ref)
run_git(repo, "checkout", "--detach", head)

# 每支：候选先挂临时 ref（绝不碰分支 ref）
candidate_sha = rev_parse(repo, "HEAD")
run_git(repo, "update-ref", candidate_ref, candidate_sha)  # refs/merge-candidates/*
outcome = await _run_gate(candidate_ref, candidate_sha, ctx)

if outcome.passed:                                          # 绿，且仅绿
    prev_head, head = head, candidate_sha
    run_git(repo, "update-ref", f"refs/heads/{integration_branch}", head)
else:                                                       # 红：头纹丝不动
    run_git(repo, "checkout", "--detach", head)             # 工作树回最后绿头
  • 结构 > 纪律：因为分支 ref 只有 if passed 这一条路径能前移，任何红候选物理上不可能落到集成头——不靠「记得回滚」。
  • --no-ff 每切片一个 merge commit → git log --first-parent 即波次归并史，天然可审计。
  • Temporal 互洽：任意集成头都是「已合切片自测并集还绿」的状态，所以它同时是下一波的安全基、也是崩溃恢复的安全点。
② per-file 调和：真冲突怎么验、ours 侧归属怎么找？

文本冲突调和有两个易踩的坑，代码都显式处理了。坑一：非零退出 ≠ 冲突——分支不存在也退出 1。必须先验真冲突态才许走调和器：

# 真冲突 = MERGE_HEAD 存在（merge 确实开始并中途停下）且 index 有 U 条目
in_merge = run_git(repo, "rev-parse", "-q", "--verify", "MERGE_HEAD",
                   check=False).returncode == 0
if not in_merge: return []                    # 空 → 非冲突失败 = git_error
out = run_git(repo, "diff", "--name-only", "--diff-filter=U").stdout

坑二：调和要喂「双方 intent」，ours 侧是谁？——用一个本波内的 touched_by 字典追踪「文件 → 最后动过它的已合切片」：

# 每冲突文件恰一次 single-shot，喂 base/ours/theirs 三版 + 双方 SliceContext
ours_ctx = touched_by.get(conflicted_path)    # 波前基线改的 → None
merged_content = reconciler(
    path=conflicted_path,
    base=_stage_content(repo, 1, path),       # index stage 1 = 共同祖先
    ours=_stage_content(repo, 2, path),       # stage 2 = 集成分支侧
    theirs=_stage_content(repo, 3, path),     # stage 3 = 当前待合切片
    ours_ctx=ours_ctx, theirs_ctx=ctx)
reconciliations.append(ReconciliationRecord(file=path, slice_ids=[...]))

调和产出整个文件（不是 patch），写回 → git add → git commit --no-edit 收束 merge commit，之后与干净合并走同一条路（候选 → 门 → 绿落头）。典型场景是共享聚合文件的双 append（路由注册表、依赖 manifest）——两边新增都该保留。add/add 冲突没有 stage 1，读到空串即「双方都是新增」。

③ 升级梯 L1：为什么「恰好一次」而不是循环重试？

门红时 L1 给 fix-Coder 恰好一次机会。这个「≤1」不是配一个 max_retries=1，而是无循环结构——代码里根本没有循环，物理上就不可能跑第二次：

# 无 while、无计数器：一个 if 就是全部
fix_attempted = False
if not outcome.passed and fix_coder is not None:
    seed = MergeFixSeed(slice_ctx=ctx, merged_contexts=list(merged_ctxs),
        regression_output=outcome.summary,
        merge_diff=run_git(repo, "diff", head, candidate_sha).stdout, ...)
    fixed = fix_coder(repo_path=repo, seed=seed)              # 那一次修复
    if inspect.isawaitable(fixed): await fixed
    fix_attempted = True
    candidate_sha = rev_parse(repo, "HEAD")                   # 取修复后 HEAD
    run_git(repo, "update-ref", candidate_ref, candidate_sha)
    outcome = await _run_gate(candidate_ref, candidate_sha, ctx)  # 门重跑一次
# 仍红 → 结构化失败上报，不再尝试
#57 教训「修不好就在原地加次数」是自纠偏空转的经典坑（无语法闸门时撞 max_iter）。Merge 的立场：merge 层修一次修不好，说明这不是「合并工伤」而是「切片本身该重做」——正确的响应是升级 L2 让 ② 在真实集成头上重派全新 loop（换个更强的解法），而不是原地换姿势硬凑。升级换的是层级，不是次数。
④ 波回归门：为什么复用 Reviewer 的「干净容器独立重跑」？

波回归门的真默认不是「随便跑个测试」，而是复用模块根的 IndependentTestRunner——这套原语本来长在 reviewer/ 里（对抗式 Reviewer 用它堵 self-gaming），#84 兑现 #51「共享原语上提」把它提到模块根，Reviewer 与 Merge 共用（reviewer/independent_test_runner.py 转 re-export shim）。

# regression_gate 真默认：起干净新容器、按 tech_stack provision、挂候选工作树重跑
async def _gate(*, repo_path, candidate_ref, candidate_sha, slice_ctx):
    runner = _build_runner(tech_stack=tech_stack, ...)  # IndependentTestRunner
    result = await runner.run(workspace_dir=repo_path)  # 候选正 detach 在 candidate_sha
    return RegressionGateOutcome(passed=result.passed, summary=result.summary)
  • 为什么「干净新容器」而非复用 Coder 的 live 容器：复用会被 Coder 装的全局包/改的环境喂绿，「独立重跑」名存实亡。干净容器从镜像起、重新 provision、bind-mount 同一 workspace，复现的是「换台机器 clone 下来能不能跑」——专逮「依赖没写进 manifest / 测试依赖脏环境 / 根本没跑」这类 self-gaming。
  • 同一把尺量两处：Reviewer 的对抗 review 和 Merge 的波回归门，本质是同一个原语（同镜像/provision/隔离容器）——所以该共用，而不是各写一份。
  • passed 容忍无关模块的 collection error（缺可选依赖），但改动真弄坏能收集的测试、或「啥都没跑起来」一律判红——反自欺硬信号不破。
⑤ host-git 安全出口：为什么一切 git 只准走 run_git？

Merge 的 git 跑在 host 上（回归与 fix-Coder 才进容器）。而集成仓库的内容来自 agent 产出，必须视作不可信——它可能带 booby-trapped hooks、fsmonitor 命令、ext:: 伪协议。git_ops.run_git 是 merge/ 里唯一的 git 出口，把所有中和配置集中在一处、不可能漏：

_SAFETY_CONFIG = (
    "-c", f"core.hooksPath={os.devnull}",  # /dev/null 非目录 = hooks 恒空（新 hook 类型也免疫）
    "-c", "core.fsmonitor=false",          # fsmonitor 可配成任意命令，禁
    "-c", "protocol.ext.allow=never",      # 禁 ext:: 伪协议（任意命令执行通道）
    "-c", "user.name=atelier-merge-agent", # 固定身份 = 提交确定性（不吸 host 配置）
    "-c", "user.email=merge-agent@atelier.local")
# 环境层再加一道：屏蔽全局/系统 gitconfig
env = {..., "GIT_CONFIG_GLOBAL": os.devnull, "GIT_CONFIG_SYSTEM": os.devnull,
       "GIT_TERMINAL_PROMPT": "0"}  # 绝不交互挂起
  • 命令行 -c 优先级高于仓库 local config：即便仓库 .git/config 把 core.hooksPath 指向陷阱目录，这里的覆盖也生效——比逐个禁用 hook 名单更彻底。
  • 单一出口 = 不可能漏：安全不是散在各调用点的 if，而是「所有 git 子进程都必须经这个函数」的结构约束。
  • 固定身份顺带保证确定性：merge commit 不吸 host 用户配置 → 同输入同 sha，Temporal 重放友好。
⑥ 三接缝真默认：为什么 None 不能兼任「用默认」和「关掉」？

三个接缝都有真实现默认：省略参数 → 真默认激活（生产真跑）；注入 → 让路（离线测试喂脚本化假件）。难点在于 reconciler=None / fix_coder=None 是刻意的「关掉这个接缝」信号（无调和器 → textual_conflict；无 L1 → 门红直接 regression_red），与「没传、给我真实现」语义相反。用 None 兼任二职会打架，所以引入哨兵：

_USE_REAL_DEFAULT = object()   # 独一无二的哨兵，区分「省略」vs「显式 None」

async def merge_wave(..., *, regression_gate=_USE_REAL_DEFAULT,
                     reconciler=_USE_REAL_DEFAULT, fix_coder=_USE_REAL_DEFAULT):
    if regression_gate is _USE_REAL_DEFAULT or ...:
        from merge.real_defaults import (...)   # 惰性 import：注入假件的测试连碰都不碰
        if regression_gate is _USE_REAL_DEFAULT:
            regression_gate = make_default_regression_gate(tech_stack=tech_stack)
        # reconciler / fix_coder 显式传 None → 跳过，保留 null 行为
传参方式行为用途
省略参数激活 real_defaults 真实现生产真跑（Docker/LLM）
注入假件用脚本化 runner/reconciler/coder离线全谱系测试
显式 None关掉该接缝、保留 null 行为测某接缝缺席的降级路径

真默认的重依赖（docker / langchain gateway / AgenticCoder）一律惰性 import——只在真被调用时才拉起，注入假件的离线单测连 real_defaults 都不 import。还有个隐式契约：真默认 fix-Coder 强制「跑完脏树补一提交」——不提交的 fixer 会留脏树、rev_parse(HEAD) 取不到修复版本、修复白做（Reviewer 追出的坑）。

面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我做了个归并器」，而是每一个选择背后都有一句「为什么必须这样」。

归并为什么不用 agent 而用确定性管线？

决策空间可穷举时，别换方差

合并的决策（排序/落头/冲突分类/升级）已能显式定案成规则。塞给 agent 自由发挥是拿确定性换方差、还丢 Temporal 友好性。只在真需要智能的三处开接缝。

怎么保证集成分支不会合进坏代码？

绿前不落头，由结构保证

候选全在 detached HEAD 做，分支 ref 只有 if gate.passed 一条路径能前移。红候选物理上不可能落到集成头——不靠「记得回滚」的纪律。

逐切片过门 vs 逐波过门，为什么选前者？

红时零成本归因

逐切片能当场知道是哪支砸了构建，不用事后 bisect。省下的 bisect 归因成本远超多跑几次容器的开销。

文本冲突和语义冲突谁来逮？

解与验分离

文本冲突（同区双改）归调和器；语义冲突（git 干净合上但代码错）任何冲突解器都逮不住，归波回归门。调和只产「合理候选」，正确性一律由门量。

修不好怎么办，为什么不多试几次？

升级换层级，不换次数

L1 一次修不好 = 这不是合并工伤而是切片该重做。升 L2 让 ② 在真实集成头重派全新 loop。原地无限重修是 #57 式空转。

host 上跑不可信仓库的 git 安全吗？

单一出口 + 全量中和

一切 git 只走 run_git，集中中和 hooks/fsmonitor/ext 伪协议 + 屏蔽全局 config。命令行 -c 压过仓库 local config，新 hook 类型也免疫。
