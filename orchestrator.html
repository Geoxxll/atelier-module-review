Plate 09 — Parallel Orchestrator

并行编排 Orchestrator

把一张校验过的切片 DAG，按拓扑波次 fan-out 成并行闭环 loop、收集、归并、收口——从 PlanOutput 到 FinalizeResult 的贯通编排。

一句话：整个编排的所有决策——哪些切片就绪、失败怎么重派、哪些被剪、何时收口——全压进一个纯确定性 Reducer（step(state,event)→(state',commands)，零 I/O / 零并发 / 零随机）；薄 Shell 只执行命令、把结果作事件喂回，零决策。同一个 Reducer，standalone 挂 asyncio 壳、⑤ 挂 Temporal workflow 壳——逐字复用。

为什么编排器要「掏空成一个纯函数」？

这是链路优化的第 ② 环，夹在 ①Planner（切出切片 DAG）与 ③Merge（波次归并）之间。它要干的活听起来很「工程」：把互相依赖的切片按拓扑波次并行跑闭环 loop、失败了重派、收集产出交给 Merge、最后收口成一个 PR。这类活最容易写成一坨 async 意大利面——并发、git 副作用、人工等待、失败分支全缠在一起，没法测、更没法搬上 Temporal。

Atelier 的做法是把它劈成三层，让「决策」和「副作用」彻底分家：

  • Reducer（纯决策核）——step(state, event) → (state', commands)。就绪判定、失败重派、blocked 剪枝、何时触人工闸、何时 finalize，全部编排智慧都在这。它零 I/O、零并发、零随机，可以脱离 Docker / Temporal / LLM，用脚本化假件把每一条状态转移离线跑一遍。
  • Shell（薄壳）——一个 while pending 主循环：执行 Reducer 吐的命令、把接缝结果包成事件喂回。零决策，所有判断都在 Reducer。
  • 三注入接缝——把有副作用/非确定性的三件事（跑 loop / 归并 / 等人工）隔成 LoopDriver / Merge / HumanGate 三个 Protocol，从 Reducer 里拎出去。
贯穿全篇的承重决策 · 纯 Reducer 的两个理由 劈出纯核不是为了并行两个 agent 实现（那个动机被明确否掉了）。真正的理由有两条、且各自独立成立：① 离线可全谱系测（项目 standalone 优先纪律）；② Temporal「workflow body 越薄越好重放」的最佳实践。正因如此，同一个 Reducer 才能被 standalone asyncio 壳和 ⑤ Temporal 壳逐字复用——这是本模块最值钱的一句话。

怎么读：命令（绿）从 Reducer 往下流经 Shell 打到接缝、副作用发生；结果作事件（蓝）逆流喂回 Reducer 推进下一步。Reducer 从不碰 I/O——它只输出「要做什么」的描述（命令），真做由 Shell + 接缝完成。这条分界线正是 ⑤ 能无痛托管上 Temporal 的原因。

命令 / 事件 / 状态 —— 冻结的 ②⑤ 接缝

Reducer 的输入是事件、输出是命令、内部维护一份状态。这三样都做成纯数据 pydantic——ADR §3.8 把它们整体冻结为 ②/⑤ 接缝：⑤ 的 Temporal 壳照搬当 workflow state（状态）/ activity 触发（命令）/ activity 返回值·Signal（事件）。改它们即改 ⑤ 契约，慎重。

图版 II · 动态流程

一条 run 怎么从 DAG 走到 PR

Happy path 是「fan-out → 归并 → 前移集成头 → 下一波 / 收口」的波次串行循环；失败 path 是「≤1 重派 → 耗尽上 L3 → blocked 剪枝 + 刹车」。两图分别看。

图版 II-a · Happy path · 一波接一波直到收口DYNAMIC · main loop
1
StartEvent → reducer._on_start

_ready_wave 算出「未处理 且 前驱全 merged」的切片（Planner 序），吐 [FanOut(wave-0)]，各切片 attempts +1。

wave-0 = 无 blocked_by 前驱的切片（通常是脚手架 scaffold）。

2
Shell → LoopDriver.fan_out

每就绪切片 clone 一个 workspace 并行跑闭环 loop（受 Semaphore(K=4) 节流）。完成序乱 → Shell canonical_sort 按 Planner 序重排 → 包成 LoopWaveCompleted 喂回。

承重铁律：乱序喂 reducer 即破重放。

3
reducer._on_loop_wave_completed

分流：approved=True 的子集 → [MergeWave(只带 approved 分支)]；未收敛的留 pending 待 settle。

4
Shell → Merge.merge_wave → WaveMerged

reducer 标 merged_slice_ids → merged，base_ref 前移到 integration_head（下一波/重派的新起点），累积进 waves。

集成头前移是波间串行的物理体现——后代只能在这之后就绪。

5
reducer._advance

再算 _ready_wave：非空 → [FanOut(下一波)] 回到步骤 2；空且无 pending → 进 finalize 决策。

6
Finalize → Merge.finalize_run → Finalized

集成头有内容 → [Finalize(waves, failed, skipped)] 收口成 feature 分支 + PR；Finalized 落 finalize_result、phase=done，主循环终止。

波间为什么串行：后代切片的 base_ref 只有在前驱已 merge 进集成头后才存在（Blocked-by = 「B 用 A 建的符号」）。所以「就绪 = merge-keyed」而非「loop 跑完」——牺牲一段并行度换取后代能在含前驱产出的真基上跑。

图版 II-b · 失败 path · 重派 / 上闸 / 剪枝 / 刹车DYNAMIC · failure core

失败 settle · 两来源共用 ≤1 重派

A
_outstanding = 失败待决集

= status==pending 且 attempts≥1——一句话统一两失败来源：loop 未收敛（approved=False）与合并门红（failed_slice_ids）都表现为「派发过但没进集成头」。

无需额外状态字段辨识两来源，省状态、易重放。

B
_settle：还有预算？

attempts < 2 → [FanOut(重派整批)]，在含已合本波兄弟的当前集成头上开全新 loop（波内 settle，语义冲突被真基逼出）。

C
_settle：预算耗尽

全部 attempts≥2 → [ResolveMerge] 逐片上 L3 人工闸。MergeResolved 分流：skip→落 failed 继续；manual/abort→ _abort 整 run 作废。

优先级 abort > manual > skip，确定性不依赖 dict 序。

blocked 剪枝 + 刹车 + finalize 决策

A
后代自动剪

永久失败前驱的后代靠 _ready_wave（要求前驱全 merged）永不就绪——无需显式传播 blocked，剪枝是就绪规则的自然结果。

B
_advance 刹车

就绪波空但仍有 pending（被永久失败前驱卡住）→ 剩下全标 skipped、break，不空转。

C
finalize 决策（ADR §3.4）

_has_integration_content（任一切片 merged）→ 局部 finalize（传 failed/skipped 素材，写清哪些没进）；零 merged（脚手架失败剪光后代）→ _abort(scaffold_failed) 停本地、不开 PR。

成本硬封顶：每切片最多 2 次 loop（首派 + ≤1 重派），两失败来源共用这一次预算——绝不为「未收敛」另设升级路径。这直接对齐 #57「修不好换姿势、不加次数」的收敛纪律，每切片 ≤2 个 Docker 容器封顶。

端到端 · 走一遍

一张两切片的 greenfield DAG，穿过整条编排

用最小但完整的例子：一个脚手架切片 + 一个依赖它的切片（含 blocked_by 边）。跟着 step() 的输入输出走，你就懂了 merge-keyed 波次串行的全貌。

场景 · greenfield · calc-cli

DAG：s-scaffold（脚手架）← s-add（blocked_by s-scaffold）

provision

greenfield → git init -b main + 空根提交 作 main，base_ref='main'。全切片 pending、attempts=0。Shell 把 StartEvent 塞进 pending 队列。

# OrchestratorState.initial
slice_status = {"s-scaffold": "pending", "s-add": "pending"}
attempts     = {"s-scaffold": 0, "s-add": 0}
base_ref     = "main"
wave-1 fan-out

step(state, StartEvent) → _ready_wave 只放 s-scaffold（s-add 的前驱还没 merged，未就绪）。

# reducer 输出
cmds = [FanOut(assignments=[(base_ref="main", slice=s-scaffold)])]
attempts["s-scaffold"] = 1
# Shell: LoopDriver.fan_out → 真 clone workspace → 闭环 loop → approved
→ canonical_sort → LoopWaveCompleted(results=[LoopResult(s-scaffold, approved=True, branch="slice/s-scaffold")])
wave-1 merge

step(state, LoopWaveCompleted) → approved 子集非空 → MergeWave。Merge 把 slice/s-scaffold 归并进集成头。

# reducer 输出 → Shell: Merge.merge_wave → WaveMerged
cmds = [MergeWave(base_ref="main", slice_branches=["slice/s-scaffold"])]
# step(state, WaveMerged)：标 merged + 前移集成头
slice_status["s-scaffold"] = "merged"
base_ref = "<integration_head 前移>"   # 关键：下一波的新起点
wave-2 ready

WaveMerged 尾部走 _advance → 再算 _ready_wave。这次 s-add 的前驱 s-scaffold 已 merged → s-add 就绪！在前移后的集成头上 fan-out。

# _ready_wave：s-add 的 blocked_by=[s-scaffold] 全 merged ✓
cmds = [FanOut(assignments=[(base_ref="<含 scaffold 的新头>", slice=s-add)])]
# s-add 的 workspace 从含 scaffold 代码的基上 clone —— re-ground 到真前驱符号
→ LoopWaveCompleted([LoopResult(s-add, approved=True)]) → MergeWave → WaveMerged
slice_status = {"s-scaffold": "merged", "s-add": "merged"}
finalize

_advance：就绪波空、无 pending → finalize 决策。_has_integration_content=True（两片都 merged）、failed/skipped 皆空 → 完整 finalize。

# reducer 输出
cmds = [Finalize(waves=[…], failed=[], skipped=[])]
# Shell: Merge.finalize_run → 收成 feat/calc-cli + PR → Finalized
Finalized(result=FinalizeResult(feature_branch="feat/calc-cli", status="local_only"))
state.phase = "done"   # 主循环 break → 返回 finalize_result

这就是 merge-keyed 波次串行的全貌：s-add 一定等到 s-scaffold 落进集成头，才在含它的真基上开跑——依赖正确性优先于并行度。

反事实 · 若 s-scaffold 失败

假设 wave-1 的 s-scaffold 两次 loop 都没收敛、L3 也 skip → 落 failed。那么 s-add 永远不就绪（前驱非 merged）→ _advance 刹车标它 skipped。

# _has_integration_content：零切片 merged → 无可用集成头
slice_status = {"s-scaffold": "failed", "s-add": "skipped"}
→ _abort(reason="scaffold_failed")   # 整 run 作废、停本地、不开 PR、无命令
→ RunAborted(scaffold_slice_ids=["s-scaffold"], failed=[…], skipped=[…])

脚手架是 wave-0 根、全 feature blocked_by 它——它失败必剪光后代。所以「零 merged ⟺ 无可用集成头」对任意 DAG 成立，比特判 kind=="scaffold" 更稳健。

核心机制 · 钻到实现级

六个撑起整个模块的承重决策

展开看代码级细节。这几个点也是面试里最容易被追问「你具体怎么保证的」的地方。

① Reducer 纯函数：Temporal 确定性重放的铁律

Reducer 是整个编排的大脑，但它禁碰任何非确定性——没有 uuid4() / now() / random，不就地改 state，每次 model_copy(update=…) 返回新态。为什么这么苛刻？因为 ⑤ 要把它挂进 Temporal workflow body，而 Temporal 靠重放事件历史恢复状态——workflow 代码每次重放必须走出完全相同的路径，任何随机/时钟/外部可变状态都会让重放分叉、run 直接崩。

def step(state, event) -> tuple[OrchestratorState, list[Command]]:
    # 纯派发：输入 (state, event) → 输出 (state', commands)，同输入必同输出
    if isinstance(event, StartEvent):        return _on_start(state)
    if isinstance(event, LoopWaveCompleted): return _on_loop_wave_completed(state, event)
    if isinstance(event, WaveMerged):        return _on_wave_merged(state, event)
    ...
# 不就地改 state —— 永远返回新态（便于 ⑤ 把 state 当 workflow state 重放）
new_state = state.model_copy(update={"slice_status": new_status, "base_ref": …})

红利：step 可以直接断言（喂一个 state + event，断言输出的命令 + 新 state），把 happy / 失败 / 重派 / blocked / 局部-finalize 全谱系离线脚本化跑一遍，不碰 Docker / Temporal / LLM。所有副作用（真 git、asyncio 并发）只落在 driver / adapters / execution——绝不渗进 reducer。这条纪律 S1–S3 立住，S4 加真接缝时也没破。

② 就绪 = merge-keyed：为什么不是「loop 跑完就放行」？

一个切片就绪（可以 spawn）的判据是：它的每个 blocked_by 前驱都已 merge 进集成头——不是「前驱的 loop 跑完了」，而是「前驱的产物真落进 base_ref 了」。

def _ready_wave(state):
    ready = []
    for s in state.plan.slices:              # 遍历序 = Planner 输出序 → 天然 canonical
        if state.slice_status.get(s.id) != "pending":  continue
        if all(state.slice_status.get(dep) == "merged" for dep in s.blocked_by):
            ready.append(s)                  # 前驱全 merged 才就绪
    return ready

依据是 Blocked-by 的语义：「B 用 A 建的符号」→ B 的 workspace 必须从含 A 代码的基上 clone，否则 loop re-ground 不到前驱符号、直接写错。而 A 的产物只有落进集成头后 base_ref 才存在（前移由 merge 做）。

被否的方案「loop-done keyed」（A 的 loop 一跑完就放行 B）并行度更高，但 B 会从缺前驱的基上检出——要么写错，要么得引入「乐观检出 + 事后 rebase」的复杂度（ADR-0003 §4.2 已显式推迟）。代价 = 波间串行、Merge 在关键路径，这是刻意接受的取舍：正确性优先于并行度。
③ canonical 收集：并发完成序 → Planner 输出序

这是承重铁律里最微妙的一条。LoopDriver.fan_out 并发跑一波切片，完成序是乱的（谁先跑完看运气）。但 reducer 是确定性的、已假定 LoopWaveCompleted.results 按 Planner 输出序排好。所以 Shell 收集完必须重排再喂：

# shell._execute（FanOut 分支）—— 唯一 I/O 触点
results = await loop_driver.fan_out(assignments)
# 确定性铁律：并发完成序 → Planner 输出序（canonical）再喂 reducer
results = canonical_sort(results, order)   # order = 切片在 plan.slices 里的位置
return LoopWaveCompleted(results=results)

为什么关键？若乱序喂 reducer，同一批 LoopResult 在两次重放里以不同顺序进入，reducer 的消费序就变了——即便每步纯函数，整体轨迹也分叉了，Temporal 重放崩。canonical_sort 把并发的不确定性归一在进 reducer 之前，reducer 内部才敢假定有序。

一个精细处：S4 的真 driver 用 asyncio.gather（按输入序返回、非完成序），交回的 results 天然是 Planner 序。但 Shell 的 sorted 仍保留作兜底——接缝契约允许「完成序乱」的其它 driver 实现（如 ⑤ 的 child activity 心跳回填），不能因当前 driver 恰好保序就删掉那道防线。⑤ 的 Temporal 壳也显式再 canonical_sort 一遍。

④ 失败核：≤1 重派 + blocked 自动剪 + 刹车

失败处理是 reducer 最密的一块，三个机制咬合：

1) 两来源共用 ≤1 重派。切片有两种失败——loop 未收敛（approved=False）与合并门红（failed_slice_ids）。reducer 用一句话把它们统一：

def _outstanding(state):   # 失败待决 = 派发过但未合入
    return [s for s in state.plan.slices
            if state.slice_status.get(s.id) == "pending"
            and state.attempts.get(s.id, 0) >= 1]     # 两来源都表现为「pending 且 attempts≥1」

def _settle(state):
    outstanding = _outstanding(state)
    redispatchable = [s for s in outstanding if attempts[s.id] < MAX_LOOPS_PER_SLICE]  # =2
    if redispatchable:  return _fan_out(state, redispatchable)  # 还有预算 → 重派整批
    return state, [ResolveMerge(slices=outstanding, …)]         # 耗尽 → 上 L3 人工闸

每切片最多 2 次 loop，成本硬封顶（≤2 容器），不为未收敛另设升级路径——直接对齐 #57 纪律。重派优先于上闸：一个 settle 点同时有可重派和已耗尽的，先重派、耗尽的下轮再上 L3（不混批、确定性）。

2) 后代自动剪。永久失败前驱的后代，靠 _ready_wave（要求前驱全 merged）永不就绪——无需显式传播 blocked，剪枝是就绪规则的副产品。3) 刹车兜底。_advance 里「就绪波空但仍有 pending」→ 把剩下的 pending 标 skipped 并 break，避免空转。

L3 _on_merge_resolved 三分流：skip→落 failed 继续收尾；manual→standalone 无人工可修、不自造语义、保守作废；abort→作废。优先级 abort > manual > skip，混批里只要有一个 abort/manual 就中止整 run（确定性、不依赖 dict 迭代序）。

⑤ ⑤ Temporal 壳：同一 Reducer 的另一副躯壳

⑤ 把 ② 托管上 Temporal，核心手法是「reducer / models 逐字复用、只有 await 循环另写」。asyncio 的 deque/gather/Semaphore 换成 Temporal 的 execute_activity——两个 runtime 的 await 语义与沙箱约束根本不同，硬共享一份反而别扭；但决策核完全不动。

元素standalone（asyncio）⑤（Temporal）

五个 activity：provision_activity（立集成仓库 + 返回钉定队列名）/ run_loop_slice（一切片一 activity + 心跳）/ merge_wave_activity / finalize_activity / cleanup_activity，全复用 S4 的 driver/adapters/provisioning 真实现。烧钱/改集成仓库的 activity 钉 maximum_attempts=1（关自动重试——失败/重试决策独占归 reducer 升级梯，守成本封顶）；provision/cleanup 便宜幂等则留自动重试。

双平面 worker（worker.py）：控制平面 orchestrator-wf（只跑 workflow、轻、无 Docker）+ 数据平面 loop-heavy（Docker+大内存，同进程起两个 Worker：公共队列 + host 专属队列 loop-heavy@<host-id>）+ 轻 I/O light-io（publish_event）。provision_activity 用公共队列随机落一 Pod、返回其专属队列名，之后所有碰集成仓库本地盘的 fs-activity 全钉该专属队列 → 同一 run 落同一 Pod（共享本地目录），不同 run 天然分散。_pinned_task_queue() 与 activity 侧 _worker_host_id() 同源，保证 provision 返回的队列恰有本 worker 认领。

真 Signal 人工闸：resolve_merge（批·增量并入）+ resolve_publish（单决定）走 workflow.wait_condition(…, timeout=gate_timeout)——等 Signal 放行，超时落默认策略（headless 不挂起）。一个 gate_timeout 旋钮统一「等人工」与「无人值守默认」。signal handler 只更新实例字段、绝不碰 reducer，控制流仍确定性。

⑥ 命令/事件/状态 = 冻结的 ②⑤ 接缝

为什么 ⑤ 能几乎无痛托管 ②？因为设计之初就把 Reducer 的 状态/事件/命令 schema 整体冻结为接缝（= ①③「切片分支契约」的等价物）。它们都是纯数据 pydantic——⑤ 只依赖这一组，不碰 reducer/shell 实现。这三样和三个 Protocol 一起，就是 ② 与 ⑤ 的全部契约面。

但有一类东西不能穿 Temporal payload：loop_fn（真 Docker 闭环 loop）/ merge 门（回归门·调和器·fix-Coder）/ publisher——它们是非可序列化的有副作用回调。解法是 SeamProvider，随 worker 启动一次性注入，activity 运行时按 RunContext（只装可序列化的仓库定位 + 发布配置）现取：

# activities.py —— 烧钱接缝不走 payload，worker 层注入
class SeamProvider(Protocol):
    def loop_fn(self, ctx): ...        # 真闭环 loop 入口
    def merge_kwargs(self, ctx): ...   # 回归门 / 调和器 / fix-Coder
    def publisher(self, ctx): ...      # PR publisher（无 GitHub 时 None）

# S6 的 live_seams.LiveSeamProvider 注真件：
#   loop_fn → run_review_loop_task（真闭环 Coder + 对抗式 Reviewer + Docker 沙箱 + LLM 经 Gateway）
#   控制平面沙箱绝不 import（牵连 langchain 重依赖）——只数据平面 activity worker + gated live e2e 用
S6 must-fix · 尸检现场workflow.run 的 finally 原本无条件 cleanup。但遇真 activity 异常（非 reducer 结构化 RunAborted）时 abort_result=None，会被误判成「成功清扫」、静默删掉尸检现场。修复：加 completed 门，仅主循环正常收口才清扫，异常路径保留全部尸检——与 standalone execute_run「异常不 _cleanup」严格对齐。尸检现场即证据。
面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我做了个并行编排器」，而是每个取舍背后都有一句「为什么必须这样」。

为什么把编排掏空成纯函数？

可测 + 可重放，双重独立成立

劈纯核不是为并行两 agent（那个动机被否）。理由是离线全谱系可测（standalone 纪律）+ Temporal「workflow body 越薄越好重放」。同一 reducer 才能被 asyncio 壳和 Temporal 壳逐字复用。

就绪判据为什么是 merge-keyed？

依赖正确性优先于并行度

后代必须从含前驱产出的基上跑，否则 loop re-ground 不到符号。前驱产物只有落进集成头后 base_ref 才存在。宁可波间串行、Merge 在关键路径，也不引乐观检出 + rebase 的复杂度。

并发完成序乱，怎么保证确定性？

canonical 收集，在进 reducer 前归一

Shell 收集 LoopResult 后按 Planner 输出序 sorted 再喂 reducer。乱序喂即改消费序、破 Temporal 重放。即便当前 driver 用 gather 保序，那道 sorted 仍作兜底不删。

失败怎么不烧爆成本？

两来源共用 ≤1 重派

loop 未收敛和合并门红统一为 pending 且 attempts≥1，共用同一次重派预算。每切片 ≤2 loop 硬封顶、≤2 容器，对齐 #57「修不好换姿势不加次数」。

失败前驱的后代怎么处理？

自动剪 + 刹车，不显式传播

永久失败前驱的后代靠就绪规则永不就绪——剪枝是规则的副产品。兜底刹车把卡住的 pending 标 skipped。零 merged ⟺ 无集成头 → 整 run 作废，比特判 scaffold 更稳健。

⑤ 托管到 Temporal 改了多少？

reducer 零改，只重写 await 循环

状态/事件/命令冻结为接缝，⑤ 照搬当 workflow state / activity 触发 / Signal。烧钱接缝不能序列化 → worker 层 SeamProvider 注入。双平面 worker + host 钉定队列让同 run 落同 Pod。
