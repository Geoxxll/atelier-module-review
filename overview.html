系统总览

一支住在 Temporal 工作流里的 AI 虚拟团队——用自然语言提需求，PM → Architect → Coder → Reviewer 接力协作，产出 PRD、技术方案、代码与 GitHub PR。

这套复盘集把整个系统拆成 15 个模块、分 4 层逐一讲透——每张子页都有静态架构图、动态流程图、端到端例子和代码级深挖。项目的真正目标是「通过造一个生产级多 Agent 系统来深入学习」，所以处处标注了值得讲的设计取舍。

先建直觉

把「一个人写代码」拆成一支会协作的 AI 团队

传统做法是让一个大模型一口气把需求变成代码。Atelier 反其道而行：把软件交付拆成四个专职角色，像真实团队一样接力——

  • PM 跟你打磨需求、澄清、写 PRD；
  • Architect 把 PRD 变成技术方案 TechSpec；
  • Coder 在 Docker 沙箱里写代码 + 跑测试；
  • Reviewer 审查、打回、要求返修，直到达标。

这条流水线跑在 Temporal 工作流里——它负责确定性编排、失败重试、和「等你点头」的人工闸（Signal）。中间所有 LLM 调用都过一个 LLM Gateway，大产物走 Blackboard、消息走 NATS，团队还带三类长期记忆越用越懂你。

而项目最有意思的一条主线是：Coder/Reviewer 有两套范式并存——早期的 open-loop 全量生成（legacy 生产链路）和对标 Claude Code 的 closed-loop 读改测增量（新管线）。两者物理隔离、各自能被 A/B 单独证明，谁更好用数据说话。下面的「模块地图」按层展开全部 15 块。

4
Coder 阶段

在 Docker 沙箱里写代码、跑 install/lint/test。产物写 Blackboard。（闭环范式是读→改→测→纠偏增量循环。）

5
Reviewer ↔ Coder 循环（≤3 轮）

审查不通过 → Coder 返修。闭环范式的 Reviewer 会在干净容器独立重跑测试堵 self-gaming。

6
收尾 · 记忆蒸馏（fire-and-forget）

workflow 完成后蒸馏情景 / SOP / 语义三类记忆，越用越懂你——不阻断主流程。

并行进阶：当一个需求大到要拆成多片并行做时，走的是「闭环管线」的另一条编排——Planner 切成 Slice DAG → Orchestrator 按波次 fan-out 并行 → Merge 归并。见对应子页。

⑂ 两套范式

open-loop 生产链路 vs closed-loop 闭环管线

项目里 Coder/Reviewer 有两套并存、物理隔离的实现——这是本项目「用可 A/B 的方式学习 Agent 能力」纪律的最大体现。

维度legacy 生产链路（open-loop）闭环管线（closed-loop）
Coder 范式全量生成：一次性写出文件增量：读→改→跑测→看反馈→纠偏
对标传统 codegen现代 coding agent（Claude Code 等）
ReviewerReflexion（Critique→Reflect）对抗式：干净容器独立重跑堵 self-gaming
并行编排单需求线性Planner 切片 → Orchestrator 波次 → Merge
地位当前生产链路 + A/B baseline挑战者，赢下盲评 A/B 才替换（ADR-0013）
为什么两套并存每加一层能力必须能被单独 A/B 证明有用，否则无法归因。闭环 vs 开环、±对抗 Reviewer 都是独立变量、分开验证——这条纪律贯穿记忆系统、Skills、评测子系统。本复盘集里 🅐 生产骨架（1–4）多为两套共用的底座，🅑 闭环管线（5–10）是新范式，🅒 横切（11–13）跨范式，🅓 评测（14）就是那把 A/B 尺子。
