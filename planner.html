Plate 07 — Planner

Planner 切片规划

把「需求」翻译成「一张能并行开工的切片依赖图」——并且自己校验自己切得对不对，切错了自己重切。

一句话：吃已批准的 PRD + TechSpec，切成一组可独立实现、可独立测试的 Slice，内联 blocked_by 边织成 DAG；产出前先过一道自纠偏四查门——引用完整 / 无环 / 覆盖度 / scaffold 结构，不过就带诊断有界重切，撞满仍不过才交人。切片喂给下游并行编排 ②按波次 fan-out、Merge ③归并分支。

为什么「需求」不能直接开工？

上游 PM 产 PRD（需求单位 = user story），Architect 产 TechSpec（组件 / 数据 / 接口）。但你不能把一条 user story 直接甩给一个编码 Agent——因为「需求边界」和「可并行实现边界」是两条完全不同的轴：

  • 一条 user story 可能横跨好几个能独立实现的增量（登录 story 里既有数据表、又有接口、又有前端）。
  • 好几条小 story 可能打包进一个增量（三个小配置项其实一次就写完了）。
  • 有些该做的增量连 story 都没点名——它只出现在 TechSpec 的某个组件里。

Planner 就是这个翻译 + 重切层：把需求切成 Slice（实现单位，不是需求单位），用 blocked_by 逻辑依赖边织成 DAG，让下游编排器能按拓扑波次最大化并行地 spawn 一堆闭环 Coder。切法归 Planner 自由裁量——它可以一对多、多对一、无中生有。

贯穿全篇的设计美学 · 自校验的规划者 Planner 最值得讲的一点：它不信任自己的 LLM 输出。切完先过一道全确定性的四查门（无 LLM、无 Docker），不过就带着「哪一条为什么挂了」的诊断自己重切，撞满预算才落 needs_human——绝不把一张坏 DAG 甩给下游。「深挖」里逐条拆这道门。

图版 II · 动态流程

produce → 校验 → 有界重切的自纠偏环

Planner 不是「切一次交货」——它切完先过四查门，不过就带诊断重切，撞满预算才认输。II-a 是这条主环；II-b 拆开那道四查门到底查什么。

图版 II-a · 自纠偏主环（produce → validate → recut）DYNAMIC · self-correcting loop
1
Planner._produce · 分流两分支

greenfield（repo_dir=None）：单次 with_structured_output(PlanOutput) 出切片。brownfield（repo_dir 存在）：先只读 agentic pass 扫既有 checkout，再结构化 emit。emit 之后两分支完全一致。

产物是 raw PlanOutput：id 空、blocked_by 用前驱下标。

2
Planner._self_critique · 一轮软自检

base chat.ainvoke（非结构化产出调用，与 produce 分开可观测）问三事：切片太大？漏依赖边？纯为并行硬拆的假独立切片？只给批评，不重写。

与机械门互补：门是硬裁决（结构/覆盖），自检是软反思（切法好坏）；批评喂进重切修复提示。

3
_assign_slice_ids · 确定性 stamp

id = uuid5(UUID(workflow_id), str(index))，并把 blocked_by 里的前驱下标重映射成对应切片的 stamped id。下标越界 / 非法条目原样保留——留给校验门当悬空引用逮住。

4
validate_plan · 四查门（全确定性，无 LLM/Docker）

① 引用完整 ② 无环（Kahn）③ 覆盖度 ④ scaffold 结构 → 返回 PlanValidation{ok, failures}。门控 by scope：③需 prd+spec，④仅 greenfield。（II-b 展开。）

5
分叉 · ok? / recut? / needs_human?

ok → PlanResult(status="ok")，交下游。fail 且 recut < 2 → 带「机械失败 + 自检批评 + 上一版概要」重切（回步 1，recut+1）。撞满 MAX_RECUTS=2 → 返回迄今最佳计划（失败最少那版）+ status="needs_human"。

一次 plan 至多 3 次 produce 调用。不 raise、绝不把坏 DAG 甩给下游。

为什么有界：无限重切会撞 #57 式空转。至多 2 次额外重切（≤3 次产出）是刻意的止损；撞满即降级交人，而不是硬撑。天然串行的需求切成链形合法 DAG 是合法 ok 输出——绝不为「制造并行」硬拆成假独立切片。

图版 II-b · 四查门拆解（validate_plan）DYNAMIC · validation gate
①
引用完整 · reference integrity

每条 blocked_by 都指向计划里真实存在的 slice id。LLM 可能编出不存在的前驱、或下标重映射后残留悬空引用——挡在下游编排之前。顺手挡重复 id。

门控：只需 plan，结构两查可 standalone 复用。

②
无环 · acyclicity（Kahn 拓扑排序）

切片 + blocked_by 边必须是 DAG。编排器按拓扑波次并行 spawn，成环会让波次推进死锁。Kahn 排序：排不完即有环；自环（a blocked_by a）天然被逮（入度永为正）。

③
覆盖度 · coverage

每条 PRD AC 被某切片溯源（_normalize 后规范化子串匹配 traces_to ∪ acceptance_criteria）、每个 TechSpec 组件被某切片 traces_to 或 file_footprint 碰到——否则漏了需求/组件。

门控：需 prd + tech_spec，缺则跳过。规范化容忍 LLM 格式漂移但非空洞（见深挖 ⑥）。

④
scaffold 结构 · 仅 greenfield

必有 wave-0 kind="scaffold" 切片（且 blocked_by 为空、落拓扑第 0 波），且每个 feature 切片沿 blocked_by 反向游走能传递触达 scaffold——「先立结构、后并行填肉」编进 DAG。

门控：仅 scope=="greenfield"；brownfield 骨架已在、跳过。

为什么校验分门（by scope）：greenfield 与 brownfield 两分支都过 ①②③，唯 ④ 只查 greenfield——brownfield 无「先立骨架」语义（骨架已在既有仓库）。这样一道 validate_plan 用同一套代码服务两条分支，靠 scope / prd·tech_spec 入参有无来开关对应检查。

端到端 · 走一遍

一版切错的计划，如何被自己逮住并重切成合法 DAG

用一个 greenfield 短链服务，看 Planner 首版漏了依赖 → 四查门 ④ 逮住 → 带诊断有界重切 → 合法 DAG 的完整链路。这正是「自校验的规划者」精髓。

场景 · greenfield · URL 短链服务

重新 stamp + validate_plan → 四查全过。返回 PlanResult(status="ok")。DAG 拓扑波次清晰，交下游并行编排 ②。

# 拓扑波次（编排器按此 fan-out）
wave-0:  [0] 项目骨架 (scaffold)
wave-1:  [1] 短码生成
wave-2:  [2] 重定向  ‖  [3] 点击统计   # 二者无互相 blocked_by → 并行

[2] 和 [3] 都碰 analytics/registry 类共享文件，但不因文件重叠强制串行化——残余的 co-edit 冲突留给未来 Merge Agent 调和（见深挖 ④）。这就是「Blocked-by 只编码逻辑依赖」的价值：并行度不被共享文件杀光。

核心机制 · 钻到实现级

六个撑起 Planner 的机制

展开看代码级细节。这几个点也是面试里最容易被追问「你具体怎么实现的」的地方。

① 四查门：一道全确定性的机械校验，为什么无 LLM？

validate_plan 只查结构与可追溯，不判切法好坏——后者是 LLM 自检的事。四查全是纯函数、不触 LLM/Docker，所以可纯测、可确定性复放、零成本。做成返回 PlanValidation{ok, failures} 而非 raise：一次把所有结构问题暴露给调用方，由 Planner.plan 决定处置。

def validate_plan(plan, prd=None, tech_spec=None, scope=None):
    failures = []
    failures += _structural_failures(plan)          # ①引用完整 ②无环
    if prd and tech_spec:
        failures += _coverage_failures(plan, prd, tech_spec)  # ③覆盖度
    if scope == "greenfield":
        failures += _scaffold_failures(plan)        # ④scaffold 结构
    return PlanValidation(ok=not failures, failures=failures)

无环用 Kahn 拓扑排序：只对存在的依赖建边（悬空引用已在①报过，不在②重复噪声），排完 visited != len(slices) 即有环。scaffold 结构用反向游走：每个 feature 沿 blocked_by 走，seen 去重防成环死循环，能传递触达任一 scaffold 即合规。门控 by scope / 入参有无——同一道门服务 greenfield 与 brownfield 两分支。

② 有界重切：自纠偏，但为什么必须「有界」？

四查门失败不是终点。Planner 先给一轮 LLM 自我批判，再带诊断重切，至多 MAX_RECUTS=2 次（故一次 plan 至多 3 次 produce 调用）。撞满仍不过 → 返回迄今最佳计划（失败最少那版）+ needs_human。

raw = await self._produce(...)          # 首版
while True:
    critique = await self._self_critique(raw, ...)   # 软批评
    stamped = _assign_slice_ids(raw, workflow_id)
    validation = validate_plan(stamped, prd, tech_spec, scope=scope)
    # best-so-far：失败最少的一版兜底
    if best is None or len(validation.failures) < len(best.failures):
        best_stamped, best = stamped, validation
    if validation.ok:      return PlanResult("ok", stamped)
    if recut >= MAX_RECUTS: return PlanResult("needs_human", best_stamped, best.failures)
    recut += 1
    raw = await self._produce(..., repair=_RepairContext(...))  # 带诊断重切
  • 为什么有界：无限重切会撞 #57 式空转（闭环模块的经典坑）。2 次是止损，撞满即降级交人，不硬撑。
  • best-so-far：即便最终交人，也交失败最少的那版，人接手成本最低（首版兜底、相等不替换保留更早版）。
  • 不 raise、不甩坏 DAG：needs_human 是显式状态，未来编排器映射为 resolve_plan 人工闸门 + Temporal Signal。
③ greenfield vs brownfield：一个 repo_dir 分出两条产出分支

Planner.plan(repo_dir=…) 分流两条分支，emit 之后完全一致（同一自纠偏尾）：

维度greenfield（repo_dir=None）brownfield（repo_dir 存在）
产出方式单次结构化 LLM 调用只读 agentic pass → 结构化 emit
工具无read / glob / grep（只读）
file_footprint预测的新路径（近似）命名真实文件
scaffold 切片必出 wave-0不出（骨架已在）
四查门 ④查 scaffold 结构跳过
沙箱无无（读宿主机 checkout）

为什么 brownfield 无需沙箱：只读一份 checkout 是安全的，需要隔离的是 mutation（写/改/跑）。为什么只给只读工具：Planner 是规划者、不是实现者——make_planner_read_tools 绝无 write/edit/bash（承重不变式，测试正向断言工具集里无任何 mutating 工具）。read 的 path 受未受信 checkout 内容控制（prompt-injection 面），_resolve_in_repo 强制落点在 repo_dir 内，../ 穿越 / 仓库外绝对路径一律判逃逸。

④ Blocked-by 语义：为什么文件重叠不该强制串行化？

blocked_by 只编码主要编码逻辑依赖（B 用到 A 建的符号/接口/表 → B blocked_by A）。预测文件足迹的重叠只作次级信号，不因两切片碰同一文件就强制串行化——否则并行度会被共享文件杀光。

  • greenfield 先出 scaffold：wave-0 骨架切片建目录布局 / 包 manifest / 共享 config / 基础接口表，所有 feature blocked_by 它——把「先立结构、后并行填肉」编进 DAG，消掉 greenfield 绝大多数冲突。
  • 残余的共享聚合文件冲突（router 注册表、__init__.py 导出、models.py 索引、迁移序列——少而规整、多为 append 形）刻意不串行化，交下游 Merge Agent 自动调和。
  • merge-keyed downstream：下游 Blocked-by「怎么看到前驱产出」——前驱代码就在下游检出的集成分支基里。DAG 波次推进即集成分支增量前移，杜绝双 tracker 漂移（ADR-0002）。
设计品味「并行边界」由逻辑依赖划，不由文件重叠划。文件重叠是 Merge 阶段的问题，不该在 Planner 阶段用串行化提前买单——各阶段各管各的便宜。
⑤ 确定性 id：为什么 LLM 不产 id、用下标？

Slice.id = uuid5(UUID(workflow_id), str(index))——确定性 stamp，遵 Temporal 确定性铁律（禁 now()/random()/uuid4()）。它同时是未来 GitHub issue 投影的幂等键（ADR-0002 §3.3：投影 activity 是 at-least-once，重试可能重复关 issue / 贴评论，幂等键取自确定性 slice id 才能查重去抖）。

ns = uuid.UUID(workflow_id)
ids = [str(uuid.uuid5(ns, str(i))) for i in range(len(raw.slices))]
index_to_id = {str(i): ids[i] for i in range(len(raw.slices))}
# blocked_by 的前驱下标 → 重映射成 stamped id；越界/非法条目原样保留
remapped = [index_to_id.get(dep, dep) for dep in s.blocked_by]

为什么 LLM 只产下标不产 id：LLM 无从产出确定性 id（它会 hallucinate 随机 UUID，破坏重放）。所以让 LLM 只用前驱切片的 0-based 下标字符串表达依赖（「依赖第 0 个切片」写 blocked_by=["0"]），Planner stamp 后重映射。下标越界 / 非下标条目不静默吞掉——原样保留，让 validate_plan 当悬空引用显式逮住 → needs_human。

⑥ 为什么校验自己的输出？兼：覆盖度门为何用规范化子串

Planner 自审自纠，刻意不引入独立的 planner-reviewer agent——① 下游 Loop B（闭环 Coder + 对抗式 Reviewer）本身兜底；② eval「Reviewer 价值 regime-dependent」结论支撑省掉它。自校验的边界是机械可判的结构问题（引用/环/覆盖/骨架），切法好坏交软自检 + 下游。

覆盖度门有个承重的硬化（#78，正是它让 live e2e 能过）：AC 覆盖从「原文 exact-string 相等」放宽为 _normalize 后的有序子串包含：

def _normalize(text):
    # 小写 + 把标点/下划线/空白折叠成单空格
    return re.sub(r"[\W_]+", " ", text.lower()).strip()
# add(2,3)==5 → "add 2 3 5"，仍要求这串词按序出现在某切片证据里
教训 · 鲁棒但非空洞真实 LLM 回填 AC 溯源不会逐字复刻原文（大小写/空白/标点会漂），exact-string 会伪失败。折叠成词元序列做子串包含——容忍格式漂移，但保留词序故非空洞（真无覆盖照样失败）。配套双侧修复：两个 system prompt 新增「traces_to 覆盖契约」，明确要求 LLM 把每条满足的 PRD AC 原文放进切片 traces_to。
面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我做了个切片器」，而是每一个决策背后都有一句「为什么必须这样」。

为什么 Slice ≠ user story？

两条不同的轴

「需求边界」（PM 的 story）与「可并行实现边界」（Planner 的 slice）本质不同。切法归 Planner 自由裁量——一对多 / 多对一 / 无中生有（TechSpec 组件），才能最大化拓扑并行。

为什么 Planner 校验自己的输出？

不甩坏 DAG 给下游

结构问题（引用/环/覆盖/骨架）是机械可判的，一道全确定性的四查门零成本兜住；失败带诊断有界重切，撞满交人。绝不让一张坏图流进并行编排。

为什么内部 DAG 而非 GitHub issue 驱动协调？

Temporal 确定性重放

workflow 函数体不能读外部可变状态（issue 的 open/closed）驱动控制流。内部 DAG 活在 Temporal state = 唯一真相源；GitHub issue 降为单向投影 + 入口，绝不回读（ADR-0002）。

为什么 id 是确定性 uuid5、LLM 不产？

铁律 + 幂等投影键

遵 Temporal 禁 now/random/uuid4；同时是 GitHub 投影的幂等键（at-least-once 重试要查重）。LLM 产不出确定性 id，只用前驱下标，Planner 重映射。

为什么 brownfield 只给只读工具？

规划者非实现者

只读一份 checkout 安全（mutation 才需隔离），故无需沙箱、直读宿主机。绝无 write/edit/bash——承重不变式测试钉死，兼防未受信 checkout 的 prompt-injection 越权改码。

天然串行的需求会被判失败吗？

链形也是合法 DAG

不会。一条 blocked_by 链本身合法、过四查门。自检明确不为「制造并行」硬拆假独立切片——强拆是拿确定性换方差，还会在下游 Merge 阶段付更贵的账。
