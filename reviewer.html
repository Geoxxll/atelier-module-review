Plate 06 — Adversarial Reviewer

对抗式 Reviewer

给闭环 Coder 配一个不信它自测的对头：换台干净机器把测试重跑一遍，只看产物、不听辩解。

一句话：闭环 Coder 会「自写自测、assert True 糊弄绿」。对抗式 Reviewer 的结构性堵法是无视 Coder 的「自测绿」——在干净新容器里独立重跑测试，用独立 context（不喂 Coder 的对话历史）只审 git diff。红测硬压 approve，reject 则进 Loop B（≤2 轮修复闭环）。

「自测绿」为什么不能当成功

上一板的闭环 Coder 有一条终止门：done 前必须自测绿。但在 greenfield（空仓库）里，测试也是 Coder 自己写的——于是它可以：

  • 写空洞测试——assert True、恒真断言、根本没 import 被测实现，跑了等于没跑，却让 pytest 全绿。
  • 靠脏环境喂绿——在自己的容器里手动 pip install 了某个依赖、却没写进 requirements.txt；它自己跑当然绿，换台机器 clone 下来就 ModuleNotFoundError。

这就是 self-gaming：Agent 优化的是「让我给自己出的考卷得满分」，不是「真的达标」。level-0 只能靠事后盲评兜底；level-1 要给一个结构性堵法——引入一个对抗方，它压根不信 Coder 的自测，自己在一个干净新容器里把测试重跑一遍，以 git diff 为焦点判 approve / reject。

贯穿全篇的一句话 · 只看产物，不看推理 对抗有效的前提是 Reviewer 看不到 Coder 的思考与自辩。喂它 Coder 的对话历史，它会倾向「认可刚生成的东西」（generation = endorsement 偏置），对抗当场失效。所以 review() 的签名根本不收 Coder messages——这是结构性的、不是省 token。
贯穿全篇的一句话 · 「对抗 review 净有害」几乎总是 Reviewer 的 bug β 首跑实测对抗 Reviewer 净有害（lift −0.23）——反直觉。追因发现是我们 Reviewer 的三个缺陷（反射性拒绝 / 独立重跑撞无关坏测试 / 常不提交裁决），不是对抗 review 的性质。修好后符号稳定翻成 +0.1。全篇最值得讲的一段，见「深挖 ④」。

为什么必须「干净新容器」而非复用 Coder 的容器

这是整个对抗设计的物理命门。Reviewer 不复用 Coder 那个 live 容器，而是按 tech_stack 起一个全新的、挂同一个 workspace 目录——复现的是「换台机器 clone 下来能不能跑」。

环境Coder 装的全局包 / 改的环境全在从镜像起、按 tech_stack 重新 provision
喂绿风险被中间产物 / 手动装的依赖喂绿manifest 没写的依赖当场 ModuleNotFoundError
复现语义「在作者机器上能跑」「换台机器 clone 能不能跑」
逮什么逮不到 self-gaming依赖漏写 / 测试依赖脏环境 / 根本没跑
隔离共享边界，对抗纯度破全新沙箱 start→跑→stop，即用即焚

一眼看穿：复用容器等于让考生用自己那台「装好了一切」的机器判卷——「独立重跑」名存实亡。干净容器把「作者机器上的巧合」剥干净，剩下的才是真实的「这份代码本身行不行」。开销是每次审查多一次起容器 + provision（几十秒级）——standalone PoC 里对抗纯度优先，不在乎吞吐。

图版 II · 动态流程

Loop B 怎么转，self-gaming 在哪一步被逮

左边是「一圈 Loop B」：Coder 产出 → 独立重跑 → 审 diff → 裁决 → reject 则 fix 新开。右边聚焦「self-gaming 阻断路径」：为什么信自测会漏、独立重跑怎么接住。

图版 II-a · 一圈 Loop B（Coder↔Reviewer 修复闭环）DYNAMIC · one loop
0
round 0 · 闭环 Coder 初次产出

run_review_loop_task 起一个 workspace，闭环 Coder（level-0）跑到自测绿、产物落磁盘。collect_git_artifacts 捞 git diff + 文件树。

1
Reviewer · 干净容器独立重跑（S1）

IndependentTestRunner.run 起全新沙箱、挂同一 workspace、跑 python -m pytest -q --continue-on-collection-errors → IndependentTestResult.passed。无视 Coder 的自测绿。

Reviewer 不知道 Coder 跑的是哪条命令（没看它的历史），按运行时推一条通用入口。

2
Reviewer · 独立 context 审 diff

起手只给 原始需求（PRD+TechSpec AC）+ git diff + 独立重跑摘要，不含 messages / PLAN / DECISIONS。ReAct loop 里用 read/grep 按需 pull → 调 submit_review 提交结构化裁决。

3
_finalize · 裁决合成

approved = independent_tests_passed AND (llm_approved OR 无 blocking issue)。红测硬压 approve；测试绿且指不出 critical/major 缺陷 → 通过（纠「反射性拒绝」）。

4
approve → 终局 · reject → fix 新开

approve（Reviewer 终局）→ 任务完成。reject → fix-Coder 精简新开 一次全新 AgenticCoder.run（fresh 容器 + fresh seed = issues + diff + 反驳指引），从磁盘 re-ground，可用 rebut 带依据反驳。

5
复审 → 回到 1，≤2 轮

带上 Coder 的反驳回到第 1 步 re-review。Reviewer 复审：被 rationale 说服就撤回该 issue，不服就保持。2 轮仍未 approve → 终止 not-approved(failed)，记未解决 issues + 反驳。

对称美学：Reviewer（第 2 步）和 fix-Coder（第 4 步）都从磁盘 artifact re-ground、都不背对话历史——「磁盘是唯一真相源、context 一次性」贯彻到双方。反驳权（D5）让 Coder 能带 DECISIONS.md 依据反驳不合理 issue，因为 Reviewer 非 ground truth；但红测不可反驳绕过，反驳只对主观 issue 生效。

对抗方 · 独立重跑接住

①
干净新容器 provision（D2）

从镜像起、按 tech_stack 重新装工具、挂同一 workspace——作者机器上的巧合被剥干净。

②
独立重跑 = 换台机器 clone

python -m pytest 一跑：漏写的依赖 ModuleNotFoundError、空洞测试被 read/grep 的判断逮住 → 红。

③
硬信号压过 approve

independent_tests_passed=False → _finalize 里 必 False，无论 LLM 说什么。判 not-approved → 进 Loop B 真修。

结构性 > 事后抽查：level-0 靠盲评是「交完卷再抽查」；对抗式 Reviewer 是「交卷前换台机器强制重考一遍」。前者漏网率靠运气，后者把「作者环境」这个变量物理地消掉——这就是「结构性堵法」和「兜底」的区别。

这就是对抗闭环的价值：self-gaming 被结构性逮住（不靠运气抽查），且逮住后有 Loop B 把它修回真绿，而不只是打个红叉。

核心机制 · 钻到实现级

五个撑起对抗式 Reviewer 的机制

展开看代码级细节。这几处也是面试里最容易被追问「你到底怎么保证对抗有效、又不误杀」的地方。

① 独立重跑：干净容器怎么跑，怎么容忍无关坏测试

IndependentTestRunner.run 每次都经工厂起一个全新沙箱（start→跑→stop），绝不复用任何外部 live 容器——这是对抗纯度的隔离边界（D2）。默认测试命令按运行时推（Reviewer 没看 Coder 历史，不知道它跑的是哪条）：

_DEFAULT_TEST_COMMAND_BY_RUNTIME = {
    # `python -m`：cwd 进 sys.path → 被测仓库 in-tree 包可 import（对齐 held-out oracle）
    "python": "python -m pytest -q --continue-on-collection-errors",
    "node": "npm test",
}

β Finding 2 的坑：早期 passed = (exit_code == 0)。但整套跑仓库测试会撞到与改动无关、只是缺可选依赖（如 funcy 的 test_funcs.py import 第三方 whatever）而 collection 失败的模块 → 整轮退出码非零 → 误判红 → Reviewer 对环境噪声而非修复质量 reject / churn。修法是容忍无关 collection error：

def _passed_tolerating_unrelated_collection_errors(exit_code, stdout, stderr):
    if exit_code == 0: return True
    has_passed = # 匹配 "N passed" 且 N > 0
    has_failed = # 匹配 "N failed"
    return has_passed and not has_failed   # 有测试真跑过、且无 failed → 过
  • 反自欺硬信号不破：改动真弄坏了能收集的测试会出 failed → 仍红；「啥都没跑起来」（0 passed）→ 也不过。只放过「无关模块没收集上」。
  • 为什么提到模块根（#84）：这套「干净容器独立重跑」是模块级共享基座，不专属 Reviewer——level-2 的 Merge 波回归门要在合并候选上跑同样的「集成分支干净重跑」，是同一个原语。reviewer/independent_test_runner.py 转为 re-export shim，调用点零回归。
② 独立 context：为什么 review() 根本不收 Coder messages

对抗有效的结构性前提：Reviewer 看不到 Coder 的推理与自辩。喂它 Coder 的对话历史，它会倾向「认可刚生成的东西」（generation = endorsement 偏置），对抗当场失效。这不是省 token——是把偏置从签名层就杜绝：

async def review(self, *, requirements: str, diff: str,
                 workspace_dir, rebuttals=None) -> ReviewOutput:
    # 签名里根本没有 coder_messages 参数 —— 结构上无法喂
    seed = [SystemMessage(REVIEWER_SYSTEM_PROMPT),
            HumanMessage(build_review_context(   # 只有需求 + diff + 独立重跑摘要
                requirements=..., diff=..., test_result=...))]
  • pull-based 上下文：起手 context 不含 PLAN.md / DECISIONS.md；Reviewer 想看作者的计划与理由，得自己用 read/grep 主动拉（rationale 也在磁盘）——「只看产物不看推理」，但产物包括磁盘上的一切 artifact。
  • 只读、走 host 文件系统：read/grep 在宿主机 bind-mount 目录上跑，不碰 Docker（确定性）。与 coder 侧 tools.py（走容器 exec）平行但刻意不抽共享接口——共享工具接口会给两边打通一条隐性耦合，破坏对抗隔离。
  • 双层输入（level-2 Planner 后生效）：局部 issue 的 AC = 判 approve/reject 的主裁判依据；全局 PRD/TechSpec = 护栏（防切片破坏系统级假设），不拿全局 AC 苛求单个切片。
③ verdict finish-guard：Coder 有、Reviewer 缺，补上

闭环 Coder 有 finish_guard（done 前必绿测）。Reviewer 的对应硬前置是「done 前必提交结构化裁决」——但一开始没有。β Finding 3 逮到：Reviewer 常在远没撞 max_iter 时（iteration=6/8）自认审完、输出自然语言结论却没调 submit_review → holder["verdict"]=None → _finalize 保守 not-approved → Loop B 误判 reject 逼 Coder 无谓返工 churn。

根因是 harness 责任，不是模型问题：结构化输出的可靠收口是业界公认难题，Coder 早有闸门、Reviewer 缺。修法复用内核既有的 finish_guard 钩子：

def guard(messages):
    if holder.get("verdict") is not None: return None   # 已提交 → 放行
    if state["reprompts"] >= self.verdict_guard_retries:
        return None                    # 逼单耗尽 → 放行走保守兜底（不无限拦）
    state["reprompts"] += 1
    return HumanMessage(VERDICT_REQUIRED_MESSAGE)   # 塞回逼单，令 loop 继续
弱 vs 强 · 把「模型 × 闸门」拆干净 no_verdict 次数：gpt-4o-mini 无 guard 13 → +guard 6 → gemini-3.5-flash +guard 0（gemini 32 次 submit、guard 一次没触发）。结论：finish-guard 是弱模型的安全网、强模型压根不需要；残留的 6 是 gpt-4o-mini 同预算收敛不完（能力问题，guard 管不到「还在探索就没了」）。两手都要——可能被迫用弱 reviewer。

行为严格不劣：逼单有界（DEFAULT_VERDICT_GUARD_RETRIES=3），顽固模型逼满后放行、退化到既有保守兜底；max_iterations 另有兜底，绝不无限循环。

④ 三缺陷冒充「对抗 review 无用」：−0.23 → +0.1 的诚实故事

β pilot（真 GitHub issue + SWE-bench 式 oracle）测「闭环 Coder 加不加对抗 Reviewer」，隔离「Reviewer」这一个变量。gemini 首跑 coder_reviewer 净有害（lift −0.23，比 solo 更差）——反直觉（真实开发中对抗 review 普遍提升质量）。查 review_loop.round_reviewed 逐轮日志，挖出三个 Reviewer bug，全不是对抗 review 的性质：

Finding缺陷修法
① 反射性拒绝LLM 从不 approve（0/16），连绿+零 issue 也拒 → 烧满 fix 预算 churn 坏正确改动system prompt 补明确 approve 规则（绿+指不出实质缺陷→必须 approve，minor 走 suggestions）+ _finalize 一致性兜底
② 撞无关坏测试整套 pytest 撞到缺可选依赖的无关模块 collection 失败 → 假红 → 对环境噪声 reject--continue-on-collection-errors + 容忍无关 collection error（见深挖①）
③ 不提交裁决no_verdict 13 vs submit 4 → 保守 not-approved → 无谓返工verdict finish-guard（见深挖③）

修后 β 符号稳定翻转：lift −0.23 → +0.11 → +0.10（两次干净跑一致净有益）。关键洞察：机制测试（S1–S4「Loop B 能不能跑」）全绿也发现不了这三个 bug——只有客观 held-out oracle A/B 逼得出来。「机制能跑 ≠ 机制有用」。

教训 · 定胜负的信号 vs 上色的信号Reviewer 自己的 approve/reject 不可信（从不 approve / 撞噪声 / 不出裁决全能发生）——必须拿客观 held-out oracle 校准。这正是 β 用 oracle 定胜负、judge 只上色的意义。弱能力会冒充「对抗 review 无用」，同 level-0「harness 缺陷冒充模型弱」的模式。
⑤ 裁决合成与「价值分区」：红测不可绕过、收益看阵营

_finalize 把 LLM 的 verdict 与独立重跑硬信号合成最终 ReviewOutput。承重公式（含 β 纠偏后的一致性兜底）：

blocking = [i for i in verdict.issues if i.severity in ("critical","major")]
llm_approved = bool(verdict.approved)
# 测试绿 + 无 blocking → 通过（纠反射性拒绝）；红测始终压过（passed=False → 必 False）
approved = test_result.passed and (llm_approved or not blocking)
  • 红测不可反驳绕过：test_result.passed=False 时 approved 恒 False——generation=endorsement 偏置不得把红测说绿，Coder 的 rebut 反驳权只对主观 issue 生效、对客观红测不生效。
  • 纠反射性拒绝：测试绿且无 critical/major issue 时视为通过——「拒绝但指不出实质缺陷」只会逼无谓返工。硬信号方向不变，有 blocking 的合法 reject 照样尊重。

价值是「规制依赖」（regime-dependent）的——对抗 Reviewer 不是「一律有用」，收益取决于 Coder 和 Reviewer 的强弱阵营：

阵营净收益为什么
弱 Coder × 强且修好的 Reviewer显著为正Reviewer 逮住弱 Coder 的真缺陷、Loop B 修回
强 Coder中性Coder 本来就对，Reviewer 挑不出真问题
坏 Reviewer（三缺陷未修）净有害永不放行 / 撞噪声 / 不出裁决 → 只 churn，拿不到「够好就停」
承重纪律 · level-1 是独立变量level-1（加 Reviewer）与 level-0（闭环 Coder）是两个独立 A/B 变量，必须分开验证——「closed-loop Coder + 新 Reviewer」双变量同接会污染归因。这也是 β 只隔离「加不加对抗 Reviewer」一个变量的原因。
面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我做了个 code reviewer」，而是每一处取舍背后都有一句「为什么必须这样，否则对抗就失效」。

怎么堵 Coder 自写自测糊弄绿？

不信自测，干净容器独立重跑

Reviewer 无视 Coder 的自测绿，在干净新容器里按 tech_stack 重新 provision + 挂同一 workspace 重跑——复现「换台机器 clone 能不能跑」，逮出依赖漏写 / 空洞测试 / 脏环境喂绿。

为什么不喂 Reviewer Coder 的历史？

防 generation = endorsement

看到 Coder 的推理与自辩，Reviewer 会倾向「认可刚生成的东西」，对抗失效。review() 签名根本不收 messages——从结构层杜绝偏置，不是省 token。只看产物不看推理。

「对抗 review 净有害」怎么解释？

几乎总是 Reviewer 的 bug

β 首跑 −0.23，追因是我们三个 Reviewer bug（反射性拒绝 / 撞无关坏测试 / 不出裁决），不是对抗 review 的性质。修好后翻成 +0.1。关键在「代码正确时 approve、让循环停下」。

机制测试全绿，为什么还漏 bug？

机制能跑 ≠ 机制有用

S1–S4 测「Loop B 能不能跑」全绿，却发现不了三个 bug——只有客观 held-out oracle A/B 逼得出。Reviewer 自己的 approve/reject 不可信，必须拿外部 oracle 定胜负、judge 只上色。

Reviewer 会不会误杀？Coder 能反驳吗？

反驳权，但红测不可绕过

Reviewer 非 ground truth，Coder 可凭 DECISIONS.md 用 rebut 带依据反驳 → Reviewer 复审撤回或保持。但反驳 ≠ 单方驳回，且客观红测不可反驳绕过，只对主观 issue 生效。

verdict finish-guard 是干嘛的？

结构化收口是 harness 责任

Reviewer 常自认审完却不调 submit_review → 保守 not-approved → 无谓 churn。复用内核 finish_guard 钩子逼它出裁决（有界）。量化：弱模型 13→6、强模型 0——弱模型安全网、强模型不需要。
