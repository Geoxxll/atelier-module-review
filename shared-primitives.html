共享基座

闭环管线四个 Agent 脚下那张「共用工具台」——被所有人依赖、自己谁也不依赖的单向叶子。

一句话：把容器执行、独立重跑、host-git 安全出口、ReAct 内核、镜像查表、翻页纯逻辑这六件谁都要用的原语抽到模块根，让 Coder / Reviewer / Merge / Orchestrator 只向下依赖它、彼此不横向勾连。这就是一次干净的依赖倒置——它掐死了 reviewer→coder 的横向耦合（#51）。

为什么要一个「谁都依赖、自己不依赖谁」的叶子？

想象一间木工工坊里的公用工具台：钉在墙上的电钻、台锯、量规。四个师傅——Coder、Reviewer、Merge、Orchestrator——各有自己的工位，干活时都伸手去够同一张工具台，但谁也不许跑到别人工位上借私人工具。

  • 如果没有这张工具台：Reviewer 想「起个干净容器重跑测试」，就得去 import coder 里那套容器代码——于是 reviewer 反向依赖 coder，两个本该独立演化的兄弟被焊死（这正是 #51 要掐死的耦合）。
  • 有了工具台：容器原语、独立重跑、git 出口全抽到模块根成为叶子，四个 Agent 一律向下依赖它。叶子不依赖任何兄弟，改动它才不会牵一发动全身。

这是一次教科书式的依赖倒置（dependency inversion）：不让高层组件互相依赖具体实现，而是让它们共同依赖一个下沉的稳定叶子。ADR-0012 把它升级成一条铁律——闭环包不许反向依赖 legacy 生产包、兄弟之间不许横向 reach。这也是「共享基座」比「一堆 util 函数」值钱的地方：它是一个被刻意约束了依赖方向的结构。

贯穿全篇的设计美学 · 单向叶子 六个原语的每一份源码开头都写着同一句话：「抽到模块根的缘由——这本是模块级共享基座，不专属某个子目录」。而老调用点靠 re-export shim 原地不动、import 路径零回归。「深挖 ⑥」有完整的依赖方向规则。

六件原语，各是什么、谁在用

II-a · IndependentTestRunner.run() 干净重跑

1
__init__ · 镜像/命令解析

resolve_image(tech_stack) → 镜像；provision_commands_for_image → 预置命令；据运行时推默认 python -m pytest -q。

全走 provisioning.py 同一套查表——与 Coder 起容器用的完全一致，隔离才公平。

2
sandbox_factory · 起全新容器

每次 run 经工厂新起一个 LiveSandbox（惰性 import），绝不复用任何外部 live 容器。

3
start → aexec(test_command)

bind-mount 同一 workspace 进干净容器，异步跑测试命令拿 ExecResult。

4
判过 + finally stop

_passed_tolerating_unrelated_collection_errors 定 passed；finally 里销毁容器（即用即焚）。

容忍无关模块 collection error：真弄坏能收集的测试→failed→不过；啥都没跑起来→也不过。

✓
IndependentTestResult

passed = 堵 self-gaming 的硬信号（= ReviewOutput / RegressionGateOutcome 的 passed）+ summary。

II-b · run_git() host 安全出口

1
调用方 · Merge / Orchestrator

任何 git 子进程只准走 run_git——merge/ 内唯一的 git 出口，安全配置集中一处、不可能漏。

2
注入 _SAFETY_CONFIG（命令行 -c）

core.hooksPath=/dev/null 中和一切 hooks、fsmonitor=false、protocol.ext.allow=never。命令行 -c 优先级高于仓库 local config。

/dev/null 不是目录 = hooks 目录恒为空，比逐个禁 hook 名单更彻底（新 hook 类型也免疫）。

3
env 再加一道

GIT_CONFIG_GLOBAL/SYSTEM=/dev/null 屏蔽用户/系统 gitconfig；GIT_TERMINAL_PROMPT=0 绝不交互挂起。

行为只由「仓库内容 + 本模块显式配置」决定 → 确定性、Temporal 友好。

4
subprocess.run + check

capture stdout/stderr；非零且 check → 抛 GitCommandError（带命令 + stderr，供上层判「真冲突 vs 基建错误」）。

✓
CompletedProcess / rev_parse

固定身份 atelier-merge-agent 保证 merge commit 确定性、不吸 host 用户配置。

两条时序的共同气质：都在把不可信输入关在门外。II-a 用「全新干净容器」隔离掉 Coder 可能装的脏环境（否则独立重跑被喂绿、名存实亡）；II-b 用「/dev/null hooks + 屏蔽 gitconfig」隔离掉 agent 产出的仓库可能埋的 booby-trapped hooks。原语的价值不只是复用，更是把一处安全边界写对一次、所有调用点共享。

端到端 · 走一遍

同一个原语，被 Reviewer 和 Merge 用在同一个仓库上

最能体现「单一来源复用」的例子：IndependentTestRunner 一份代码，先给对抗式 Reviewer 当独立重跑，又给 Merge 波回归门当默认 runner——两处跑的是同一套镜像 / provision / 隔离容器。

场景 · 一份 runner · 两个消费者 · 同一 workspace

切片改动如何被「同一把尺子」量两次

① Reviewer

对抗式 Reviewer 拿到 Coder 产出的 workspace，不看 Coder 的对话历史（只看产物），起一个干净容器独立重跑。

# reviewer/review_agent.py —— import 走 re-export shim，路径不回归
# from reviewer.independent_test_runner import IndependentTestRunner
runner = IndependentTestRunner(tech_stack=tech_spec.tech_stack)
result = await runner.run(workspace_dir=coder_workspace)
review.independent_tests_passed = result.passed   # 堵 self-gaming 的硬信号
② Merge

同一个 run 里，Merge 波回归门要在集成分支的合并候选上跑「项目自己的测试套件干净重跑」——它不另造轮子，直接复用同一个原语。

# merge/real_defaults.py · make_default_regression_gate
# 波回归门 = 复用模块根 IndependentTestRunner（ADR-0003 §3.4）
runner = IndependentTestRunner(tech_stack=...)
outcome = await runner.run(workspace_dir=candidate_worktree)
gate = RegressionGateOutcome(passed=outcome.passed)  # 绿了集成头才前移
单一来源

两处的 passed 来自同一个 _passed_tolerating_unrelated_collection_errors、同一套 resolve_image / provision、同一个 LiveSandbox 隔离边界。

# 若各写各的：Reviewer 与 Merge 判「过」的口径就可能漂移
# —— 同一改动在两处得出不同结论，回归门与复审自相矛盾
单一来源 → 隔离边界一致、判过口径一致、修一处两处都受益

这就是叶子基座的兑现：不是「少写点代码」，而是让两个独立子系统对「测试到底过没过」达成同一个定义。

反例 · 若横向 reach

假如当初让 Reviewer 直接 import coder 里的容器代码：Merge 想复用就得再 import Reviewer，形成 coder ← reviewer ← merge 的链式横向耦合，任一改动波及全线。

抽成叶子后三者都只向下依赖 independent_test_runner，彼此零感知——这正是 #51/#84 把原语上提的动机。

核心机制 · 钻到实现级

六个撑起基座的原语与一条铁律

展开看代码级细节。前五个是原语本身怎么实现的，第六个是「为什么它们该在这儿、依赖为什么必须单向」——面试里最容易被追问「你怎么切模块边界」的地方。

① LiveSandbox / ExecResult：长生命容器 vs 一次性焚烧

LiveSandbox 是 task 级长生命容器，与开环 DockerSandbox（跑完即焚、被动判分）根本不同：容器以 sleep infinity 空转保持在线，所有动作事后经 docker exec 注入，而不是 run 时塞死一个固定脚本。

self._container = client.containers.run(
    image, command=["sleep","infinity"], detach=True,
    volumes={workspace_dir: {"bind": "/workspace", "mode":"rw"}},
    mem_limit="512m", cpu_period=100000, cpu_quota=100000)  # 1 CPU
# exec：demux=True 才能分离 stdout/stderr（默认会合流拿不到干净 stderr）
stdout_b, stderr_b = container.exec_run(["sh","-c",cmd], demux=True).output
  • bind-mount：workspace 是宿主机 git init 目录，容器内写入对宿主即时可见——产出物捞取靠它。bind-mount 源必须绝对路径，否则 Docker 当成 named volume。
  • provision best-effort：起容器后一次性装通用工具（ruff/pytest），失败只记 warning 不阻断——通用工具缺位不该让整个 task 起不来。
  • stop 幂等：finally / __exit__ 里 remove(force=True) 并吞掉异常，重复调用安全，杜绝僵尸容器。aexec 是把阻塞 docker SDK 丢线程池的异步包装。
② CustomReActAgent / finish_guard：从 legacy fork 出的通用内核

这套 ReAct 内核（LLM 决策 → 工具分发 → 执行 → _should_continue → max_iter 兜底）本就是通用的、与 GitHub/PR 无关。闭环包从 legacy coder-agent/react_agent.py 的 CustomReActAgent fork 一份、剥掉 PR 专用部分，只留通用语义 + 两个钩子：

# ainvoke 主循环：LLM 想结束（不再调工具）时，先过终止门
if not self._should_continue(ai_message):
    if finish_guard is not None:
        injected = finish_guard(messages)
        if injected is not None:        # 门未放行
            messages.append(injected); continue   # 逼它继续干，不结束
    break                                    # 放行 → 收工
  • finish_guard：闭环「done 前必自测绿」的硬前置——LLM 想收工时被拦，塞回一条「先去跑测试」消息逼它继续。max_iterations 始终兜底，guard 永远拦也不会死循环。
  • on_step：每轮工具执行后回调，返回 StepDecision(inject/stop)——用于 best-so-far 快照 / 无进展止损。
  • 为什么 fork 而非 import：见深挖 ⑥——实验包不许反向依赖 legacy 生产包，各持各的 loop 内核独立演化（ADR-0012 §3.2）。
③ IndependentTestRunner：为什么必须是「干净新容器」？

对抗式复审 / 波回归门的正确性全靠一个前提：独立重跑必须真独立。若复用 Coder 的 live 容器，会被 Coder 装的全局包 / 改的环境 / 中间产物喂绿，「独立」名存实亡。所以每次 run 都经工厂起一个全新沙箱，从镜像起、按 tech_stack 重新 provision、bind-mount 同一 workspace：

sandbox = self._sandbox_factory(image=..., provision_commands=...)
sandbox.start()
try:    exec_result = await sandbox.aexec(self.test_command)
finally: sandbox.stop()   # 干净容器即用即焚，隔离边界

它复现的是「换台机器 clone 下来能不能跑」，专逮依赖没写进 manifest / 测试依赖脏环境 / 根本没跑这类 self-gaming。判过逻辑还容忍无关模块的 collection error（β pilot 发现：缺可选依赖的无关测试模块会让整轮退出码非零，旧逻辑 passed=exit_code==0 误判为红，Reviewer 对环境噪声无脑 reject）。

复用点同一个 runner 被 Reviewer（对抗 review）与 Merge（波回归门 real_defaults）共用——见「端到端」。sandbox_factory 可注入：离线确定性测试喂 fake factory，不触 docker 依赖的惰性 import。
④ git_ops 安全出口：把不可信仓库的 hooks/filters 中和掉

Merge 的 git 跑在 host 上（只有回归与 fix-Coder 才进容器）。集成仓库内容来自 agent 产出，必须视作不可信：可能带 booby-trapped hooks、core.hooksPath 指向陷阱目录、fsmonitor 配成任意命令。run_git 是模块内唯一 git 出口，安全配置集中一处：

_SAFETY_CONFIG = (
  "-c", "core.hooksPath=/dev/null",   # 中和一切 hooks（含 local config 指定）
  "-c", "core.fsmonitor=false",         # fsmonitor 可配成任意命令，禁
  "-c", "protocol.ext.allow=never",     # 禁 ext:: 伪协议（执行任意命令的通道）
  "-c", "user.name=atelier-merge-agent", ...)  # 固定身份 → 提交确定性
# env 再加一道：屏蔽用户/系统 gitconfig
env = {..., "GIT_CONFIG_GLOBAL": os.devnull, "GIT_CONFIG_SYSTEM": os.devnull}
  • 命令行 -c 优先级高于仓库 local config：即便 .git/config 把 hooksPath 指向陷阱目录，这里的覆盖也生效。
  • /dev/null 不是目录 = hooks 目录恒为空，比逐个禁用 hook 名单更彻底（新 hook 类型也免疫）——这是「白名单机制 vs 黑名单枚举」的经典取舍。
  • 确定性：行为只由「仓库内容 + 本模块显式配置」决定，不吸 host 用户配置——Temporal 友好。rev_parse 是解析 ref→sha 的常用捷径。
⑤ provisioning 查表：镜像与依赖决策为什么是纯逻辑？

闭环容器开机要按 TechSpec.tech_stack 选镜像、并预置通用工具。这套「语言/运行时关键词 → 镜像 / 命令」是纯数据查表，可脱离 Docker 单测。resolve_image 故意分三层兜底：

def resolve_image(*, tech_stack=None, files=None, default="python:3.11-slim"):
    if tech_stack: # ① TechSpec 是结构化权威源，最该信
        if (img := select_image(tech_stack)): return img
    if files:       # ② brownfield：detect_stack 特征文件嗅探（惰性 import coder.stack）
        from coder.stack import detect_stack
        if (img := _IMAGE_BY_DETECTED_STACK.get(detect_stack(files))): return img
    return default  # ③ greenfield 空目录 → 退默认 python 镜像
  • greenfield / brownfield 统一：greenfield 走 tech_stack 或退默认；brownfield 才用 detect_stack 嗅探——且它是唯一会碰 coder.* 的路径，故做成运行期惰性 import，不在 import 期反向依赖 coder（Reviewer 走 tech_stack 根本不触此路径）。
  • 不烘焙自定义镜像（#41 取舍）：构建/缓存/维护太重；务实做法是 LiveSandbox.start() 里一次性装通用工具。项目级依赖不在这里——那是 agent 写 manifest 后自己 bulk install。
⑥ 依赖方向：为什么抽到根、禁横向 reach、禁 import legacy？

这是全模块最值得讲的一条结构决策。六个原语能存在于模块根，靠的是三条被 ADR-0012 钉死的铁律：

铁律为什么怎么落地
抽到模块根一个能力被 ≥2 兄弟需要 → 下沉成单向叶子，而非让兄弟横向够兄弟#47/#51/#84 逐个上提 provisioning / sandbox / runner
禁横向 reachReviewer import Coder 会焊死两个独立子系统，改一个牵动另一个共用能力只准来自叶子；子目录间零 import
禁 import legacy实验包反向依赖生产包 = 方向反了，被生产演进牵着走ReAct 内核从 legacy fork 而非 import
命名按能力refactor/v2/new 是活动词，做完就过时叶子名描述「它是什么」（sandbox / provisioning）

代价也诚实记在 ADR-0012 §4.2：内核 fork 引入「两份相似 loop」的一次性重复，A/B 决胜前不收敛。但换来的是——experimental 与 production 物理隔离、闭环世界自包含可独立演化。而所有上提都配一层 re-export shim（coder.agentic_coder / reviewer.independent_test_runner 仍 re-export 这些名字），老调用点与测试的 import 路径零回归——这是「搬家但不通知邻居改地址」的工程温柔。

面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我抽了几个 util」，而是每一次抽取背后都有一句「为什么依赖必须朝这个方向」。

为什么把这些抽到模块根？

单向叶子 + 依赖倒置

一个能力被 ≥2 个兄弟需要就下沉成叶子，四个 Agent 一律向下依赖它、彼此零横向边。这掐死了 reviewer→coder 的耦合（#51）。

Reviewer 独立重跑为什么要干净新容器？

复用 live 容器会被喂绿

Coder 的容器带着它装的全局包 / 脏环境，复用等于自己给自己判过。干净容器复现「换台机器 clone 下来能不能跑」，才逮得住 self-gaming。

抽取后老调用点会不会全炸？

re-export shim 保稳定

原文件位置留一层 re-export，coder.agentic_coder 等仍导出这些名字，既有 import 路径与测试零回归。搬家但不改邻居的地址簿。

为什么禁止 import legacy？

方向：实验不反依赖生产

新实验包依赖旧生产包 = 方向反了，会被生产演进牵着走。ReAct 内核宁可从 legacy fork 一份（吃短期重复），也不反向 import（ADR-0012）。

git 跑 host 为什么还要中和 hooks？

不可信仓库 + 白名单更彻底

集成仓库来自 agent 产出，可能埋 booby-trapped hooks。core.hooksPath=/dev/null「目录恒为空」比逐个禁 hook 名单彻底，新 hook 类型也免疫。

纯逻辑为什么单独成叶子？

无 IO / 无重依赖可单测

provisioning 查表、paginate_text 全是无副作用纯函数，可脱离 Docker 单测；docker/LLM 等重依赖靠惰性 import，离线测试根本不触发。

已知限制 · 刻意推后

哪些没做，为什么可以先不做

限制影响当前缓解
镜像/工具硬编码字典只认 python / node，未知运行时退默认或返回 Nonetech_stack 显式给 image / test_command 覆盖
独立重跑跑整套测试brownfield 大套件慢，无选测/分层运行时优化，不进门语义，推后
ReAct 内核 fork 出两份相似 looplegacy 与闭环并存的短期重复ADR-0012 §4.2 明记，A/B 决胜前不收敛

已交付：git_ops 上提模块根（#90，与 sandbox / provisioning 并列的叶子，merge/git_ops.py 退为 re-export shim）。更远的未来：provision 查表数据化（配置驱动而非硬编码字典）· 回归门选测分层（brownfield 大套件按 file footprint 裁剪）· 内核收敛（闭环赢 A/B 后 legacy 退役、两份 loop 归一）。

一句话记住这个模块共享基座不是「一堆 util」，而是一个被刻意约束了依赖方向的结构：一个能力被 ≥2 兄弟需要就下沉成单向叶子，四人向下够、彼此不横向勾连，re-export shim 让搬家零回归。这是把「依赖倒置 + 命名按能力」两条原则写进目录结构的一次实践（ADR-0012）。
