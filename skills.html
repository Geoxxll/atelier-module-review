Agent Skills 系统

给每个 Agent 装一本「出厂方法论手册」——启动时读一次，always-on 注入进 system prompt。

为什么每个 Agent 需要一层 Skill？

Agent 的 SYSTEM_PROMPT（底座）已经写了「你是 PM，PRD 要可测试、有边界、发现歧义就提问」这类原则。但一个刚知道原则的模型，未必真的会写出好东西——它缺的是「照着做」的具体材料：一份好/坏对照的范例、一套「每条用户故事都逐一自问空态/失败/边界/并发/权限」的操作清单。

这正是 Skill 层的定位：把 base 缺的、可操作的领域知识，以一份 SKILL.md 的形式打包，启动时注入 system prompt。它跟人类「入职时发的方法论手册」是一回事——常驻、随时可查、对所有任务生效。

  • 出厂静态——skill 是 authored、随代码发布的固定方法论，区别于从用户交互里学出来的 SOP（那是「口味」，见记忆系统复盘）。
  • always-on——该角色的所有 skill 全量注入，不像 Claude Code 那样按 description 匹配、按需 progressive-disclosure 调用。
  • 分层而非合并——底座 / Skill / SOP 三层顺序拼接、各自成节，冲突时靠「排在后面 + 显式标注优先级」让 LLM 遵循，而不引入 LLM 融合。
贯穿全篇的一课 · 非冗余原则 Skill 的价值不在「换个说法重申 system prompt」，而在携带 base 缺的知识——领域细节、worked example、可操作脚手架。第一版 prd-writing 只复述原则，盲评反而总分 −1.0；重写成 few-shot + 边界枚举脚手架后 pairwise 4/0 全胜。「深挖 ②」有完整证据链。

三份 Skill 各给底座补了什么

三个 Agent 各带一份 skill，物理落点是 services/<svc>/skills/<name>/SKILL.md（per-service 文件系统）。每份都遵循同一条铁律：开篇先声明「本能力包不重复 base 已有的原则」，只补两样 base 缺的东西——好/坏对照 few-shot + 强制枚举的脚手架。

图版 II · 动态流程

启动时加载一次，每个任务重新组装一次

Skill 生命周期分两拍：worker 启动时 load + compose 出「底座+Skill」半成品（II-a）；每次干活再叠上 per-task 的 SOP（II-a 末端）。II-b 讲它和 SOP 为什么必须正交。

图版 II-a · 加载 → 组装 → 发送（以 PM 写 PRD 为例）DYNAMIC · load & compose
1
worker 启动 · __init__

load_skills(settings.skills_dir) 扫目录，逐个读 <name>/SKILL.md，剥 YAML frontmatter 取 name/description、其余为正文，按 name 排序返回 list[Skill]。

优雅降级：目录缺失 / 单个 skill 损坏只丢它一个、Agent 照常启动（同「无 GitHub token → Coder 跳过 PR」哲学）。

2
按阶段正交切分 skill

PM 把 skill 按阶段分流：refine skill 只进打磨 prompt，其余（prd-writing）只进写 PRD prompt——避免把打磨方法论塞进 PRD 上下文（反之亦然）。

3
compose_prompt(base, skills) · 启动组一次

拼出「底座 + Skill」半成品并缓存：self._write_prd_system_prompt = compose_prompt(base=SYSTEM_PROMPT, skills=skills)。SOP 依赖租户，此刻先不拼。

4
任务到达 · _run_write_prd（per-task）

按 task.tenant_id 查 SOP，重新 compose_prompt(base=SYSTEM_PROMPT, skills=self._skills, sop=sop_text) 叠上第三层。无 SOP 时直接复用启动时的半成品。

Skill 层每次都在，是常量；只有 SOP 层随租户变——所以 Skill 启动组一次、SOP per-task 组。

5
组 messages 发 LLM

System(底座 + Skill + SOP) + Human(需求) → 结构化 PrdOutput。skill 的 few-shot / 脚手架就在 system prompt 里持续生效。

两拍的关键：Skill 是启动加载的静态层（读一次进内存、不随任务变），SOP 是运行时的动态层（每任务查库）。把「启动组一次」和「per-task 组」分开，既省掉每次重读文件，又让依赖租户的 SOP 能正确叠加。

Skill ⟂ SOP：为什么必须正交

Skill 和 SOP 都注入 system prompt、都想影响「怎么写」，很容易踩重叠。项目定了一条硬边界：skill 管方法论、SOP 管口味——重叠就是「非冗余原则」的另一种翻车。

图版 II-b · Skill 与 SOP 的正交分工DYNAMIC · orthogonality
维度Skill（出厂静态方法论）SOP（学出来的口味）
来源authored，随代码发布从用户 PRD 修订轨迹蒸馏
管什么方法论（强制边界枚举 / AC 可测）口味（用 GWT 还是 checklist）
粒度全局共享，对所有租户一样per-租户，动态进化
加载启动读一次进内存per-task 查库
compose 槽位Skill 层（中）SOP 层（后，优先级更高）
重叠后果= #7 冗余陷阱：两个能力单元抢同一件事，内耗、不可归因

具体到 prd-writing：skill 管「每条 story 都要枚举边界」（方法论），SOP 管「这个用户的验收标准爱用 checklist」（口味）。同一个「AC 怎么写」的旋钮，skill 只碰「要覆盖全」、SOP 只碰「用什么格式」——刻意不重叠。

为什么正交是硬约束：每加一个能力单元都要能被单独 A/B 证明有用。skill 和 SOP 若抢同一件事，就无法归因是谁在起作用——这跟「三类记忆刻意不合并」是同一条设计美学。

端到端 · 走一遍

同一份 prd-writing，冗余版让它变差、非冗余版让它 4/0 全胜

这是本模块最值钱的一条实证故事。跟着走一遍，你就懂了「skill 的价值到底在哪」，以及「为什么必须用 eval 来判」。

场景 · 给 PM 装 prd-writing skill

从「复述原则反而 −1.0」到「重写脚手架 pairwise 4/0」

起点

PM 的 base SYSTEM_PROMPT 早已写清 PRD 要「可测试、有边界、发现歧义就提问」。现在想再加一份 skill 让它更好。

# base 底座（节选）——原则已在此
你是资深 PM。PRD 的验收标准要可测试、覆盖边界，
发现需求歧义时主动提问，不要臆断。
第一版 · 冗余

skill v1 内容是「验收标准要可测试 / 要有边界 / 发现歧义就提问」——把 base 的原则又说了一遍。盲评 LLM judge（n=3，秒杀硬需求）：

# compose_prompt(base, skills=[prd_writing_v1])
# 已装配能力包（Skill）：prd-writing
> 验收标准要可测试、要有边界、发现歧义要提问   # ← base 已有

# 绝对打分盲评结果
总分:        base+skill  −1.0  # 装了反而更差！
edge_coverage: 4.0 → 3.33       # 最该提升的维度还掉了

根因：对一个已知道该方法论的模型，复述原则的信息增量 ≈ 0，反而多上千字符稀释注意力。

重写 · 非冗余

把 skill 重写成 base 没有的东西：好/坏 AC 对照 few-shot + 强制边界枚举脚手架（每条 story 自问空态/失败/边界/并发/权限）。

# SKILL.md 开篇就声明非冗余
本能力包不重复这些原则，只给你 base 没有的两样：
❌ 坏：「系统响应要快」
✅ 好：「点击下单后 500ms 内返回结果或排队提示」
# + 脚手架：对每条 story 逐一自问
空态 / 失败态 / 边界值 / 并发 / 权限 → 命中即写成可测标准
重跑 · 判决

换 pairwise 盲评 + 位置 counterbalance（同时给两份判哪个更好，一半样本调换 A/B 前后消除位置偏好）——差异立刻清晰：

# pairwise：非冗余版 vs 冗余版，多维分别裁决
整体质量:   非冗余 胜
边界覆盖:   非冗余 胜
可测性:     非冗余 胜
———————————— 4 / 0 全胜

结论：skill 的价值不在「换个说法重申 system prompt」，而在携带 base 缺的知识——领域细节、worked example、可操作脚手架。这不是 skill 系统的失败，正是 eval 把「这份内容冗余」抓了出来——系统的价值就是「能测、能换」。

核心机制 · 钻到实现级

四个撑起整个模块的机制

展开看代码级细节。这几个点也是面试里最容易被追问「你具体怎么实现的 / 为什么这么定」的地方。

① 分层 compose：为什么拼接而不 merge？优先级怎么定？

compose_prompt(base, skills, sop) 不做任何智能融合——就是按顺序把三层拼成一段字符串，每层自成一节、带显式的优先级措辞，让 LLM 自己遵循：

parts = [base.rstrip()]
for skill in skills or []:
    parts.append(f"\n\n# 已装配能力包（Skill）：{skill.name}\n"
                 f"> {skill.description}\n"
                 f"> 与上文底座冲突时以本节为准。\n\n{skill.body}")
if sop:
    parts.append("\n\n# 用户偏好（SOP，长期默认）\n"
                 "> 与上文 Skill / 底座冲突时以本节为准；\n"
                 "> 但用户在【本次对话】中另有明示，一律以本次为准（实时指令 > SOP）。\n\n" + sop)
return "".join(parts)
  • 为什么不 merge：LLM 融合两段方法论既不确定、又不可控。拼接 + 显式优先级措辞是确定性的——冲突按固定顺序解决，而非 LLM 临场仲裁（呼应项目「集中确定性规则而非 LLM 仲裁」的一贯选择）。
  • 完整优先级链：实时指令 > SOP > Skill > 底座。越靠后的层、优先级越高；但当次对话里用户明说的话（实时指令）永远压过一切历史层——SOP 是「先验默认」而非「笼子」。
  • frontmatter 极简解析：只认 key: value 行取 name/description，刻意不引 pyyaml——字段简单，不给共享包加传递依赖。
② 非冗余原则：skill 必须携带 base 缺的东西（被 eval 实证）

全模块最贵的一课。第一版 prd-writing 复述 base 已有的原则，绝对打分盲评总分 −1.0、最该提升的 edge_coverage 反而 4.0→3.33。重写成 few-shot + 边界枚举脚手架后，pairwise 4/0 全胜。

对照冗余版（v1）非冗余版（v2）
内容复述「可测/有边界/发现歧义就问」好/坏 AC 对照 + 边界枚举脚手架
信息增量≈ 0（base 已有）高（base 缺的 worked example）
副作用上千字符稀释注意力给出可照做的操作清单
eval 结果总分 −1.0，edge 掉分pairwise 4/0

方法论上还有一课（#8）：同一份改动，绝对打分（每维 1–5）分数全挤在 4/5、测不出差异；换 pairwise + 位置 counterbalance（同时给两份判优劣、一半样本调换前后消除位置偏好）后差异立刻清晰。绝对打分有量程压缩，pairwise 做相对比较、对细微优劣远更敏感——这是 eval 驱动开发的基本功。

诚实版结论第一版翻车不是「skill 系统的失败」，恰恰是 eval 把「这一份 skill 内容冗余」抓了出来——系统的价值正是「能测、能换」。
③ always-on vs progressive-disclosure：为什么全量注入？

Claude Code 的 skill 是按需调用——只有一行 description 常驻上下文，模型判断相关才「打开」读正文（progressive-disclosure）。Atelier 的 skill 反过来：该角色所有 skill 启动时全量注入 system prompt、always-on 生效。

Atelier（本模块）Claude Code 风格
激活always-on 全量注入按 description 匹配、按需打开
常驻上下文整份正文仅一行 description
数量假设每角色少量（当前 1 份）可挂几十上百份
选择器无（角色即选择）需要检索/匹配器

为什么这样够用：当前每个角色只有一份高度相关的方法论 skill，没有「skill 太多、上下文装不下」的问题，也就不需要按需检索的复杂度（YAGNI）。角色本身就是天然的选择器——PM 的任务永远该用 prd-writing。skills.py 文件头也写明：未来「用户提交 skill」会改变加载时机 / 来源 / 隔离，但都是在此之上增量，不推翻本模块。

④ Skill ⟂ SOP 正交：两个都改「怎么写」，边界画在哪？

Skill 和 SOP 都注入 system prompt、都影响「怎么写 PRD」，边界不清就会重叠内耗。硬边界：skill 管方法论、SOP 管口味。

  • skill（方法论）：「每条 story 都要枚举空态/失败/边界/并发/权限」「AC 必须可测量化」——做得全不全。
  • SOP（口味）：「这个用户的 AC 爱用 checklist 而非 GWT」——用什么风格。
  • 同一个「AC 怎么写」旋钮，skill 只碰「覆盖度」、SOP 只碰「格式」，刻意不重叠。SOP 的槽位（ac_format / story_template / … 6 枚举）在设计时就被要求「与已有 skill 正交」。
重叠 = #7 陷阱若 skill 也去规定「用 checklist」、SOP 也去教「要枚举边界」，两个能力单元抢同一件事——不仅内耗，更致命的是无法归因：A/B 时分不清是谁在起作用。这跟「三类记忆刻意不合并、各自可单独 A/B」是同一条设计美学：能力单元之间正交，才能各自被证明有用。
面试可讲 · 取舍点

这个模块能讲出什么「设计品味」

不是「我给 Agent 加了 skill」，而是每个决策背后都有一句「为什么必须这样」。

给 Agent 加了 skill 反而变差？

非冗余是硬门槛

对已知方法论的模型，复述原则信息增量 ≈ 0，还稀释注意力。skill 只该带 base 缺的：few-shot + 可操作脚手架。冗余版 −1.0，非冗余版 pairwise 4/0。

怎么判断一份 skill 值不值得留？

eval 说了算

绝对打分量程压缩、测不出差异；pairwise + counterbalance 对细微优劣敏感得多。skill 系统的价值正是「能测、能换」——翻车是 eval 的功劳，不是系统的失败。

为什么拼接而不让 LLM 融合各层？

确定性优先

三层顺序拼接 + 显式优先级措辞，冲突按固定顺序解决。LLM 融合既不确定又不可控——集中确定性规则，而非临场仲裁。

Skill 和 SOP 都改「怎么写」，不打架吗？

方法论 ⟂ 口味

skill 管「覆盖全不全」、SOP 管「什么格式」，同旋钮不同面。重叠即 #7 冗余陷阱、无法归因。正交才能各自单独 A/B。

为什么 always-on 而不像 Claude Code 按需调用？

YAGNI

每角色当前就一份高相关 skill，没有「装不下」问题，角色本身即选择器。不需要检索匹配的复杂度——未来用户提交 skill 再增量。

一份 skill 损坏会不会拖垮启动？

优雅降级

目录缺失 / 单份解析失败只丢它一个，Agent 照常无-skill 启动。同「无 GitHub token → Coder 跳过 PR」一脉——增强项从不是主链路单点。

已知限制 · 刻意推后

哪些没做，为什么可以先不做

限制影响当前缓解
只做 prompt 注入型 skill不能带「工具/可执行动作」型能力v1 明确范围；工具型是另一轴
always-on 全量注入skill 一多会撑爆上下文当前每角色 1 份，YAGNI
skill 出厂静态用户不能自带 skill加载时机/来源/隔离留作增量
Coder 无 skill方法论不走 prompt 注入走自研 ReAct + MCP，另一范式
Architect / Reviewer skill 未单独 A/B非冗余靠同模具类推只 PM 侧跑了 pairwise 实证

更远的未来：用户提交 skill（改加载时机 / 来源 / 隔离，在本模块之上增量）· 工具型 skill（携带可执行动作而非纯 prompt）· 按需激活（当 skill 多到装不下再引入 progressive-disclosure 与选择器）。三条都遵循同一原则：不推翻 v1、只在其上增量。
