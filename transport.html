传输层

Workflow 怎么把一个要跑几分钟的任务交给 Agent，又不让编排线程干等？——持久化队列投递 + 异步回执。

一句话：Activity 把任务塞进 JetStream 持久队列并附上自己的 task_token，然后立刻 raise_complete_async() 释放 worker、不在原地等；Agent worker 跑完用 handle.complete() 把结果回填进 Temporal Event History。重投与重试靠 uuid5 幂等兜底，绝不重复烧 LLM。

一张投递单，一个取餐号

FeatureWorkflow 里的一步「让 Coder 写代码」，背后是一次 LLM + Docker 沙箱，可能跑好几分钟。最朴素的做法是：编排器发个请求，然后占着一条线程干等对面回包。这有两个致命问题——等的这几分钟里 worker 线程被白白占死；万一 Agent 收到消息、烧完了昂贵的 LLM、正要回包时崩了，消息没有重投，整个 Agent 得从头再跑一遍。

Atelier 的传输层把它拆成「投递单 + 取餐号」两段，就像在一家忙碌的餐厅点餐：

  • 下单（请求腿）——Activity 把任务写进一张持久化的订单队列（JetStream），订单上附一个取餐号（task_token）。厨房（Agent worker）没空？订单在队列里排着，不丢。
  • 离开柜台（不干等）——下完单 Activity 立刻 raise_complete_async()，告诉 Temporal「结果稍后由外部送达」，然后释放线程走人。它不站在柜台前等。
  • 叫号取餐（完成腿）——厨房做好后，凭取餐号直接把结果送回 Temporal 的 Event History（handle.complete）——有 ack、持久化，不靠「发个包祈祷对面收到」。

这套「投递 + 异步回执」正是 Temporal 官方推荐的长任务范式（async activity completion）。它把「谁在跑、跑多久」和「编排器要不要占资源等」彻底解耦——这就是传输层比「一个 NATS 请求-回复」值钱的地方。

贯穿全篇的一句话 · 持久化 + 幂等 可靠不是靠「不出错」，而是靠「出错也能安全重来」：JetStream 保证「无 worker 不丢、崩溃就重投」，uuid5 幂等保证「重投/重试都不会重复烧 LLM」。两者配合，缺一不可——「深挖」里有完整推演。
范式说明 · 示例是 legacy 链路，原语是活基础设施 本页的传输原语（JetStream 异步完成 · uuid5 幂等 · Blackboard 指针瘦身）是范式无关的真实活基础设施。但文中示例用的 FeatureWorkflow + round-0 Coder 派发是 legacy open-loop 链路；当前主链路的编码已走闭环管线（⑤ Temporal 编排的 Orchestrator），其中 Blackboard 尤其在退役路上（闭环用 git 分支传产物）。读时把示例当「投递机制怎么工作」，别当「Atelier 现在这么编码」。

怎么读：右侧红箭头是请求腿（任务向下：Activity → JetStream → worker）；左侧绿箭头是完成腿（结果向上：worker → Event History）。注意二者物理不同管道——请求走 JetStream 持久队列，完成走 Temporal gRPC，事件另走 Core NATS。大产物完全不上总线，直接落 Blackboard，总线上只跑指针与瘦身子集。

一眼看穿的切分：左表——请求腿走 JetStream（要持久、要重投），事件腿走 Core NATS（尽力投递，前端断线重连即可）。右表——大产物（代码文件 500KB+）绝不上总线，落 Blackboard；总线消息只带指针 (round, agent_role)。为什么不用 Redis 存 Blackboard？workflow 可能等人工审批跑很久，Redis TTL 会驱逐，Postgres 天然持久还能事务一致。

图版 II · 动态流程

一次任务的完整往返：投递 → 消费 → 回填

请求腿把任务持久化后 Activity 就走人；worker 拉起、幂等把关、干活、写 Blackboard、回填 Event History、再 ack。右边单列出「重试 / 重投」这条最能体现设计精髓的岔路。

图版 II-a · 正常往返（首次处理）DYNAMIC · happy path
1
workflow-engine · Activity 派发

payload 里塞入本次尝试的 task_token（base64）→ js.publish(agent.coder.task)。js.publish 返回 PubAck 才算持久化成功，失败抛异常由 Temporal retry_policy 兜底。

2
Activity · 挂起不等

raise_complete_async()：告诉 Temporal「结果稍后由外部交付」，Activity 立刻返回、释放 worker 线程——不在这儿站着等 LLM 跑完。

这一步是整个范式的核心：编排线程从「干等 5 分钟」变成「投完就走」。

3
JetStream · 持久排队

AGENT_TASKS（WORK_QUEUE retention）持久保存任务。没有 coder worker 在线？任务排队等着，不会超时丢失。

4
coder worker · 拉取 + 幂等把关

psub.fetch(1) 拉到消息 → 先查 idem.get(task_id)。首次处理未命中，继续；命中则直接用缓存结果完成（见 II-b）。

5
handler · 干活 + 写大产物

跑 LLM / Docker 沙箱（几分钟）。代码文件、沙箱结果等大产物写 Blackboard，返回值只留瘦身摘要 / 指针。

attribution_scope 包住 handler，让嵌套 ReAct 子调用的 LLM 流量都 publish 到 workflow.{wf}.llm。

6
完成腿 · 回填 + ack

idem.put(task_id, result) 先落幂等表 → handle.complete(result) 凭 task_token 把结果写进 Event History → 最后才 msg.ack()。SSE 事件也在此刻首发。

顺序关键：先完成 activity 再 ack。ack 前崩溃 → JetStream 重投到另一 worker，靠幂等避免重复烧 LLM。

为什么 ack 放最后：WORK_QUEUE 里消息一旦 ack 就删除。若先 ack 再 complete，complete 途中崩溃就两头落空——任务从队列没了、结果也没交付。「先 complete、后 ack」保证任何崩溃点都能安全重来。

图版 II-b · 岔路：重试 / 重投都走幂等短路DYNAMIC · retry & redelivery

Temporal 超时重试（新 token）

A
activity 超时

Temporal 认为 activity 超时，按 retry_policy 重跑 Activity——生成一个全新的 task_token，publish 一条新消息。

B
worker 幂等命中

task_id 是 uuid5 确定性——两次尝试同一个 task_id。idem.get 命中缓存结果，不重跑 LLM，直接用新 token complete。

这正是「不开 publish 去重」的原因：去重会把新 token 消息误删，导致新 token 永远送不达 → 死锁。

C
旧 token 无害失效

旧 token 若也来完成，会 RPCStatusCode.NOT_FOUND——被 _complete_activity 吞掉记 warning，无害。

JetStream 重投（ack 前崩溃）

A
worker 崩溃

worker 跑完、complete 了，但 ack 之前进程崩了。JetStream 没收到 ack → ack 超时后重投给另一个 worker。

B
第二 worker 幂等命中

同 task_id，idem.get 命中 → 用缓存结果再 complete 一次 → ack。LLM 只烧过一次。

at-least-once：不仅业务要幂等，complete 回调也要幂等——重复 complete 已完成的 activity 会 NOT_FOUND，同样被吞掉。

C
handler 抛异常

handler 报错则 _fail_activity 标记失败（Temporal 决定是否重试）并 ack 消息，避免无限重投——重试改由 Temporal 重新 publish 驱动。

两条岔路一个答案：无论是 Temporal 层重试（新 token 新消息）还是 JetStream 层重投（同消息再送），最终都撞在同一道 uuid5 幂等闸门上短路。持久化负责「不丢」，幂等负责「不重复烧钱」——两者正交、各司其职。

端到端 · 走一遍

跟一个 Coder 任务，从派发到回填再到一次重投命中

用最真实的 round-0 Coder 派发把 task_id / payload / Blackboard 指针 / 完成 / 幂等命中串成一个连续故事。看完你就懂了整条投递→消费→回填→重来链路。

场景 · legacy FeatureWorkflow Step 3 首次派发 Coder

workflow 让 Coder 写代码（round=0） · legacy 示例，演示投递机制

派发 · 编排层

workflow 用 uuid5(workflow_id, "coder-round-0") 算出确定性 task_id，调 run_coder_agent Activity。Activity 组瘦身 payload、塞 token、publish。

# workflows.py：确定性 task_id（replay 时不变，不能用 uuid4/now）
coder_task_id = str(uuid.uuid5(UUID(workflow_id), "coder-round-0"))

# activities.py：payload 只带 ~3KB 子集 + token，不带大文件
payload = {
  "task_id": "a3f1…-coder-0", "workflow_id": "wf-7c2e…",
  "kind": "write_code", "round": 0,
  "prd": {acceptance_criteria, out_of_scope},      # _build_prd_subset ~1KB
  "tech_spec": {api_endpoints, data_models, nfr},  # _build_tech_spec_subset ~2KB
  "_activity_token": "CicKB2RlZmF1bHQ…",           # base64(task_token)
}
await js.publish("agent.coder.task", json.dumps(payload))
activity.raise_complete_async()  # 挂起，Activity 走人
消费 · 执行层

coder worker 从 coder-workers durable consumer 拉到消息，幂等未命中，跑 ReAct + 沙箱，把代码文件写进 Blackboard。

# _handle_msg：先查幂等（首次未命中）
cached = await idem.get("a3f1…-coder-0")   # → None，继续

# handler 干活，大产物落 Blackboard（不上总线）
await blackboard.write("wf-7c2e…", round=0,
    agent_role="coder", artifact="files",
    content={files:[…12 个文件, 340KB…]})   # 键空间 4 元组

reply.result = RunCoderAgentResult(pr_url=…, summary=…)  # 瘦身结果
回填 · 完成腿

先落幂等表，再凭 token 把瘦身结果回填 Event History，最后 ack。前端此刻收到一条 SSE 事件。

await idem.put("a3f1…-coder-0", result_payload)   # ① 先幂等
handle = temporal.get_async_activity_handle(task_token=token)
await handle.complete(result_payload)                  # ② 回填 Event History
await nc.publish("workflow.wf-7c2e….event", evt)   # ③ SSE（仅首次）
await msg.ack()                                       # ④ 最后才 ack
岔路 · 重投命中

假设第 ③ 步后、④ 之前 worker 崩了。JetStream 没等到 ack，重投给另一个 coder worker。这次幂等命中。

# 第二个 worker 重投处理同一 task_id
cached = await idem.get("a3f1…-coder-0")   # → 命中！返回缓存 result
log.info("worker.idempotent_hit")
await _complete_activity(temporal, token, cached)  # 用缓存直接完成
# complete 可能 RPCStatusCode.NOT_FOUND（已完成）→ 吞掉记 warning，无害
await msg.ack()   # LLM 全程只烧过一次 ✅

这就是传输层的抗故障核心：崩在任何一步，重投/重试都能安全收敛，且绝不重复烧 LLM。

核心机制 · 钻到实现级

六个撑起传输层的机制

展开看代码级细节。这几个点也是面试里最容易被追问「你具体怎么保证不丢、不重复」的地方。

① raise_complete_async + handle.complete：为什么不在 Activity 里等？

传统 Activity 是「调用 → 阻塞等返回」。但 Agent 任务要跑几分钟 LLM，若 Activity 原地等，就会占死一条 worker 线程好几分钟，还得把超时设很长。Temporal 的 async activity completion 把它拆成两半：

# 派发端（Activity 内）：投完就挂起，不阻塞
payload[ACTIVITY_TOKEN_KEY] = base64.b64encode(activity.info().task_token).decode()
await js.publish(subject, json.dumps(payload).encode())
activity.raise_complete_async()   # 抛特殊信号：结果稍后由外部交付

# 完成端（worker 内，可能几分钟后、在另一进程）：
handle = temporal.get_async_activity_handle(task_token=token)
await handle.complete(result)      # 结果写进 Event History，有 ack、持久化
  • task_token 是钥匙：它唯一标识「这一次 Activity 尝试」，base64 塞进消息随任务旅行，worker 凭它找回对应的挂起 activity。
  • 响应腿从此可靠：旧方案 Core NATS msg.respond() 回包丢了结果就丢；现在结果直接进 Event History，有确认、可持久化、replay 可见。
  • 代价：Agent worker 现在得持一个 Temporal client，打破了「Agent 只认 NATS」的干净分层——缓解办法是把这套传输细节全封在 run_agent_worker，各 Agent 只写薄 handler（ADR-0011 §4.2）。
② uuid5 幂等：agent_task_results 短路，重投/重试不重复烧 LLM

JetStream 是 at-least-once：崩溃重投、Temporal 重试，同一个逻辑任务可能被处理多次。LLM 调用又贵又慢，绝不能重复烧。幂等层就一件事——处理前按 task_id 查缓存，命中即短路：

cached = await idem.get(task_id) if task_id else None
if cached is not None:
    log.info("worker.idempotent_hit")
    await _complete_activity(temporal, token, cached)  # 用缓存完成，不跑 handler
    await msg.ack()
    return
# 未命中才跑 handler（LLM）；成功后先落幂等、再完成、再 ack
if reply.success:
    await idem.put(task_id, result_payload)   # INSERT ... ON CONFLICT DO NOTHING
    await _complete_activity(temporal, token, result_payload)
  • 为什么 uuid5 而非 uuid4：workflow 里 uuid5(workflow_id, "coder-round-0") 保证同一逻辑任务的每次尝试都算出同一个 task_id——幂等键才稳定。用 uuid4 每次尝试 id 都变，幂等直接失效（也违反 workflow 确定性，见深挖⑤）。
  • complete 也要幂等：重投时对已完成的 activity 再 complete 会 RPCStatusCode.NOT_FOUND，_complete_activity 专门吞掉这个错记 warning（e2e-common-errors #5：「不仅业务要幂等，完成回调也要幂等」）。
  • agent_task_results（0005_idempotency.sql）：task_id 主键，result jsonb。put 用 ON CONFLICT DO NOTHING，并发双写也安全。
③ 为什么刻意不开 JetStream publish 去重？——一个会死锁的陷阱

JetStream 自带 publish 去重：给消息带 Nats-Msg-Id，重复 id 在窗口期内会被自动丢弃。看起来正好能用 task_id 去重——但用了会死锁。关键在于 token 与 task_id 的生命周期不同：

task_id（幂等键）task_token（完成钥匙）
生命周期整个逻辑任务不变（uuid5）每次 Activity 尝试都换新
作用判「跑没跑过」回填「这次尝试」
死锁推演若用 Nats-Msg-Id = task_id 开去重：Temporal 超时重试产生新 token 的新消息，但它的 task_id 和上次一样 → 被 JetStream 当重复丢弃 → 新 token 永远到不了 worker → 挂起的 activity 永远等不到完成 → 死锁。

正确分工：JetStream 只负责「无 worker 不丢 + 消费崩溃重投」；去重交给 worker 的 task_id 幂等层。每次尝试都 publish 一条带新 token 的新消息全部送达 worker——旧 token 完成失败但无害（NOT_FOUND），新 token 完成成功。这是「持久化」与「幂等」职责正交的经典范例。

④ 消息瘦身 + Blackboard 指针：大产物为什么不上总线？

代码文件可能 500KB+，多个文件轻松超过 NATS 默认 max_payload=1MB；Temporal Event History 也有 gRPC payload 限制（默认 4MB）。所以传输层立了条铁律：总线只跑「瘦身子集 + 指针」，大产物落 Blackboard。

# 派发 Coder：只带 PRD/TechSpec 子集（~3KB），不带任何代码文件
"prd": _build_prd_subset(prd),         # acceptance_criteria + out_of_scope ~1KB
"tech_spec": _build_tech_spec_subset(ts) # api/data_models/nfr ~2KB

# fix_code 更极端：只传 Blackboard 指针，代码从黑板读
"round": input.round,
"prev_coder_round": …, "prev_reviewer_round": …   # 指针，非内容
  • 键空间 (workflow_id, round, agent_role, artifact)：round=0 初次 / 1..3 修复；artifact ∈ files|sandbox|review。四元组唯一定位一份产物。
  • write 幂等：ON CONFLICT DO UPDATE——Activity 超时重试重复写同一键，覆盖而非报错或堆重复。
  • read 陷阱：asyncpg 读 jsonb 不自动反序列化，返回的是字符串，直接 dict() 会按字符遍历报错（e2e-common-errors #1）——故 read 里 json.loads(raw) if isinstance(raw, str) else dict(raw)。
  • 为何 Postgres 不用 Redis：workflow 可能等人工审批跑很久，Redis TTL 会驱逐；Postgres 持久 + 与 workflows 同库可事务一致。
⑤ 确定性：workflow 里为什么只能 uuid5，不能 uuid4 / now / random？

Temporal workflow 靠 Event History replay 恢复状态：worker 崩了或代码更新后，会把历史事件重新「放一遍」来重建当前状态。这要求 workflow 代码纯确定性——同样的历史必须算出同样的结果。uuid4() / now() / random 每次 replay 值都变，会直接触发 nondeterminism error。

# ✓ 确定性：salt 派生，replay 时恒等
coder_task_id   = uuid.uuid5(UUID(workflow_id), "coder-round-0")
reviewer_task_id= uuid.uuid5(UUID(workflow_id), f"reviewer-round-{review_round}")
coder_fix_id    = uuid.uuid5(UUID(workflow_id), f"coder-fix-round-{fix_round}")

# ✗ 禁止：uuid4() / datetime.now() / random —— replay 值会变

这与幂等层互为因果：确定性保证「重试拿到同一个 task_id」，task_id 稳定，幂等短路才成立。两个约束串在一起，才让「重试不重复烧 LLM」这句话真正落地。改 workflow 逻辑（增删 Activity / 改顺序）也是确定性雷区——旧 workflow 被新代码 replay 会报 [TMPRL1100] Nondeterminism（e2e-common-errors #4），须先排空旧 workflow 或用 workflow.patched() 版本化。

⑥ pydantic_data_converter：三端统一的跨服务契约

worker 用 handle.complete(result) 交付结果，于是结果类型（Run*Result）成了 worker ↔ workflow 的跨服务契约——它得能在两个不同进程间可靠序列化/反序列化。三端的 Temporal client 必须统一数据转换器：

# API client / workflow-engine worker / 各 agent worker —— 三端一致
temporal = await TemporalClient.connect(
    host, namespace=ns,
    data_converter=pydantic_data_converter,   # temporalio.contrib.pydantic
)
  • 为什么必须三端一致：API 端起 workflow、workflow-engine 跑编排、agent 端回填结果——任一端转换器不同，Pydantic 模型就会序列化失配。
  • 契约上升为共享库：Run*Result 全收进 atelier_shared.contracts（ADR-0010），workflow-engine 的 types.py 全量 Pydantic 化，避免五个服务各自重复定义漂移。
  • 多态 Result 陷阱：RunCoderAgentResult 有 pr_url、RunCoderFixAgentResult 没有——workflow 共用一行 result.pr_url 会 AttributeError（e2e-common-errors #3），须 getattr(..., None) 或统一基类。
面试可讲 · 取舍点

这个模块能讲出什么「工程品味」

不是「我用了 NATS + Temporal」，而是每个决策背后都有一句「为什么必须这样、不这样会怎样」。

为什么不在 Activity 里同步等结果？

解耦资源与时长

Agent 任务要跑几分钟，同步等会占死 worker 线程。async completion 让 Activity 投完就走，结果稍后凭 task_token 回填 Event History——编排资源不再为长任务买单。

为什么不开 JetStream publish 去重？

会死锁

token 每次尝试都换新，去重会把重试的新 token 消息误删，新 token 永远送不达 → 挂起 activity 死等。去重必须交给 worker 的 task_id 幂等层，不能交给消息总线。

at-least-once 怎么不重复烧 LLM？

uuid5 幂等短路

确定性 task_id 让重试/重投拿到同一幂等键，处理前查 agent_task_results 命中即用缓存完成。持久化管「不丢」，幂等管「不重复」，两者正交。

大产物为什么不走 NATS？

总线只跑指针

代码 500KB+ 超 NATS 1MB / Event History 4MB 限制。大产物落 Blackboard，消息只带 ~3KB 子集 + 四元组指针。消息总线保持轻、快、不撑爆。

complete 和 ack 谁先谁后？

先 complete 后 ack

WORK_QUEUE 里 ack 即删消息。先 ack 后 complete 一旦中途崩就两头落空。先完成 activity、成功了再 ack，保证任何崩溃点都能安全重投。

这条链路怎么测？

坦白单测测不到

传输链路 mock 了 NATS，单测覆盖不到（ADR-0011 §4.2）。靠起整套栈做集成验证；Agent 业务逻辑单测仍保持绿。诚实标注盲区本身就是加分项。

已知限制 · 刻意推后

哪些没做，为什么可以先不做

限制影响当前缓解
Agent worker 持 Temporal client打破「Agent 只认 NATS」的干净分层细节全封在 run_agent_worker，Agent 只写薄 handler
传输链路单测覆盖不到只能靠集成栈验证正确性业务逻辑单测保持绿；e2e 报告兜底
每次派发 nats.connect + drain高频派发有连接建立开销量级可接受，未来可池化连接
幂等表无 TTL / 清理agent_task_results 长期只增体量小；待加保留期或按 workflow 归档

更远的未来：把「结果」也改成 JetStream 结果 stream + 编排侧统一消费完成（ADR-0011 §5），进一步解耦 Agent 与 Temporal——但复杂度更高，当前 async completion 已够用。另一条并行演进是闭环 orchestrator（②，reducer 纯核 + asyncio shell）逐步替代这套 Temporal 编排，届时传输语义会随之迁移。

一句话总纲传输层的全部可靠性都压在一句话上：「JetStream 保证不丢，uuid5 幂等保证不重复，先 complete 后 ack 保证任何崩溃点都能安全重来。」 三根支柱缺一，链路就不再可靠。
