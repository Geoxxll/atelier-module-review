前端工作台

一块能实时「看见」四个 AI 工匠接力干活的工作台——提交委托，看它沿 PM·ARCH·CODE·REVIEW 逐站显影。

一句话：前端把一条抽象的 Temporal 工作流，翻译成看得见的工坊流水线——一条四工位导轨指示当前进度，六种产出卡逐张显影，而驱动这一切的是一根 SSE 事件流：后端每推一个 workflow.{id}.event，前端就把它 reduce 进「哪站在亮、哪张卡该出」的 UI 状态。

先建直觉

把「一条工作流」变成「一块看得见的工作台」

想象你把一张订单丢进一间手工作坊。作坊里有四位师傅——产品经理、架构师、程序员、审查员，他们沿一条流水线把活儿一站站往下传。你站在玻璃窗外，希望能实时看见：现在活儿传到了谁手上、每站产出了什么、卡在哪里需要你拍板。这块「玻璃窗」就是前端工作台干的事。

工作台把这层观感做成三样东西：一条 rail 导轨（当前委托停在哪一站，一眼可见），六张晒图卡（PRD / 技术方案 / 构建 / 审查 / 交付 / 需求画布，各站产物逐张显影），以及一份工单日志（逐条流水账）。而真正把这些点亮的，是后端一路推过来的事件。

为什么是 SSE（服务端推）而不是轮询 工作流是长时间、被动进展的——Agent 可能几十秒才产出一步，中途还会挂起等你裁决。轮询要么太密（浪费）、要么太疏（延迟）。SSE 让后端在「有事发生」的那一刻主动推，前端一根 EventSource 长连接就够，浏览器原生支持、断线自动重连、比 WebSocket 更轻——这是「单向服务端推流」的教科书场景。

两条路由：面向用户的台 + 面向开发者的台

前端有两个刻意分开的入口，不互相揉进对方外壳：

图版 I-c · 路由地图STATIC · routes
路由 · app/page.tsx/ 工作台面向终端用户：提交委托、看四站显影、内联裁决。单租户视角、单条 workflow 的 SSE 订阅。
路由 · app/dev-console/page.tsx/dev-console 可观测台DEV_MODE-only 跨租户控制室：Registry → Run Ledger → Telemetry → LLM Wire 四窗格 + 记忆档案，走 /api/admin/* 只读 SSE。

图版 II · 动态流程

SSE 的一生：从订阅到把事件 reduce 成 UI

前端不「拉」进度，而是订一根事件流，被动接收。核心是一个 handleMessage reducer——每收一个事件，按 type 决定「点亮哪站、出哪张卡、写哪条日志」。

图版 II-a · SSE 生命周期（提交 → 订阅 → 归约 → 渲染）DYNAMIC · SSE lifecycle
1
handleSubmit · 用户提交委托

POST /api/projects/{id}/features，拿回 { workflow_id } 写进 state；先手动把 PM 站点亮为 active。

2
useEffect([workflowId]) · 开连接

new EventSource(`/api/workflows/${id}/events`)，给约 18 种事件名逐个 addEventListener，外加 onmessage 兜底。

workflowId 变化即重建连接；卸载 / 终止时 closedRef=true + es.close()，防止已关连接的迟到帧再改状态。

3
收到帧 · JSON.parse

每帧 evt.data 反序列化成 AgentEvent（扁平 camelCase）。解析失败（ping 帧等）静默忽略，不炸。

4
handleMessage · 事件 → 状态归约

一长串 if (eventType === …) 分支：coder_started → 把 architect 置 done、coder 置 active；coder_completed → setBuild(…)；review_completed → setReview + 按 approved 决定 reviewer 站 done/active。每条都追加一行 ledger。

四站状态机 + 六卡 setState 全在这一个函数里收口——事件是唯一的状态推进器。

5
遇裁决点 · 切 interactionPhase

prd_pending_approval / tech_spec_pending_approval / clarification_needed / refine_turn 会把 phase 切到对应交互态，内联冒出 Sheet 面板等人操作。

6
终止事件 · 收束

workflow_completed / failed / rejected 命中 TERMINAL_EVENTS → es.close()、phase=done、点亮交付印章。React 按新 state 重渲染导轨与卡。

关键切分：「连接生命周期」（开 / 关 / 防迟到帧）在 useEffect 里管；「事件语义」（哪站亮、哪卡出）在 handleMessage 这个 reducer 里管。UI 是 state 的纯函数——前端从不主动查进度，只把服务端推来的事件投影成界面。

图版 II-b · RefineCanvas · 纯投影（不持久化）DYNAMIC · ephemeral projection

refine_canvas 事件 → 三栏视图

A
收到 refine_canvas 帧

PM 打磨每一轮都推一帧，载荷是当轮完整快照：{ round, confirmed[], pending[], parked[] }。

B
projectRefineCanvas(data)

纯函数投影：容错读三个字符串数组（缺失/非数组降级为 []），产出 RefineCanvasView。不做任何累积/合并。

C
setCanvas → CanvasBoard 覆盖渲染

直接覆盖上一轮，渲染 已确认 / 待定 / 延展 idea 三栏。

为什么它可以「不入库」

  • 画布只是展示投影，不是真相源——唯一真相源是后端的 conversation_history。
  • 每帧是完整快照，所以「覆盖」而非「合并」就够——无状态累积 = 无一致性负担。
  • 刷新 / 重连丢了也不怕：可由对话历史随时重算，前端不背持久化的锅。
  • 投影逻辑抽成零依赖纯函数（refineCanvas.mjs），能用 node:test 直接验证「事件 → 三栏」的外部行为。

设计品味：把「实时性」和「持久性」拆开——需要持久的（对话）留在后端当真相源，前端只做无副作用的投影。这让画布组件极简、可单测、可随时重建，是「让易变的东西无状态」的一个干净样本。

核心机制 · 钻到实现级

五个撑起工作台的机制

展开看代码级细节，也是面试里最容易被追问「你前端具体怎么做的」的地方。

① SSE 订阅与「事件 → 状态归约」

工作台的心脏是一个 useEffect([workflowId])：workflowId 一变就建连接，给约 18 种命名事件逐个挂监听，全部汇进同一个 handleMessage(evt, type) reducer。它把「网络帧」翻译成「四站状态机 + 六卡 setState + 一行日志」。

const es = new EventSource(`${API_BASE}/api/workflows/${workflowId}/events`);

const handleMessage = (evt, eventType) => {
  if (closedRef.current) return;              // 已关连接的迟到帧，丢弃
  let data; try { data = JSON.parse(evt.data); } catch { return; } // ping 帧静默忽略

  if (eventType === "coder_started") {
    setStation("architect", "done"); setStation("coder", "active");
  } else if (eventType === "review_completed") {
    const ok = data.approved ?? false;
    setReview({ approved: ok, overallScore: data.overallScore ?? 0, … });
    setStation("reviewer", ok ? "done" : "active");   // 未过=退回返工
  } /* …约 18 个分支… */

  setLedger(prev => [...prev, { id: ++eventCounter.current, type, text, tone, timestamp }]);
  if (TERMINAL_EVENTS.has(eventType)) { closedRef.current = true; es.close(); setDone(true); }
};
eventTypes.forEach(t => es.addEventListener(t, e => handleMessage(e, t)));
es.onmessage = e => handleMessage(e, "update");   // 无名帧兜底
  • 为什么命名事件 + onmessage 双挂：后端多数事件带 event: 名（走 addEventListener），但也可能发无名帧（走 onmessage），两头都兜住不漏。
  • closedRef 守卫：SSE 关闭是异步的，用一个 ref 标志挡住「关闭后迟到的帧再 setState」——否则会在已卸载/已完成的界面上乱改状态。
  • reducer 收口：所有状态推进只在 handleMessage 里发生，读代码时「事件→界面」一处可查，是可维护性的关键。
② token 重映射：为什么换皮几乎不动 JSX？

2026-07-01 把整站从「Cyanotype 工程蓝图」深色美学重绘成 claude-style 暖色，却几乎没改一行 JSX。秘诀：颜色从不写死在组件里，而是全部引用语义 token；重构时只改 token 的值，类名 / 变量名 / DOM 结构原封不动。

/* 旧（蓝图）与新（claude 暖色）共用同一个变量名，只换值 */
:root, [data-theme="light"] {
  --ground: #faf9f5;    /* was: 深蓝图底 */
  --brass:  #d97757;    /* clay coral：活动站 / 印章 / 强调 */
  --verdigris: #5a8f6b; /* 暖 sage 绿：通过 */
}
[data-theme="dark"] { --ground: #0d0d0c; --brass: #e0855f; … }

// 组件侧一直是这样引用，重构前后一字未改：
<div className="rail-fill" style={{ width: fillWidth }} />  // 背景色来自 --brass
为什么这招值钱把「设计决策」（具体色值）和「结构决策」（类名/DOM）解耦：换整套视觉语言变成「改一张变量表」，而非「全局搜替颜色、逐组件回归」。连旧类名 .develop（原是蓝图显影模糊）都保留，只把动画重定义为暖色淡入上浮——名字是接口，值是实现。

代价 / 边界：语义 token 必须起对名字（--brass 叫「活动强调色」而非「蓝色」），否则重映射后名不副实（旧 --brass 黄铜、新值 clay coral，名字已略微失真，属可接受的历史包袱）。

③ 无闪烁主题初始化 + localStorage 持久化

暗色模式最怕「白屏一闪」——SSR 先渲染 light，客户端 JS 再改 dark，中间那一帧刺眼白。解法：在 <head> 里放一段阻塞式内联脚本，在首次 paint 之前就把主题落到 <html data-theme>。

// layout.tsx —— 同步执行、先于 paint
const themeInit = `(function(){try{
  var t = localStorage.getItem('atelier-theme');
  if (t!=='light' && t!=='dark')
    t = matchMedia('(prefers-color-scheme: dark)').matches ? 'dark':'light';
  document.documentElement.setAttribute('data-theme', t);
}catch(e){}})();`;
<html data-theme="light" suppressHydrationWarning>
  <head>…<script dangerouslySetInnerHTML={{ __html: themeInit }} /></head>
  • 先读 localStorage，再退回系统偏好：用户选过就尊重选择，没选过跟系统。
  • suppressHydrationWarning：脚本会在水合前改 DOM 属性，React 会检测到 SSR/客户端不一致——这里刻意压掉这个警告（是正当的外部同步）。
  • ThemeToggle 水合后单次从 DOM 回读真实主题（no-flash 脚本可能已改成 dark），再由用户点击 toggle 时写回 localStorage + 翻转 data-theme。
④ RefineCanvas 的临时投影：为什么坚持不入库

需求打磨每轮推一帧 refine_canvas，前端把它渲染成「已确认 / 待定 / 延展 idea」三栏。关键克制：画布只是投影，不是真相源——不累积、不合并、不持久化。

// refineCanvas.mjs —— 零依赖纯函数，node:test 可直测
function asStringList(v) { return Array.isArray(v) ? v.filter(x => typeof x === "string") : []; }

export function projectRefineCanvas(data) {
  return {
    round: typeof data?.round === "number" ? data.round : 0,
    confirmed: asStringList(data?.confirmed),   // 每帧=当轮完整快照
    pending:   asStringList(data?.pending),      // 直接覆盖，不 merge
    parked:    asStringList(data?.parked),
  };
}
取舍把「持久」的责任留在后端的 conversation_history（唯一真相源），前端做无状态覆盖。收益：组件极简、可单测、刷新/重连丢了能由历史重算；代价：断线期间的中间态不会自动补回（见「限制」的 SSE 重连回填）。这是「让易变层无状态」的一次干净应用。
⑤ 动效原语：ambient mesh、scroll-reveal 与 reduced-motion

三个共享 primitives（app/components/）承载 claude-style 的「有呼吸感但不喧哗」的动效，且每一个都尊重 prefers-reduced-motion。

// AmbientMesh：光斑跟随光标，用 rAF lerp 而非 CSS transition
if (matchMedia("(prefers-reduced-motion: reduce)").matches) return;  // 直接不启动 rAF
const frame = () => {
  sx += (tx - sx) * EASE;  sy += (ty - sy) * EASE;   // 缓动追逐光标
  spot.style.transform = `translate3d(${sx}px,${sy}px,0)`;
  // 附近的暖色光晕被「推开」，settled 后停帧省电
};
  • AmbientMesh：几团重模糊暖色 blob 各自慢时钟漂移 + 光标光斑（rAF-lerp，比 CSS transition 更跟手），mask 向下淡出，垫在 hero 之下。
  • Reveal：IntersectionObserver 进视口才加 .in 触发淡入上浮，兄弟元素用 delay={i*70} 错峰；reduced-motion 下 CSS 直接显示、无需 JS 特判。
  • ThemeToggle：第三个共享原语，见深挖 ③。三者构成跨两条路由复用的 UI 底座。
  • 无障碍floor：globals.css 里 @media (prefers-reduced-motion: reduce) 统一冻结 blob 漂移、隐藏光斑、关掉 stamp/rail 动画；:focus-visible 全局可见焦点环。
面试可讲 · 取舍点

这块前端能讲出什么「设计品味」

不是「我用 React 写了个页面」，而是每个决定背后都有一句「为什么这样更好」。

为什么用 SSE 而不是轮询 / WebSocket？

单向服务端推的最简解

进度是后端被动产出、单向下推，不需要双工。SSE 一根长连接、浏览器原生、断线自动重连，比 WebSocket 轻，比轮询准且省。选型贴合数据流方向。

整站换皮为什么几乎不动 JSX？

token 重映射 · 名字是接口

颜色全走语义 CSS 变量，重构只改变量的值、不动类名与 DOM。换整套视觉语言 = 改一张变量表，而非全局搜替 + 逐组件回归。

暗色模式怎么不闪白？

paint 前阻塞式内联脚本

<head> 里同步落 data-theme，先读 localStorage 再退回系统偏好；配 suppressHydrationWarning 压掉水合不一致。首帧就是对的主题。

需求画布为什么不持久化？

让易变层无状态

真相源在后端对话历史，前端只做无副作用覆盖投影。组件极简、可单测、丢了能重算——「持久 vs 实时」被干净地拆开。

18 种事件怎么不写成一团乱麻？

单一 reducer 收口

所有状态推进只在 handleMessage 里发生，四站机 + 六卡 + 日志一处可查。加 closedRef 守卫挡住已关连接的迟到帧。

产品台和调试台为什么分两条路由？

关注点 + 暴露面隔离

工作台面向用户、走 RLS 内数据；可观测台跨租户、仅 DEV_MODE、信息密度高。两者共用 token 与 primitives，但布局与数据源各自独立。
