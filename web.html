前端工作台

一块能实时「看见」一支 AI 团队接力干活的工作台——登录后进项目，提交委托，看它沿 PM · ARCH · 闭环编排 逐站显影。

一句话：前端把一条抽象的 Temporal 工作流，翻译成看得见的工坊流水线——一条三工位导轨指示进度，产出卡逐张显影，闭环段还铺开一张切片 DAG 波次视图；驱动这一切的是一根经 sseConnect 换票管理的 SSE 事件流：后端每推一个 workflow.{id}.event，前端就把它 reduce 进「哪站在亮、哪张卡该出」的 UI 状态。

先建直觉

把「一条工作流」变成「一块看得见的工作台」

想象你把一张订单丢进一间手工作坊。先经产品经理把需求打磨清楚、架构师定下技术方案，随后进入一段闭环编排——它把活儿切成若干片，铺开让多名程序员 / 审查员并行去做、再归并成一条集成分支。你站在玻璃窗外，希望能实时看见：现在活儿到了哪一站、每站产出了什么、卡在哪里需要你拍板。这块「玻璃窗」就是前端工作台干的事。

工作台把这层观感做成三样东西：一条 rail 导轨（当前委托停在 PM / ARCH / 闭环 哪一站，一眼可见），若干晒图卡（需求画布 / PRD / 技术方案，以及闭环段那张切片 DAG 波次视图，各站产物逐张显影），以及一份工单日志（逐条流水账）。而真正把这些点亮的，是后端一路推过来的事件。整个应用先经一道 AuthGate 认人闸——未登录先落 /login，登录后才进得来。

为什么是 SSE（服务端推）而不是轮询 工作流是长时间、被动进展的——Agent 可能几十秒才产出一步，中途还会挂起等你裁决。轮询要么太密（浪费）、要么太疏（延迟）。SSE 让后端在「有事发生」的那一刻主动推，前端一根 EventSource 长连接就够，浏览器原生支持、比 WebSocket 更轻——这是「单向服务端推流」的教科书场景。只是端点靠一次性 ticket 鉴权，原生重连不可复用，故换票与断线重连收进一层 sseConnect 封装（见深挖 ①）。

前端从单页长成一套多路由应用（issue #124 起）。除公开登录页外，每个页面都裹一层 AuthGate（打 /api/auth/me 认人、未登录跳 /login），产品台与调试台仍刻意分开、不互揉外壳：

图版 II · 动态流程

SSE 的一生：从换票订阅到把事件 reduce 成 UI

前端不「拉」进度，而是订一根事件流，被动接收。连接生命周期（换票 / 建连 / 断线重连）交给 sseConnect；事件语义交给一个 handleMessage reducer——每收一个事件，按 type 决定「点亮哪站、出哪张卡、写哪条日志」。

图版 II-a · SSE 生命周期（提交 → 换票订阅 → 归约 → 渲染）DYNAMIC · SSE lifecycle
1
handleSubmit · 用户提交委托

authFetch POST /api/projects/{id}/features（401→refresh→retry 透明），拿回 { workflow_id } 写进 state；先手动把 PM 站点亮为 active。

402 是配额预检拦下（M2-3），单独标出、引导去 /settings，不把英文原始 detail 糊脸上。

2
useEffect([workflowId]) · sseConnect 开连接

sseConnect({ workflowId, eventTypes, onEvent, onStatusChange, onFatalError })：先 POST /api/sse-ticket 换一枚一次性票，再 new EventSource(?ticket=…)，给约 23 种事件名逐个 addEventListener + onmessage 兜底。

端点靠一次性 ticket 鉴权，原生重连拿旧票必失败——故 onerror 时退避 → 换新票 → 整个重建（不复用），重连带 last_event_id 补历史。卸载 / 终止时 closedRef=true + conn.close()。

3
收到帧 · 去重 + JSON.parse

先按 `${type}:${evt.data}` 内容签名过 seenEventKeys（挡重连补档的重放），再把 evt.data 反序列化成 AgentEvent。解析失败（ping 帧等）静默忽略。

4
handleMessage · 事件 → 状态归约

一长串 if (eventType === …) 分支：architect_started → PM done、ARCH active；tech_spec_pending_approval → setTechSpec；闭环段 planner_done / wave_started / slice_completed / wave_merged / finalized 一律委托 reduceClosedLoopEvent 折叠进 closedLoop。每条都追加一行 ledger。

三站状态机 + 各卡 setState 全在这一个函数里收口——事件是唯一的状态推进器。

5
遇裁决点 · 切 interactionPhase

refine_turn / clarification_needed / prd_pending_approval / tech_spec_pending_approval / plan_pending_approval 会把 phase 切到对应交互态，内联冒出 Sheet 面板等人操作。

6
终止事件 · 收束

finalized / workflow_completed / failed / rejected 命中 TERMINAL_EVENTS → conn.close()、phase=done、点亮闭环终局卡。React 按新 state 重渲染导轨与卡。

关键切分：「连接生命周期」（换票 / 重连 / 防迟到帧）在 sseConnect 里管；「事件语义」（哪站亮、哪卡出）在 handleMessage 这个 reducer 里管，闭环 DAG 再下沉到 reduceClosedLoopEvent 纯核。UI 是 state 的纯函数——前端从不主动查进度，只把服务端推来的事件投影成界面。

为什么它可以「不入库」

  • 画布只是展示投影，不是真相源——唯一真相源是后端的 conversation_history。
  • 每帧是完整快照，所以「覆盖」而非「合并」就够——无状态累积 = 无一致性负担。
  • 刷新 / 重连丢了也不怕：可由对话历史随时重算，前端不背持久化的锅。
  • 投影逻辑抽成零依赖纯函数（refineCanvas.mjs），能用 node:test 直接验证「事件 → 三栏」的外部行为。

设计品味：把「实时性」和「持久性」拆开——需要持久的（对话）留在后端当真相源，前端只做无副作用的投影。这让画布组件极简、可单测、可随时重建，是「让易变的东西无状态」的一个干净样本。

核心机制 · 钻到实现级

五个撑起工作台的机制

展开看代码级细节，也是面试里最容易被追问「你前端具体怎么做的」的地方。

① SSE 订阅（sseConnect 换票 + 重连）与「事件 → 状态归约」

工作台的心脏是一个 useEffect([workflowId])：workflowId 一变就调 sseConnect 建连（换票 / 断线重连都封在里面），事件全汇进同一个 handleMessage(evt, type) reducer。它把「网络帧」翻译成「三站状态机 + 各卡 setState + 一行日志」，闭环 DAG 再下沉到共享纯核 reduceClosedLoopEvent。

const seenEventKeys = new Set();  // 重连补档去重（内容签名）
const conn = sseConnect({ workflowId, eventTypes, onEvent: handleMessage,
  onStatusChange: setConnStatus,   // connecting / open / reconnecting
  onFatalError: () => { /* 线路彻底断，提示重提委托 */ } });

const handleMessage = (evt, eventType) => {
  if (closedRef.current) return;                // 已关连接的迟到帧，丢弃
  const key = `${eventType}:${evt.data}`;
  if (seenEventKeys.has(key)) return;            // 重放同一帧 → 整条跳过
  seenEventKeys.add(key);
  let data; try { data = JSON.parse(evt.data); } catch { return; } // ping 帧忽略

  if (eventType === "architect_started") {
    setStation("pm", "done"); setStation("architect", "active");
  } else if (eventType === "planner_done") {                // 闭环段 → 共享纯核
    setClosedLoop(prev => reduceClosedLoopEvent(prev, eventType, data));
  } /* …约 23 个分支（wave/slice/merge 同样走 reducer）… */

  setLedger(prev => [...prev, { id: ++eventCounter.current, type, text, tone, timestamp }]);
  if (TERMINAL_EVENTS.has(eventType)) { closedRef.current = true; conn.close(); setDone(true); }
};
  • 为什么不用原生重连：SSE 端点靠 URL 上一次性 ticket 鉴权（POST /api/sse-ticket 换、30s TTL、订阅即核销），原生 EventSource 断线会拿着同一枚已核销的票重试、注定失败。sseConnect 接管：退避 → 换新票 → 整个新建（不复用旧连接），换票走 authFetch（401→refresh→retry 透明）。
  • seenEventKeys 去重：重连带 last_event_id 补历史时，会重推「live 已发但 event_log 还没落库」那批帧；工单日志是 append-only，用「事件类型 + 原始 payload」内容签名挡掉重放，避免日志多出一行。
  • reducer 幂等 + 共享：闭环 DAG 分支全走 reduceClosedLoopEvent，按 key（sliceId / wave）整体覆盖、重复投递结果不变；这同一个纯核被 runs/[runId] 历史回看复用（拉 events/history 折叠），实时与回看不各写一套累积逻辑。
  • closedRef 守卫：连接关闭是异步的，用一个 ref 挡住「关闭后迟到帧再 setState」，否则会在已卸载/已完成的界面上乱改状态。
② token 重映射：为什么换皮几乎不动 JSX？

2026-07-01 把整站从「Cyanotype 工程蓝图」深色美学重绘成 claude-style 暖色，却几乎没改一行 JSX。秘诀：颜色从不写死在组件里，而是全部引用语义 token；重构时只改 token 的值，类名 / 变量名 / DOM 结构原封不动。

/* 旧（蓝图）与新（claude 暖色）共用同一个变量名，只换值 */
:root, [data-theme="light"] {
  --ground: #faf9f5;    /* was: 深蓝图底 */
  --brass:  #d97757;    /* clay coral：活动站 / 印章 / 强调 */
  --verdigris: #5a8f6b; /* 暖 sage 绿：通过 */
}
[data-theme="dark"] { --ground: #0d0d0c; --brass: #e0855f; … }

// 组件侧一直是这样引用，重构前后一字未改：
<div className="rail-fill" style={{ width: fillWidth }} />  // 背景色来自 --brass
为什么这招值钱把「设计决策」（具体色值）和「结构决策」（类名/DOM）解耦：换整套视觉语言变成「改一张变量表」，而非「全局搜替颜色、逐组件回归」。连旧类名 .develop（原是蓝图显影模糊）都保留，只把动画重定义为暖色淡入上浮——名字是接口，值是实现。

代价 / 边界：语义 token 必须起对名字（--brass 叫「活动强调色」而非「蓝色」），否则重映射后名不副实（旧 --brass 黄铜、新值 clay coral，名字已略微失真，属可接受的历史包袱）。

③ 无闪烁主题初始化 + localStorage 持久化

暗色模式最怕「白屏一闪」——SSR 先渲染 light，客户端 JS 再改 dark，中间那一帧刺眼白。解法：在 <head> 里放一段阻塞式内联脚本，在首次 paint 之前就把主题落到 <html data-theme>。

// layout.tsx —— 同步执行、先于 paint
const themeInit = `(function(){try{
  var t = localStorage.getItem('atelier-theme');
  if (t!=='light' && t!=='dark')
    t = matchMedia('(prefers-color-scheme: dark)').matches ? 'dark':'light';
  document.documentElement.setAttribute('data-theme', t);
}catch(e){}})();`;
<html data-theme="light" suppressHydrationWarning>
  <head>…<script dangerouslySetInnerHTML={{ __html: themeInit }} /></head>
  • 先读 localStorage，再退回系统偏好：用户选过就尊重选择，没选过跟系统。
  • suppressHydrationWarning：脚本会在水合前改 DOM 属性，React 会检测到 SSR/客户端不一致——这里刻意压掉这个警告（是正当的外部同步）。
  • ThemeToggle 水合后单次从 DOM 回读真实主题（no-flash 脚本可能已改成 dark），再由用户点击 toggle 时写回 localStorage + 翻转 data-theme。
④ RefineCanvas 的临时投影：为什么坚持不入库

需求打磨每轮推一帧 refine_canvas，前端把它渲染成「已确认 / 待定 / 延展 idea」三栏。关键克制：画布只是投影，不是真相源——不累积、不合并、不持久化。

// refineCanvas.mjs —— 零依赖纯函数，node:test 可直测
function asStringList(v) { return Array.isArray(v) ? v.filter(x => typeof x === "string") : []; }

export function projectRefineCanvas(data) {
  return {
    round: typeof data?.round === "number" ? data.round : 0,
    confirmed: asStringList(data?.confirmed),   // 每帧=当轮完整快照
    pending:   asStringList(data?.pending),      // 直接覆盖，不 merge
    parked:    asStringList(data?.parked),
  };
}
取舍把「持久」的责任留在后端的 conversation_history（唯一真相源），前端做无状态覆盖。收益：组件极简、可单测、刷新/重连丢了能由历史重算；代价：断线期间的中间态不会自动补回（见「限制」的 SSE 重连回填）。这是「让易变层无状态」的一次干净应用。
⑤ 动效原语：ambient mesh、scroll-reveal 与 reduced-motion

三个共享 primitives（app/components/）承载 claude-style 的「有呼吸感但不喧哗」的动效，且每一个都尊重 prefers-reduced-motion。

// AmbientMesh：光斑跟随光标，用 rAF lerp 而非 CSS transition
if (matchMedia("(prefers-reduced-motion: reduce)").matches) return;  // 直接不启动 rAF
const frame = () => {
  sx += (tx - sx) * EASE;  sy += (ty - sy) * EASE;   // 缓动追逐光标
  spot.style.transform = `translate3d(${sx}px,${sy}px,0)`;
  // 附近的暖色光晕被「推开」，settled 后停帧省电
};
  • AmbientMesh：几团重模糊暖色 blob 各自慢时钟漂移 + 光标光斑（rAF-lerp，比 CSS transition 更跟手），mask 向下淡出，垫在 hero 之下。
  • Reveal：IntersectionObserver 进视口才加 .in 触发淡入上浮，兄弟元素用 delay={i*70} 错峰；reduced-motion 下 CSS 直接显示、无需 JS 特判。
  • ThemeToggle：第三个共享原语，见深挖 ③。三者（连同 AuthGate）构成跨全部路由复用的 UI 底座。
  • 无障碍floor：globals.css 里 @media (prefers-reduced-motion: reduce) 统一冻结 blob 漂移、隐藏光斑、关掉 stamp/rail 动画；:focus-visible 全局可见焦点环。
面试可讲 · 取舍点

这块前端能讲出什么「设计品味」

不是「我用 React 写了个页面」，而是每个决定背后都有一句「为什么这样更好」。

为什么用 SSE 而不是轮询 / WebSocket？

单向服务端推的最简解

进度是后端被动产出、单向下推，不需要双工。SSE 一根长连接、浏览器原生，比 WebSocket 轻，比轮询准且省。端点靠一次性 ticket 鉴权，所以换票 + 断线重连收进一层 sseConnect 封装，而非裸原生重连。

整站换皮为什么几乎不动 JSX？

token 重映射 · 名字是接口

颜色全走语义 CSS 变量，重构只改变量的值、不动类名与 DOM。换整套视觉语言 = 改一张变量表，而非全局搜替 + 逐组件回归。

暗色模式怎么不闪白？

paint 前阻塞式内联脚本

<head> 里同步落 data-theme，先读 localStorage 再退回系统偏好；配 suppressHydrationWarning 压掉水合不一致。首帧就是对的主题。

需求画布为什么不持久化？

让易变层无状态

真相源在后端对话历史，前端只做无副作用覆盖投影。组件极简、可单测、丢了能重算——「持久 vs 实时」被干净地拆开。

23 种事件怎么不写成一团乱麻？

单一 reducer 收口 + 闭环下沉纯核

所有状态推进只在 handleMessage 里发生，三站机 + 各卡 + 日志一处可查；闭环 DAG 再委托 reduceClosedLoopEvent。closedRef 挡迟到帧、seenEventKeys 挡重放。

实时进度和历史回看如何不各写一套？

共享一个累积纯核

实时工作台逐条推 SSE、历史回看一次性拉 events/history，都喂给同一个 reduceClosedLoopEvent 折叠。按 key 覆盖天然幂等，数据源变了、累积规则不变。
