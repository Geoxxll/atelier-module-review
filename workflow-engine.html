工作流编排

一段能横跨几天、中途等人拍板、崩了还能原地续跑的「导演脚本」——把 PM→Architect→Planner→闭环执行 串成一条可靠流水线。

一句话：FeatureWorkflow 是这条端到端链路的总导演（PM→Architect→Planner→child 编排闭环执行，闭环就跑在它内部）。它自己不干活、不碰 IO——只用确定性的编排逻辑决定「下一步派谁、等哪个信号、循环几轮」。真正的活（LLM、数据库、NATS）全丢给 Activity；真正的拍板（批 PRD、审技术方案）靠 Signal 让人插手。崩溃后 Temporal 靠重放恢复到崩前一刻——代价是 Workflow 代码必须字字确定。

为什么这条流水线不能用一个 async 函数写？

你可能会想：不就是「PM 写完 → Architect 写完 → Coder 写完」吗，写个 async def 一路 await 下去不就行了？问题出在三个字：它太久、要等人、会崩。

  • 太久——一次功能开发可能跑几十分钟到几天，中间挂着好几次 LLM 长调用。进程重启、机器换代，一个裸 async 函数的内存状态就全没了。
  • 要等人——PRD 要用户点「批准」、技术方案要用户点「通过」。这个「等」可能是 5 秒，也可能是明天早上。你不能真的让一个线程干等一整夜。
  • 会崩——闭环执行跑到某个切片时 worker 崩了，你希望它从那一步续，而不是从 PM 写 PRD 重来一遍（那要重烧多少 LLM 钱）。

Temporal 的答案是 Durable Execution（可持久化执行）：把编排逻辑写成一个可重放的 Workflow。它把每一步的输入输出记进 Event History；崩溃后重新执行一遍代码，遇到已记录的步骤直接取历史结果、不重跑，一路快进到崩前那一刻，再继续。等人？用 wait_condition 挂起，不占线程，Signal 一到就醒。

贯穿全篇的一条铁律 · 确定性是地基 「重放能恢复」的前提是同样的代码 + 同样的历史 = 同样的执行路径。所以 Workflow 里绝不能出现 uuid4() / datetime.now() / random / 直接 IO——它们每次跑结果都不同，重放立刻错位。本模块所有设计（uuid5 定 id、IO 全塞进 Activity、时间用 workflow.now）都是为守住这条线，「深挖 ①」细讲。

怎么读：竖着一路往下是主时间线。蓝点（R/0/1/2/P）是「等人」的地方——Workflow 在此挂起等 Signal（P 仅在 Planner 报 needs_human 时才出现）；两个绿星 ✦是「悄悄写记忆」的旁支，失败也不影响主交付。注意几个循环的上限性质不同：打磨是软上限（提醒不砍）、澄清是硬闸（到顶 break）、PRD 修订无上限——上限该软该硬，取决于「停止权该归谁」。真正的编码不在本 workflow 直接派 Agent，而是下沉进 Step 6 的 child OrchestratorWorkflow。

八个 Signal：人在哪几处能插手

Workflow 靠 @workflow.signal 装的方法接收外部输入——本质是「往运行中的 Workflow 里推一个值 + 唤醒它」。每个 Signal 只改一个实例字段，配一个 wait_condition 醒来判断。八个 Signal 覆盖五处人机拍板点（refine / 澄清 / PRD / TechSpec / Plan）：

图版 II · 动态流程

放大两处：闭环执行怎么挂进来，一次上游派活怎么往返

静态图看的是「有哪些步」，动态图看的是「步与步之间怎么转」。挑两处最能体现编排精髓的：把闭环交给 child OrchestratorWorkflow 的交接（spec-threading + 事件回流），和一次上游 Activity 派活的完整往返（ADR-0011 的核心）。

图版 II-a · child OrchestratorWorkflow 交接 · spec-threading + 事件回流DYNAMIC · child handoff
1
父 · execute_child_workflow(OrchestratorWorkflow.run, …)

闸门通过后，父 workflow 把校验过的 planner_result.plan（Slice DAG）打包成 OrchestratorWorkflowInput，child id 用 uuid5(wf,"orchestrator-child")（禁 uuid4，守确定性），投到控制平面队列 orchestrator-wf。

2
spec-threading · per-run 的 prd/tech_spec/model 随 payload 透传

把本次 run 的 prd / tech_spec / closed_loop_model 塞进 OrchestratorWorkflowInput → child ProvisionRequest → RunContext，供闭环 Coder 现取（而非 worker 启动时钉死）。保证每个用户 run 都用自己的 spec 生成代码。

3
event_workflow_id · 闭环段事件回流父频道

child 不知道父的 workflow_id，父把 event_workflow_id=input.workflow_id 注入 payload；闭环段事件（wave_started / slice_started / … / finalized）据此路由回本 workflow 的 SSE 频道，前端一条流看到全程。

4
终态收口 · FinalizeResult / RunAborted

child 返回 FinalizeResult（成功收口，自发 enriched finalized 含 prUrl/integration_branch）→ 父置 completed；返回 RunAborted（脚手架失败/中止，从不进 Finalize 分支、不发 finalized）→ 父置 failed 并兜底补发 FINALIZED status="aborted"。

ADR-0015 + #133：终局事件的发出方职责单一——正常由 child 发、abort 由父兜底，同一终局语义不被父子各发一次。

为什么闭环是 child workflow 而非平行链：把编排下沉进独立 child，父只管「交接 DAG + 收终态」，child 内部的波次并行 / loop / 归并各自可持久化、可重放、可独立重投。spec-threading 让 per-run 配置随 payload 走，避免 worker 级全局状态；event_workflow_id 把父子两层事件缝成前端的一条 SSE 流。

那个挂起的 Activity 被外部 complete，Workflow 收到 RunArchitectAgentResult，发 tech_spec_pending_approval 事件、挂起等 resolve_tech_spec，批准后进 Planner。整条链路任务不丢、结果不丢、不重复烧 LLM。

三端一致的隐形前提：API client / workflow-engine worker / 各 Agent worker 都用 pydantic_data_converter——因为 handle.complete(result) 让「结果类型」成了跨服务契约（Run*Result 全在 atelier_shared.contracts）。三端序列化口径不一，结果就反序列化不出来。为什么不开 JetStream publish 去重？见「深挖 ④」——那会把重试的新 token 误删导致死锁。

端到端 · 走一遍

从「批准 PRD」到「写下情景记忆」的一条真实轨迹

跟着一个 workflow 实例走完主链路后半段：定稿 → 蒸馏 → 技术方案 → Planner 切片 → child 编排闭环 → finalize → 记忆。留意每一步的 uuid5 task_id 与 Signal payload——它们就是「确定性」和「人机回环」在真实数据里的样子。

场景 · workflow_id = 7f3a…c091 · feature「报销审批」

用户批准 PRD 后，Workflow 一路跑到 child 编排闭环并收口

T0 · Signal

用户在 PRD 第 1 版就满意，前端发 approve_prd。挂起在 wait_condition 的 Workflow 醒来，_prd_approved=True → break 出 PRD 修订环。

# Step 1 while 循环内
await workflow.wait_condition(
    lambda: self._prd_approved or self._prd_feedback is not None)
# approve_prd 到达 → _prd_approved=True → break，此刻 history 最全
T1 · 蒸馏

Step 1.5 / 1.6 两个 fire-and-forget 蒸馏触发。从整段 conversation_history 抽偏好 SOP + 用户事实，各包在 try/except 里。

# Step 1.5
try: await execute_activity(distill_pm_sop, …)
except Exception: workflow.logger.warning("workflow.sop_distill_failed")
# Step 1.6 紧随，同样 try/except —— 失败只丢一条记忆，不阻断交付
T2 · Architect

Step 2 派 Architect。task_id 用固定 salt（此处无循环，一个方案）。出方案后挂起等 resolve_tech_spec。

architect_task_id = uuid5(wf, "architect")  # → 固定，replay 一致
# 用户批准：
resolve_tech_spec({"approved": true, "reason": null})
# 若 approved=false → return FeatureWorkflowResult(status="rejected")
T3 · Planner

Step 3′ 派 run_planner（自包含 activity、无 task_id）把 PRD+TechSpec 切成 Slice DAG，发 planner_done 带切片列表。本例 status=="ok"——无依赖歧义，跳过 resolve_plan 人工闸，直接进编排。

planner_result = run_planner(prd, tech_spec)  # 无 task_id
# planner_done: sliceCount=4, status="ok" → 无需 resolve_plan
T4 · child 编排

Step 6 execute_child_workflow(OrchestratorWorkflow)。spec-threading 透传本 run 的 prd/tech_spec/model；event_workflow_id 把闭环段事件（wave_started/slice_started/…）路由回本 SSE 频道，前端一条流看到波次并行全程。

child_id = uuid5(wf, "orchestrator-child")  # 队列 orchestrator-wf
finalize = execute_child_workflow(OrchestratorWorkflow.run,
  OrchestratorWorkflowInput(prd=…, tech_spec=…,
    model=settings.closed_loop_model, event_workflow_id=wf))
T5 · finalize 收口

child 返回 FinalizeResult → 父置 completed；child 已自发 enriched finalized（含 prUrl/integration_branch）路由回本频道，父不重发。若为 RunAborted 则父置 failed + 兜底补发 FINALIZED status="aborted"。

if isinstance(finalize, FinalizeResult): status="completed"
else: status="failed"  # RunAborted → 父兜底补发 FINALIZED aborted
T6 · 情景记忆

Step 7 fire-and-forget：把整条 workflow 蒸馏成摘要，写 agent_memory(episodic, role=workflow)，留给未来相似任务的 Coder 检索。legacy Review 已摘，写入用闭环安全默认。

try: await execute_activity(write_episodic_memory,
       WriteEpisodicMemoryInput(prd_title="报销审批",
         reviewer_approved=false, review_rounds=0))  # 闭环安全默认
except Exception: workflow.logger.warning("workflow.memory_write_failed")
# FeatureWorkflowResult(status="completed")

整段轨迹里，每个 task_id 都由 uuid5(workflow_id, salt) 算出——同一实例重放，每一步拿到一模一样的 id，命中同一份幂等缓存，绝不重烧 LLM。这就是「确定性」落到实处的样子。

核心机制 · 钻到实现级

撑起整条流水线的五个机制

展开看代码级细节。这几处也是面试里最容易被追问「你具体怎么保证不出错」的地方。

① 确定性：为什么 task_id 必须用 uuid5、不能用 uuid4？

Temporal 靠重放代码恢复崩溃前状态：崩了就把 Workflow 从头再跑一遍，遇到已记录的 Activity 直接取历史结果、不重跑。这要求「同代码 + 同历史 = 同路径」。而 uuid4() 每次调都吐不同值——重放时新算的 id 和历史里那个对不上，Temporal 直接判 non-determinism 错误。

# ✗ 错：每次重放都是新 id → 幂等表命中不了 → 重烧 LLM / 报错
task_id = str(uuid.uuid4())
# ✓ 对：workflow_id 是种子，salt 定唯一 → 重放必得同一个 id
architect_task_id = str(uuid.uuid5(uuid.UUID(input.workflow_id), "architect"))
prd_task_id = str(uuid.uuid5(uuid.UUID(input.workflow_id), f"pm-prd-round-{r}"))
child_id = str(uuid.uuid5(uuid.UUID(input.workflow_id), "orchestrator-child"))
  • salt 带上循环变量（round）：同一实例的每一轮拿不同 id，但「同一轮」跨重放拿同一个 id——正好对上幂等缓存。
  • 同理禁的还有 datetime.now()（要用 workflow.now()）、random、直接 IO（httpx/nats/db）。所有非确定的东西全下沉进 Activity——Activity 的结果被记进历史，重放时取历史值，就「确定」了。
  • 连 import 都要用 workflow.unsafe.imports_passed_through() 包起来，保持沙箱纯洁。
② Signal 驱动的循环：软上限 vs 硬闸，停止权归谁

三个人机循环的骨架都一样：派活 → 发事件 → wait_condition 挂起 → Signal 唤醒 → 判断进退。但「什么时候不让再循环」的策略刻意不同，取决于「停止权该归用户还是归系统」：

循环上限策略为什么这么定
需求打磨软上限 20（REFINE_SOFT_CAP）到顶只置 nudge_converge=True 让 PM 善意提醒，不 break。用户想聊多久聊多久——探索需求的停止权在人。
PM 澄清硬闸 3（MAX_CLARIFICATION_ROUNDS）澄清是「补齐信息」，问 3 轮还不清就该往下走了；for range(3) 到顶自然 break。
PRD 修订无上限（while True）「PRD 满不满意」纯主观，只有用户 approve 才停——系统无权替他判「够好了」。
 
# 软上限：提醒但不砍，wait_condition 仍等用户下一步
nudge_converge = (not finalize) and (refine_round + 1 >= REFINE_SOFT_CAP)
await workflow.wait_condition(
    lambda: self._refine_input is not None or self._refine_finalized)

Signal handler 本身极简——只做一行赋值（self._refine_input = content），绝不含业务逻辑。这样「外部何时来」与「来了怎么处理」解耦，确定性主线集中在 wait_condition 之后，重放不会因 handler 里的复杂逻辑而错位。

③ fire-and-forget 蒸馏：为什么写记忆要 try/except 吞掉异常？

三处记忆写入（SOP、语义、情景）都用同一个模式：try: execute_activity(...) except Exception: logger.warning(...)。这是刻意的——记忆是锦上添花，绝不能拖垮或阻断主交付（PRD / 代码）。

# Step 1.5 —— 三处蒸馏是同一个防御骨架
try:
    await workflow.execute_activity(distill_pm_sop, …,
        retry_policy=RetryPolicy(maximum_attempts=2))
except Exception:
    workflow.logger.warning("workflow.sop_distill_failed")
    # 吞掉 → 主流程照常往下走，只是这次没学到偏好
  • 时机也讲究：SOP/语义在 Step 1.5/1.6——PRD 刚定稿、conversation_history 最完整、Architect 还没往里追加内容，此刻蒸馏信号最纯。情景记忆在 Step 5——要等整条 workflow 有了「审查通过 / 几轮」的结果才蒸馏得出。
  • retry 更克制：主链路 Activity 是 maximum_attempts=3，蒸馏只给 2——反正失败也不致命，不值得多重试烧钱。
  • 对比：resolve_tech_spec 拒绝会 return rejected 终止全流程——那是主链路决策，和 fire-and-forget 是两个极端。
④ _dispatch_agent_task + async completion：任务不丢、结果不丢、不重烧

所有派 Agent 的 Activity（run_pm_agent / run_architect_agent / …）都收敛到一个 _dispatch_agent_task。它做三件事：塞 token、publish 到 JetStream、raise_complete_async()。这是 ADR-0011 的核心——从 Core NATS request-reply（at-most-once、会丢）升级到持久化队列 + 异步完成。

async def _dispatch_agent_task(subject, payload):
    payload[ACTIVITY_TOKEN_KEY] = base64(activity.info().task_token)
    js = (await nats.connect(...)).jetstream()
    await js.publish(subject, json.dumps(payload).encode())  # 拿 PubAck 才算持久化
    activity.raise_complete_async()  # 结果稍后由 worker.complete() 交付

为什么不开 JetStream publish 去重？这是最容易踩的坑。token 是「每次 Activity 尝试」独有的。若用 Nats-Msg-Id=task_id 开 publish 去重：Temporal 超时重试产生的新 token 消息会被当重复丢掉 → 新 token 永远到不了 worker → 死锁。

正确做法不开 publish 去重；改由 worker 按 task_id 幂等（agent_task_results 表）。每次 Activity 尝试都 publish 一条带新 token 的新消息；worker 处理前查 task_id，命中就用缓存结果完成（过期 token 完成失败但无害，新 token 完成成功）。JetStream 负责「无 worker 时不丢 + 崩溃重投」，去重交给幂等层——职责分离才不打架。

代价（ADR-0011 也承认）：Agent worker 现在得持一个 Temporal client 来调 handle.complete()，打破了「Agent 只认 NATS、对编排器无感」的干净分层——靠 atelier_shared.agent_transport.run_agent_worker 把这层封起来缓解。

⑤ REFINE 模式：为什么它是「一个阶段」而不是「一个新 Agent」？

需求打磨看着像个独立能力，但实现上它不是新 Agent、也不是新 Workflow——只是 FeatureWorkflow 开头一段 if input.mode == "refine" 的可选前置阶段，复用同一个 PM Agent（只是换个 kind="refine" 的任务）。

# 入口分流：一个字段决定走不走打磨，默认 direct 向后兼容
class FeatureWorkflowInput(BaseModel):
    mode: Literal["direct", "refine"] = "direct"
# workflow.run 里：
if input.mode == "refine":
    while True: # 打磨循环，产物直接进 conversation_history
  • 零序列化衔接：打磨每轮的 Q/A、收束轮的需求综述，全 append 进同一个 conversation_history。这个 list 就是唯一真相源——打磨结束后自然流入 Step 0 澄清、Step 1 写 PRD，中间不落库、不转格式。
  • 画布只是投影：每轮发 refine_canvas 事件（confirmed/pending/parked 三栏）给前端渲染，但不落库——停止/重连后可由 history 重算。真相在 history，画布是派生视图。
  • 为什么不做成独立 Workflow？那样得跨 Workflow 传递打磨结果（序列化 + 启动开销），还割裂了「同一段对话」。塞进同一个 Workflow 的前置阶段，反而最省、最连贯——这是「阶段 vs 服务」边界的一次务实取舍。
面试可讲 · 取舍点

这个模块能讲出什么「工程判断」

不是「我用了 Temporal」，而是每一个约束背后都有一句「为什么必须这样」。

Workflow 里为什么不能直接调 LLM / 数据库？

确定性换持久性

Temporal 靠重放恢复，要求同代码同历史同路径。IO 和随机数每次结果不同，重放必错位。所以非确定的一切下沉进 Activity，结果记进历史再取——这是可持久化执行的地基。

怎么让一个跑几天、要等人的流程可靠？

wait_condition 挂起 + Signal 唤醒

等人不占线程：wait_condition 挂起，Signal 一到就醒。崩了重放快进到崩前一刻续跑。等 5 秒和等一整夜，代码写法完全一样。

上限该软还是该硬，怎么决定？

看停止权归谁

探索需求（打磨）停止权在人 → 软上限只提醒；PRD 满意度纯主观 → 无上限；Agent 互审会无限烧钱 → 硬闸止损。上限策略是「谁有权喊停」的直接映射。

任务传输为什么从 NATS request-reply 换掉？

持久化 + 幂等，不丢不重烧

request-reply 是 at-most-once，worker 崩在 respond 前就整个重跑。改 JetStream 投递 + async completion：任务持久化、结果进 Event History、按 task_id 幂等——重试重投都不重烧 LLM。

为什么偏偏不开 JetStream publish 去重？

去重放错层会死锁

token 每次尝试都不同，用 task_id 做 publish 去重会把重试的新 token 误删 → 新 token 永远到不了 worker → 死锁。投递层保不丢、幂等层管去重，职责分离。

写记忆失败会影响主交付吗？

fire-and-forget 优雅降级

三处蒸馏都 try/except 吞异常 + 只 retry 2 次。失败只丢一条记忆、warning 一行，PRD/代码照常产出。记忆是增强，不是主链路的单点。

一句话总结FeatureWorkflow 的全部设计，都是在回答一个问题：「一段又长、又要等人、又会崩的协作流程，怎么写得可靠？」答案是三件事——确定性守住重放、Signal 接住人、fire-and-forget 兜住旁支。守住这三条，其余都是细节。
